# Claude Code 用語集

> 本用語集は Claude Code トレーニング資料から抽出した専門用語をアルファベット順に整理したものです。

---

## A

### Agent Loop（エージェントループ）
Claude Code の基本動作構造。`while(tool_call)` ループで動作し、分類器・RAGパイプライン・DAGオーケストレーターは使用しない。コンテキスト収集→アクション実行→結果検証の3フェーズで流動的に遷移する。
**参照:** 01_基礎と設定 R-001, R-002

### Agent Skills
agentskills.io のオープンスタンダードに準拠したモジュール式の能力パッケージ。`skills:` フロントマターフィールドでサブエージェント起動時にプリロードされる形式と、Skill ツール経由でオンデマンド呼び出しされる形式がある。
**参照:** 01_基礎と設定 R-076, 04_拡張機能 R-013

### Agent Teams（エージェントチーム）
複数の Claude Code インスタンスを協調動作させる実験的機能。1つのセッションがチームリードとして機能し、チームメイトはそれぞれ独自のコンテキストウィンドウを持つ。標準セッション比で約7倍のトークンを使用する。
**参照:** 04_拡張機能 R-082〜R-093

### Anti-rationalization Gate
Stop フックで Claude が不完全な作業を「完了」と宣言することを防止する仕組み。高速モデル（Haiku）に最終レスポンスを評価させ、問題を「スコープ外」と主張したりテスト失敗をスキップする等の合理化パターンを拒否する。
**参照:** 06_ワークフローと方法論 R-096〜R-097, 08_セキュリティ R-047

### ATDD（Acceptance Test-Driven Development）
コーディング前に受入基準を協調定義する開発手法。ステークホルダー合意を重視する場面に適する。
**参照:** 06_ワークフローと方法論 R-007

### Auto Dream
バックグラウンドのメモリ統合機能。24時間ごと（5セッション以上経過後）に実行され、メモリファイルの自動プルーニング・マージ・リフレッシュを行う。スキャン→マップ構築→パターン検索→統合の4フェーズで動作し、MEMORY.md を200行以下に維持する。
**参照:** 01_基礎と設定 R-047〜R-052

### Auto Memory（自動メモリ）
Claude が自動的にセッション間の知識（ビルドコマンド、デバッグ知見、アーキテクチャノート等）を蓄積する機能。デフォルトで有効。`~/.claude/projects/<project>/memory/` に保存される。
**参照:** 01_基礎と設定 R-041〜R-046, 02_コンテキストエンジニアリング R-089

## B

### BDD（Behavior-Driven Development）
Given-When-Then シナリオでステークホルダーと協業する開発手法。受入基準重視の場面に適する。
**参照:** 06_ワークフローと方法論 R-006

### BMAD（Business-driven Methodical AI Development）
マルチエージェント・ガバナンスとコンスティチューション・ガードレールを提供するエンタープライズ向け開発方法論。34以上のワークフローと12以上のドメイン専門エージェントを持ち、Party Mode（マルチエージェント協調）をサポートする。
**参照:** 06_ワークフローと方法論 R-001, R-025〜R-035

### Boris Tane パターン
Cloudflare Engineering 発祥のカスタム Markdown プラン手法。Emphatic Research → Annotation Cycle → Mechanical Implementation の3フェーズで構成され、コンテキスト圧縮を生き延びる永続化マークダウンを使用する。
**参照:** 06_ワークフローと方法論 R-048〜R-053

### bypassPermissions
全パーミッションチェックをバイパスするモード。ただし deny ルールは引き続き有効であり、deny にマッチした操作はブロックされる。
**参照:** 01_基礎と設定 R-062, R-063

## C

### Cascade Method
→ Boris Tane パターンの Annotation Cycle 部分を指す。Claude がプランをドラフトし、人間がコメントで注釈、Claude がプランを更新するサイクルを全質問解決まで繰り返す手法。
**参照:** 06_ワークフローと方法論 R-050〜R-051

### CDD（Contract-Driven Development）
API 契約を実行可能なインターフェースとして定義する開発手法。マイクロサービスの並行開発に適する。
**参照:** 06_ワークフローと方法論 R-008

### CLAUDE.md
会話開始時に読み込まれる特別なファイル。Bash コマンド、コードスタイル、ワークフロールールなどの永続的コンテキストを提供する。グローバル（`~/.claude/CLAUDE.md`）、プロジェクトルート、親ディレクトリ、子ディレクトリに配置可能。200行を超えると最適パラメータを約33%超過し、遵守率が低下する。
**参照:** 01_基礎と設定 R-021〜R-032, 02_コンテキストエンジニアリング R-081〜R-088

### Compact（コンパクト）
`/compact` コマンドによるコンテキスト圧縮機能。コンテキスト使用率70%で実行すると利用可能スペースの約50%を回復する。会話が単一ブロックに圧縮され、キャッシュヒット率が回復するが、詳細な文脈やニュアンスは失われる。
**参照:** 02_コンテキストエンジニアリング R-039〜R-046

### Context Debt Score（コンテキスト負債スコア）
コンテキスト使用率に基づくパフォーマンス指標。Green（0〜50%: 最適）、Yellow（50〜75%: 安定）、Orange（75〜90%: 劣化開始）、Red（90%以上: 強制クリア必要）の4ゾーンで評価する。
**参照:** 02_コンテキストエンジニアリング R-064〜R-067

### Context Engineering（コンテキストエンジニアリング）
コンテキストを設計要素として扱う手法（Thoughtworks 2025）。LLM に供給する情報の量・質・タイミングを最適化し、パフォーマンスとコストのバランスを取る。
**参照:** 06_ワークフローと方法論 R-011

### Context Injection（コンテキストインジェクション）
v2.1.32以降の機能。`/rewind` を使用してチェックポイントから先を要約する。完全な `/compact` よりも外科的なアプローチ。`@path` インポートで外部ファイルを参照してコンテキストに注入することも含む。
**参照:** 02_コンテキストエンジニアリング R-062〜R-063

### Context Rot（コンテキスト腐敗）
会話履歴が長くなるにつれて古い・無関係・矛盾する情報が蓄積し、AI 応答品質が段階的に劣化する現象。65%でニュアンス喪失開始、75%で目に見える劣化、2時間以上のセッションで顕著になる。
**参照:** 02_コンテキストエンジニアリング R-051〜R-055

### Curse of Instructions（命令の呪い）
「より多くの命令 = 各ガイドラインへの遵守が悪化する」という現象。約150の異なるルールを超えるとモデルは一部を選択的に無視し始める。メカニズムは注意散逸（Attention Diffusion）。
**参照:** 06_ワークフローと方法論 セクション7

## D

### DDD（Domain-Driven Design）
ソフトウェアをビジネス言語に整合させる設計手法。ドメインが複雑な場面に適する。
**参照:** 06_ワークフローと方法論 R-005

### Dual-Instance Planning（二重インスタンス計画）
Claude Zero（Planner）と Claude One（Implementer）の2つのインスタンスを使い分ける手法。Planner がコードベース探索・プラン作成・レビューを、Implementer がコーディング・コミットを担当する。
**参照:** 06_ワークフローと方法論 R-054〜R-056

## E

### Effort Level（エフォートレベル）
モデルの推論の深さを制御する設定。low（高速・低コスト）、medium（デフォルト）、high（深い推論）、max（最も深い推論）の4段階。low/medium/high はセッション間で持続する。
**参照:** 01_基礎と設定 R-058〜R-060

### Eval-Driven Development
LLM の出力を evals（評価関数）でテストする開発手法。LLM ネイティブ製品の開発に適する。
**参照:** 06_ワークフローと方法論 R-013

## F

### FDD（Feature-Driven Development）
1フィーチャー最大2週間のイテレーションで進める開発手法。短サイクルデリバリーに適する。
**参照:** 06_ワークフローと方法論 R-010

### Fresh Context Pattern（Ralph Loop）
1時間以上の自律タスクに対して、外部状態ファイル（`TASK.md`, `PROGRESS.md`）を通じて新しいセッションで再開するパターン。各イテレーションで完全に新しい Claude Code プロセスを生成し、コンテキストウィンドウを0%にリセットする。タスクごとにフル200Kのアテンションを維持できる。
**参照:** 02_コンテキストエンジニアリング R-056〜R-058, 06_ワークフローと方法論 R-108〜R-111

## G

### GSD（Get Shit Done）
6フェーズワークフローでタスク毎に新鮮な200Kトークンコンテキストを確保する開発方法論。コンテキスト劣化対策に特化している。
**参照:** 06_ワークフローと方法論 R-002, R-020

## H

### Headless Mode（ヘッドレスモード）
Claude Code を UI なしでプログラマティックに実行するモード。CI/CD パイプラインや自動化スクリプトから SDK 経由で利用し、`--allowedTools` や `--permissionMode` でツール・権限を制御する。
**参照:** 01_基礎と設定 R-062

### Hooks（フック）
ライフサイクルの特定ポイント（PreToolUse、PostToolUse、SessionStart 等）で実行されるユーザー定義のシェルコマンド/ハンドラ。LLM の判断に依存せず決定論的に実行される（確実性100%）。command/http/prompt/agent の4タイプがある。
**参照:** 01_基礎と設定 R-005, R-031, R-040, 04_拡張機能 R-028〜R-063

## I

### Instruction Budget（インストラクション予算）
フロンティア LLM が確実に従える命令数の上限（約150-200）。システムプロンプトが約50を消費するため、ユーザーが使える実質的な枠は約100-150。
**参照:** 01_基礎と設定 R-026, 02_コンテキストエンジニアリング R-017〜R-018

### Instruction Debt（命令負債）
セッション間でプロジェクトの慣習を手動で繰り返し説明することで生じる累積的なコンテキストオーバーヘッド。未保存の判断1つあたり200〜500トークンのコストが再説明のたびに発生する。CLAUDE.md への一度の記述で解消できる。
**参照:** 02_コンテキストエンジニアリング R-068〜R-069

## J

### JiTTesting（Just-in-Time Testing）
PR 提出時に LLM がミューテーションテストを自動生成するエフェメラルテスト手法。マージ後にテストは破棄される。Meta 社の大規模導入で従来テスト比4倍のリグレッション捕捉を実現。
**参照:** 06_ワークフローと方法論 R-009, R-077〜R-078

## K

### Kill Switch（キルスイッチ）
エージェント暴走時の停止機構。3レベルで構成: (1) スコープ付き取り消し（特定機能の無効化）、(2) 速度ガバナー（5分間で20回以上のコマンドでブロック）、(3) グローバルハードストップ（Ctrl+C / Esc で即時終了）。
**参照:** 08_セキュリティ R-026〜R-029

## L

### Lead-Teammate アーキテクチャ
1つのチームリードが複数のチームメイトを調整するオーケストレーションパターン。Git ベース調整でロックファイルによるタスククレームと継続的 pull/push を行う。推奨上限はデフォルトで5エージェント以下。
**参照:** 06_ワークフローと方法論 R-112〜R-115

### Linear Decay（線形減衰）
命令数を2倍にすると遵守率が半減するという法則。低価値ルールの追加は高価値ルールの遵守率も一様に低下させる。
**参照:** 02_コンテキストエンジニアリング R-020〜R-021

### LSP（Language Server Protocol）サポート
プラグインを通じて Claude Code に定義へのジャンプ、参照の検索、型エラーの即座表示を提供する機能。TypeScript/JavaScript、Python、Rust は公式プリビルトプラグインあり。
**参照:** 04_拡張機能 R-069〜R-071

## M

### MCP（Model Context Protocol）
クライアント-サーバーアーキテクチャに基づく外部サービス接続プロトコル。stdio（ローカル）と Streamable HTTP + SSE（リモート）の2種類のトランスポートを持つ。サーバー1つにつき約2,000トークンのコンテキストオーバーヘッドが発生する。
**参照:** 01_基礎と設定 R-005, 02_コンテキストエンジニアリング R-003, 04_拡張機能 R-001〜R-012

### MCP Rug Pull
信頼確立後に良性 MCP サーバーが悪意あるものに変換される攻撃パターン。Phase 1 で良性公開→ Phase 2 で2週間正常動作→ Phase 3 で悪意ある更新をプッシュ（再承認不要）。ワンタイム承認モデルが根本原因。バージョンピンニングとハッシュ検証で緩和する。
**参照:** 08_セキュリティ R-021〜R-022

### MEMORY.md
Auto Memory が蓄積した知識を保存するファイル。最初の200行または最初の25KB（いずれか先に到達した方）のみが会話開始時に読み込まれる。
**参照:** 01_基礎と設定 R-045, 02_コンテキストエンジニアリング R-087

## P

### Path-Scoping（パススコーピング）
`@src/auth.ts:45-60` のように特定の行範囲を指定してファイルを読み込む手法。ファイル読み込みコストを約90%削減（約5,000トークンから約500トークンへ）。サブディレクトリ別 CLAUDE.md も含む。
**参照:** 02_コンテキストエンジニアリング R-032〜R-034

### Plan Mode（プランモード）
Shift+Tab 2回押下で開始する読み取り専用モード。Claude が関連ファイルを読みパターンを検索するが変更は行わない。プランは `.claude/plans/[name].md` に生成され、人間がレビュー後に実装を開始する。
**参照:** 01_基礎と設定 R-062, 06_ワークフローと方法論 R-039〜R-047

### Plugins（プラグイン）
Skills、Agents、Hooks、MCP サーバー、LSP サーバーをバンドルして配布可能なパッケージ。公式マーケットプレイスから `/plugin` コマンドでインストール可能。カスタムマーケットプレイスの作成も可能。
**参照:** 04_拡張機能 R-064〜R-073

### Progressive Disclosure（段階的開示）
SKILL.md を500行以内に収め、詳細リファレンスを `references/` 等の別ファイルに分割するパターン。大量のドキュメントを事前にコンテキストに読み込まずに済み、必要時のみ追加ファイルを読み込む。
**参照:** 04_拡張機能 R-019〜R-021

### Prompt Injection（プロンプトインジェクション）
PDF、スクリーンショット、メール添付等にインジェクションを埋め込む攻撃手法。ゼロ幅文字、右から左オーバーライド、ANSI エスケープシーケンス、コメント内 Base64 等の回避テクニックが存在する。5段階の防御戦略で対応する。
**参照:** 08_セキュリティ R-001, R-009〜R-013

## R

### Ralph Loop
→ Fresh Context Pattern を参照。Geoffrey Huntley のワークフローパターンで「生成→レビュー→修正→繰り返し」の反復パターン。常にイテレーション上限を設定し、15-20イテレーション以降はコンテキスト長制約が顕在化する。
**参照:** 06_ワークフローと方法論 R-108〜R-111

## S

### Sandbox（サンドボックス）
Claude Code の実行環境を分離する機構。ファイルシステム分離（CWD とサブディレクトリに書き込み制限）、ネットワーク分離（ドメイン接続制限）、OS レベルプリミティブ（macOS Seatbelt、Linux bubblewrap）で構成。権限プロンプトを84%削減する。
**参照:** 08_セキュリティ R-019, R-043〜R-046

### Santa Method
→ SDD（Spec-Driven Development）のコミュニティ別名。コード実装前に仕様を記述し、仕様が「サンタのリスト」のように実装を導くアプローチ。
**参照:** 06_ワークフローと方法論 R-003

### SDD（Spec-Driven Development）
コード実装前に仕様を記述する開発手法。グリーンフィールドプロジェクトやガバナンスが求められる場面に適する。Spec Kit、OpenSpec、Specmatic 等のツールがある。
**参照:** 06_ワークフローと方法論 R-003, R-036〜R-038

### Skills（スキル）
Claude Code の機能を拡張するモジュール式の能力パッケージ。`.claude/skills/<name>/SKILL.md` に定義し、トリガーされると Bash 経由で読み込まれコンテキストウィンドウに入る。確実性は50-80%で、コンテキストコストは低〜中。
**参照:** 01_基礎と設定 R-005, R-030, R-076〜R-077, 04_拡張機能 R-013〜R-027

### Strategic Compact（戦略的コンパクト）
コンテキスト使用率が一定の閾値に達した時点で計画的に `/compact` を実行する手法。推奨閾値は70%（多くの実践者は60%を共有）。タスク途中での無計画な自動圧縮を回避し、コンテキスト品質を維持する。
**参照:** 02_コンテキストエンジニアリング R-039〜R-044

### Subagent（サブエージェント）
Task tool で生成される隔離されたサブエージェント。独自のコンテキストウィンドウを持ち、タスクの異なる部分を並列処理する。組み込み（Explore、Plan、General-purpose）とカスタム（`.claude/agents/` に定義）がある。`isolation: worktree` で独立した worktree を作成可能。
**参照:** 01_基礎と設定 R-004, R-005, R-079, 04_拡張機能 R-074〜R-081

## T

### TDD（Test-Driven Development）
Red-Green-Refactor サイクルで品質重視の実装を行う手法。Claude は本来「実装を先に書きテストで検証」する傾向があるため、FAILING テストを明示的に要求する必要がある。1機能 = 1 TDD サイクルを厳守する。
**参照:** 06_ワークフローと方法論 R-012, R-067〜R-076

### Tool Poisoning（ツール汚染）
MCP ツールスキーマ内の悪意あるメタデータが LLM の動作に影響を与える攻撃手法。MCP 固有のインジェクションパターンの一つ。
**参照:** 08_セキュリティ R-011

### Tool Search（ツールサーチ）
MCP ツールの遅延読み込み機能。セッション開始時にはツール名のみがコンテキストに読み込まれ、必要になった時点でスキーマが読み込まれる。`ENABLE_TOOL_SEARCH=auto` でコンテキストウィンドウの10%以内ならスキーマを upfront 読み込みする。Sonnet 4以降、Opus 4以降が対応。
**参照:** 04_拡張機能 R-007〜R-009

### Trigger-Table Lazy Loading（トリガーテーブル遅延読み込み）
全てのプロジェクトコンテキストを起動時にロードせず、CLAUDE.md にトリガー条件を記載してオンデマンドでロードする手法。ベースラインのコンテキスト消費を20〜30%削減する。
**参照:** 02_コンテキストエンジニアリング R-070〜R-072

## U

### UVAL Protocol
→ Anti-rationalization Gate の一部として機能する検証プロトコル。Claude の出力が実際のタスク完了基準を満たしているかを外部エージェントが評価し、合理化（不完全な作業を完了と宣言すること）を防止する。
**参照:** 06_ワークフローと方法論 R-096〜R-097, 08_セキュリティ R-047

## W

### Worktree（ワークツリー）
Git worktree を活用したサブエージェントの分離機構。`isolation: worktree` でサブエージェントごとに独立した worktree を作成し、ファイル競合を防止する。変更なしで完了時は自動クリーンアップされる。`orchestrate-worktrees.js` で複数ワーカーが個別 worktree を使用する構成もある。
**参照:** 04_拡張機能 R-080, 06_ワークフローと方法論 R-116〜R-117

### 20%ルール
ルール配置の基準。セッション出現頻度20%以上のルールは CLAUDE.md に（毎セッション自動ロード、追加コスト0）、5〜20%はスキルモジュールに（オンデマンド、500〜1,000トークン）、5%未満はスラッシュコマンドに配置する。
**参照:** 02_コンテキストエンジニアリング R-035〜R-038

### 30行ルール
CLAUDE.md を200行から約30行に削減するベストプラクティス。Claude が既存コードから推測できないものだけを含め、「これを削除したら Claude は間違うか？」をテストする。
**参照:** 02_コンテキストエンジニアリング R-090
