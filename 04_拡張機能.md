## 1. MCP（Model Context Protocol）

### 1.1 アーキテクチャ

R-001: MCPはクライアント-サーバーアーキテクチャに基づく。Claude Code（MCPホスト）が各MCPサーバーに対して1つのMCPクライアントを作成し接続を確立する。

R-002: トランスポートプロトコルは2種類ある。

| プロトコル | 用途 | 特徴 |
|---|---|---|
| **stdio** | ローカルプロセス間通信 | サブプロセスとしてサーバーを起動。stdin/stdoutでJSON-RPC交換。単一クライアント向け |
| **Streamable HTTP + SSE** | リモートサーバー通信 | HTTP POST + SSEストリーミング。Bearer Token等の標準HTTP認証対応。複数クライアント対応 |

> 情報源: [MCP Architecture](https://modelcontextprotocol.io/docs/learn/architecture), [MCP Transports](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports)

### 1.2 設定スコープ

R-003: MCPサーバーは3つのスコープで設定可能。

| スコープ | 保存先 | 用途 |
|---|---|---|
| **Local**（デフォルト） | `~/.claude.json` のプロジェクトパス配下 | 個人用、現プロジェクトのみ |
| **Project** | `.mcp.json` | チーム共有、VCS管理可能 |
| **User** | `~/.claude.json` | 全プロジェクト共通の個人設定 |

R-004: CLIでの追加コマンド:
```bash
claude mcp add <server-name> -- <command> [args...]
claude mcp add --scope project my-server -- npx -y @modelcontextprotocol/server-filesystem /path
claude mcp add --scope user my-server -- node server.js
```

R-005: `.mcp.json` の設定例:
```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/project"],
      "env": {}
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxxx"
      }
    },
    "brave-search": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-brave-search"],
      "env": {
        "BRAVE_API_KEY": "BSA_xxxx"
      }
    }
  }
}
```

> 情報源: [Claude Code MCP Docs](https://code.claude.com/docs/en/mcp), [Claude Code Settings](https://code.claude.com/docs/en/settings)

| **Filesystem** | セキュアなファイル操作、アクセス制御設定可能 |

| **Brave Search** | Web検索・ローカル検索 |

> 情報源: [Top 10 MCP Servers](https://apidog.com/blog/top-10-mcp-servers-for-claude-code/), [MCP Servers Repository](https://github.com/modelcontextprotocol/servers)

### 1.4 Tool Search（遅延ツール読み込み）

R-007: MCPツールはデフォルトで遅延読み込み（deferred）される。セッション開始時にはツール名のみがコンテキストに読み込まれ、必要になった時点でスキーマが読み込まれる。

R-008: `ENABLE_TOOL_SEARCH=auto` でコンテキストウィンドウの10%以内に収まる場合はスキーマをupfront読み込み、溢れる分のみ遅延する。

R-009: Tool Search対応モデルはSonnet 4以降、Opus 4以降。Haikuは非対応。

R-010: MCPツール出力が10,000トークンを超えると警告表示。`MAX_MCP_OUTPUT_TOKENS`環境変数で最大出力トークン数を調整可能（デフォルト25,000）。

R-011: MCPツールの命名規則:
```
mcp__<server名>__<tool名>
例: mcp__github__search_repositories
     mcp__filesystem__read_file
```

> 情報源: [Claude Code MCP](https://code.claude.com/docs/en/mcp), [Manage Costs](https://code.claude.com/docs/en/costs)

### 1.5 デバッグ

R-012: MCPサーバーの管理コマンド:
```bash
claude mcp list           # 状態確認
claude mcp get <name>     # 詳細表示
claude mcp remove <name>  # 削除
```

> 情報源: [Claude Code MCP Docs](https://code.claude.com/docs/en/mcp)

## 2. Skills

### 2.1 概要

R-013: SkillsはClaude Codeの機能を拡張するモジュール式の能力パッケージ。Agent Skills（agentskills.io）のオープンスタンダードに準拠。

R-014: Skillの動作原理: トリガーされるとClaudeがBashで`SKILL.md`をファイルシステムから読み込み、指示内容がコンテキストウィンドウに入る。指示が他のファイルを参照していれば追加でそれらも読み込む。

> 情報源: [Agent Skills Specification](https://agentskills.io/specification), [Claude Code Skills](https://code.claude.com/docs/en/skills)

### 2.2 SKILL.md 仕様

R-015: ディレクトリ構造:
```
.claude/skills/
  my-skill/
    SKILL.md          # 必須: スキル定義
    scripts/           # オプション: ヘルパースクリプト
    references/        # オプション: 参照ドキュメント
    assets/            # オプション: リソースファイル
```

R-016: 配置場所は2つ。プロジェクトスキル: `.claude/skills/`（VCS管理可能、チーム共有）。ユーザースキル: `~/.claude/skills/`（全プロジェクト共通の個人スキル）。

### 2.3 フロントマター全項目

R-017: SKILL.md フロントマター仕様:
```yaml
---
name: my-skill-name              # スキル名（必須推奨）
description: |                   # スキルの説明と使用タイミング（必須推奨）
  このスキルは〇〇の時に使用する
disable-model-invocation: true   # true: ユーザーのみ呼び出し可（Claudeは呼べない）
user-invocable: false            # false: Claudeのみ呼び出し可（ユーザーは呼べない）
effort: high                     # モデルのeffortレベルをオーバーライド
model: sonnet                    # 使用モデル（sonnet/opus/haiku/full ID/inherit）
---
```

R-018: SKILL.md フロントマター完全版の例:
```yaml
---
name: testing-patterns
description: |
  テストパターンとベストプラクティスを提供する。
  "テスト" "test" "Jest" "Vitest" に関する作業時に使用。
disable-model-invocation: false
user-invocable: true
effort: medium
model: inherit
---

# テストパターンガイド

## ユニットテスト
テストは `__tests__/` ディレクトリに配置する。
詳細なパターン集は `references/test-patterns.md` を参照。
```

> 情報源: [Claude Code Skills](https://code.claude.com/docs/en/skills), [Skills Specification](https://agentskills.io/specification)

### 2.4 Progressive Disclosure（段階的開示）

R-019: SKILL.mdは500行以内に収める。超過時は詳細リファレンスを`references/`等の別ファイルに分割する。

R-020: Progressive Disclosureにより、大量のリファレンスドキュメントを事前にコンテキストに読み込まずに済む。

R-021: 段階的開示パターン:
```markdown
# SKILL.md（簡潔に保つ）
API設計ガイドライン。詳細は `references/api-spec.md` を参照。

# references/api-spec.md（必要時のみ読み込み）
## 詳細なAPIエンドポイント仕様
...（大量のドキュメント）
```

### 2.5 トリガー設計

R-022: Skillのトリガー方法は3つ: (1) Claudeによる自動発見（descriptionに基づく）、(2) ユーザーによるスラッシュコマンド `/skill-name`、(3) サブエージェントの`skills`フロントマターで起動時に注入。

R-023: スキル推薦エンジン（ChrisWiles/claude-code-showcase）: UserPromptSubmitフックでプロンプトを解析し、キーワード（2点）、正規表現パターン（4点）、ディレクトリマッチ（5点）でスコアリング。`skill-eval.sh` → `skill-eval.js` → `skill-rules.json` の構成。

> 情報源: [ChrisWiles/claude-code-showcase](https://github.com/ChrisWiles/claude-code-showcase)

### 2.6 注意点

R-024: SKILL.mdが500行を超えるとパフォーマンスが低下する。

R-025: Claudeのシステムプロンプトは既に約50の個別指示を含む。スキルの追加でさらに増えるため、指示は最小限にする。

R-026: `disable-model-invocation: true`を設定しないと、Claudeが意図しないタイミングでスキルを呼ぶことがある。

R-027: スキルの読み込みはBash経由のため、パスの問題に注意する。

> 情報源: [Claude Code Skills](https://code.claude.com/docs/en/skills)

## 3. Hooks

### 3.1 概要

R-028: Hooksはライフサイクルの特定ポイントで実行されるユーザー定義のシェルコマンド/ハンドラ。決定論的な制御を提供し、LLMの判断に依存せず確実に実行される。

R-029: 4つのハンドラタイプ:

| タイプ | 説明 |
|---|---|
| **command** | シェルコマンドを実行 |
| **http** | HTTPリクエストを送信 |
| **prompt** | Claudeにプロンプトを注入 |
| **agent** | エージェントを起動 |

> 情報源: [Hooks Reference](https://code.claude.com/docs/en/hooks), [Hooks Guide](https://code.claude.com/docs/en/hooks-guide)

### 3.3 Exit Code 制御フロー

R-031: Hookは2つのアプローチで制御を行う（同時使用不可）。

R-032: 方式1 - Exit Codeのみ: `Exit 0` → 続行（ツール実行を許可）、`Exit 2` → ブロック（ツール実行を拒否）。

R-033: 方式2 - JSON出力（exit 0 + stdout JSON）: JSONの内容に基づく制御。Exit 2の場合JSONは無視される。

R-034: PreToolUseのJSON制御（hookSpecificOutput）:
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "allow",
    "permissionDecisionReason": "安全な操作です"
  }
}
```

R-035: permissionDecisionの値: `"allow"`（許可）、`"deny"`（拒否）、`"ask"`（ユーザーにエスカレーション）。

R-036: PostToolUse等のJSON制御（トップレベル）:
```json
{
  "decision": "block",
  "reason": "この操作は許可されていません"
}
```

R-037: 制御フロー図:
```
Hook実行
  ├── Exit 2 → 即座にブロック（JSON無視）
  └── Exit 0
       ├── stdout にJSON なし → 続行
       └── stdout にJSON あり
            ├── PreToolUse → hookSpecificOutput.permissionDecision で判定
            └── その他 → トップレベル decision で判定
```

> 情報源: [Hooks Reference](https://code.claude.com/docs/en/hooks)

### 3.4 settings.json のHooks設定例

R-038: Hooks設定の全体構造例:
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.command' | grep -q 'rm -rf /' && exit 2 || exit 0",
            "timeout": 5
          }
        ]
      },
      {
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | grep -q '\\.env' && exit 2 || exit 0"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write $(jq -r '.tool_input.file_path')"
          }
        ]
      },
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.command' >> ~/.claude/command-log.txt"
          }
        ]
      }
    ],
    "UserPromptSubmit": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "uv run $CLAUDE_PROJECT_DIR/.claude/hooks/user_prompt_submit.py --log-only"
          }
        ]
      }
    ]
  }
}
```

### 3.6 UV Single-File Script パターン

R-060: UV single-fileスクリプトを使うと、仮想環境やrequirements.txtの管理なしにPythonフックを実行できる。

R-061: shebangパターン:
```python
#!/usr/bin/env -S uv run --script
# /// script
# requires-python = ">=3.11"
# dependencies = ["requests", "pyyaml"]
# ///

import sys
import json

# stdinからフックのJSON入力を読み取り
data = json.load(sys.stdin)
tool_name = data.get("tool_name", "")

# 処理ロジック
if "dangerous" in tool_name:
    print(json.dumps({
        "hookSpecificOutput": {
            "permissionDecision": "deny",
            "permissionDecisionReason": "Dangerous tool detected"
        }
    }))
else:
    sys.exit(0)
```

R-062: settings.json での設定:
```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "*",
      "hooks": [{
        "type": "command",
        "command": "uv run $CLAUDE_PROJECT_DIR/.claude/hooks/pre_tool_use.py"
      }]
    }]
  }
}
```

R-063: UV single-file scriptの利点: 仮想環境管理不要、requirements.txt不要、依存関係がスクリプトファイル内に自己完結、`$CLAUDE_PROJECT_DIR`でプロジェクトルートからの相対パス解決。

> 情報源: [pydevtools.com - Claude Code Hooks for uv](https://pydevtools.com/blog/claude-code-hooks-for-uv/), [disler/claude-code-hooks-mastery](https://github.com/disler/claude-code-hooks-mastery)

R-064: PluginsはSkills、Agents、Hooks、MCPサーバー、LSPサーバーをバンドルして配布可能なパッケージ。

R-065: プラグインのコンポーネント:

### 4.2 マーケットプレイス

R-066: 公式マーケットプレイス（claude-plugins-official）はClaude Code起動時に自動的に利用可能。`/plugin` コマンドの Discover タブからインストール可能。

R-067: マーケットプレイスは集中的なディスカバリ、バージョン追跡、自動更新、複数ソースタイプ（gitリポジトリ、ローカルパス等）をサポート。

## 5. サブエージェント

### 5.1 組み込みサブエージェント

R-074: Claude Codeには以下の組み込みサブエージェントが含まれる:

| サブエージェント | 説明 | ツール |
|---|---|---|
| **Explore** | 高速・読み取り専用。コードベース検索・分析。thoroughnessレベル: quick/medium/very thorough | 読み取り系のみ |
| **Plan** | planモード用リサーチエージェント。プラン提示前にコンテキスト収集 | 読み取り系のみ |
| **General-purpose** | 複雑なマルチステップタスク。探索と変更の両方が必要な場合 | 全ツール |
| **その他のヘルパーエージェント** | 特定タスク用（自動的に呼び出される） | タスクに応じて |

> 情報源: [Create Custom Subagents](https://code.claude.com/docs/en/sub-agents)

### 5.2 カスタムサブエージェント定義

R-075: 配置場所: プロジェクト `.claude/agents/my-agent.md`（VCS管理可能）、ユーザー `~/.claude/agents/my-agent.md`（全プロジェクト共通）。

### 5.3 フロントマター全16項目

R-076: サブエージェントのフロントマター全項目:
```yaml
---
# === 必須フィールド ===
name: code-reviewer                    # 一意の識別子（小文字+ハイフン）
description: |                         # Claudeがこのエージェントに委任するタイミング
  コードレビューを行う。PRの変更を分析し、
  バグ、セキュリティ問題、改善提案を提示する。

# === モデル・パフォーマンス ===
model: sonnet                          # sonnet/opus/haiku/full ID/inherit（デフォルト: inherit）
effort: high                           # モデルのeffortレベル
maxTurns: 20                           # 最大ターン数

# === ツール制御 ===
tools:                                 # 使用可能ツール（省略時は全ツール継承）
  - Read
  - Grep
  - Glob
  - Bash
disallowedTools:                       # 拒否ツール（tools/継承リストから除去）
  - Edit
  - Write

# === 権限・セキュリティ ===
permissionMode: plan                   # default/acceptEdits/dontAsk/bypassPermissions/plan

# === 外部接続 ===
mcpServers:                            # 利用可能なMCPサーバー
  github:
    command: npx
    args: ["-y", "@modelcontextprotocol/server-github"]
skills:                                # 起動時に注入するスキル
  - testing-patterns
  - code-style

# === ライフサイクル ===
hooks:                                 # サブエージェントのライフサイクルフック
  PostToolUse:
    - matcher: Edit
      hooks:
        - type: command
          command: "npx prettier --write $(jq -r '.tool_input.file_path')"

# === 高度な設定 ===
memory: persistent                     # メモリ設定
background: |                          # バックグラウンドコンテキスト
  このプロジェクトはTypeScript + React。
  ESLintとPrettierで品質管理。
isolation: worktree                    # 分離設定（worktree）

# === プロンプト（frontmatter外） ===
# initialPrompt は --agents フラグのJSON形式でのみ使用
---

# コードレビューエージェント

## 手順
1. `git diff` で変更を確認
2. 各ファイルの変更を分析
3. バグ、セキュリティ問題、改善点を報告
4. レビュー結果をMarkdownで出力
```

### 5.4 設計原則

R-077: 最小権限の原則: 必要なツールのみを付与（`tools` / `disallowedTools`で制御）。

R-078: スキル注入: `skills`で起動時にドメイン知識を注入（実行中にスキル発見する必要がない）。

R-079: ライフサイクルフック: `hooks`でサブエージェント固有のフックを定義（そのサブエージェントがアクティブな時のみ実行）。

R-080: worktree分離: `isolation: worktree` でサブエージェントごとに独立したworktreeを作成（変更なしで完了時は自動クリーンアップ）。

R-081: プラグインサブエージェントの制限: プラグインからのエージェントは `hooks`, `mcpServers`, `permissionMode` が無視される。

> 情報源: [Create Custom Subagents](https://code.claude.com/docs/en/sub-agents), [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice)

## 6. エージェントチーム

### 6.1 概要

R-082: エージェントチームは複数のClaude Codeインスタンスを協調動作させる実験的機能（デフォルトで無効）。

R-083: 有効化方法:
```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```
または settings.json で設定。

### 6.2 仕組み

R-084: 1つのセッションがチームリードとして機能し、作業の調整・タスク割り当て・結果統合を行う。

R-085: チームメイトは独立して動作し、それぞれ独自のコンテキストウィンドウを持つ。チームメイト間で直接メッセージングが可能。

R-086: サブエージェントとの違い: サブエージェントは単一セッション内で動作しメインエージェントにのみ報告するが、チームメイトには個別に直接対話可能。

### 6.3 最適なユースケース

R-087: エージェントチームの最適なユースケース:

| ユースケース | 説明 |
|---|---|
| **リサーチ＆レビュー** | 複数のチームメイトが問題の異なる側面を同時調査、発見を共有・挑戦 |
| **新モジュール/機能** | チームメイトがそれぞれ別の部分を担当（干渉なし） |
| **競合仮説デバッグ** | チームメイトが並列に異なる仮説をテスト、早期収束 |
| **クロスレイヤー調整** | フロントエンド、バックエンド、テストの変更を各チームメイトが担当 |

### 6.4 コスト

R-088: 標準セッション比で約7倍のトークン使用（planモードのチームメイト）。各チームメイトが独自のコンテキストウィンドウを維持し、アクティブチームメイト数と各チームメイトの実行時間に比例してスケール。

### 6.5 コスト管理戦略

R-089: チームメイトにはSonnetを使用（能力とコストのバランス）。

R-090: 作業完了後はチームをクリーンアップする（アイドル状態でもトークン消費継続）。

R-091: タスクを小さく自己完結的に保つ。

### 6.6 ガードレール

R-092: プロンプト自体にガードレールを追加して制御。各タスクに独自のpermissionModeを設定可能。worktree分離でファイル競合を防止。

### 6.7 既知の制限事項

R-093: 既知の制限: セッション再開の問題、タスクステータスの遅延、シャットダウンの遅延、1セッションにつき1チームのみ、ネストしたチームは不可。

> 情報源: [Agent Teams](https://code.claude.com/docs/en/agent-teams), [Manage Costs](https://code.claude.com/docs/en/costs)

## 7. 拡張レイヤーの確実性比較

R-094: 各拡張レイヤーの確実性:

| レイヤー | 確実性 | 説明 |
|---|---|---|
| **CLAUDE.md** | ~80% | LLMの指示追従に依存。システムプロンプトに約50の指示が既にあり、追加指示が多すぎると追従率が低下 |
| **Skills** | 50-80% | Claudeがdescriptionに基づき自動発見するため、適切なタイミングで読み込まれない可能性あり。`disable-model-invocation`で制御可能 |
| **Hooks** | **100%** | 決定論的実行。イベント発火時に毎回必ず実行される。プロンプトの表現やモデルの判断に関係なく確実に動作 |
| **Subagents** | ~80-90% | Claudeの委任判断に依存するが、明示的な呼び出しも可能 |

R-095: 設計指針:
- **絶対に守るべきルール** → Hooks（exit code 2でブロック）
- **強く推奨するガイドライン** → CLAUDE.md
- **ドメイン知識の提供** → Skills
- **複雑なタスクの委任** → Subagents

> 情報源: [Extend Claude Code](https://code.claude.com/docs/en/features-overview), [Best Practices](https://code.claude.com/docs/en/best-practices), [HumanLayer Blog](https://www.humanlayer.dev/blog/writing-a-good-claude-md)

## 8. コンテキストコスト比較

R-096: 各拡張レイヤーのコンテキストコスト:

| レイヤー | コンテキストコスト | 詳細 |
|---|---|---|
| **MCP** | **最大（高い）** | ツール定義のスキーマがコンテキストに入る。Tool Searchで軽減可能だが、使用するとスキーマ+結果がコンテキストを消費。MAX_MCP_OUTPUT_TOKENS=25,000がデフォルト上限 |
| **Agent定義** | **中〜高** | サブエージェントのプロンプト・設定がコンテキストに入る。ただしサブエージェントは独自のコンテキストウィンドウで動作するため、メインウィンドウへの影響は委任指示分のみ |
| **Skills** | **低〜中** | SKILL.md読み込み時のみコンテキスト消費。Progressive Disclosureで必要な時だけ追加ファイルを読み込み。500行以内推奨 |
| **Hooks** | **ほぼゼロ** | コンテキスト外で実行される。JSON入出力のみで、コンテキストウィンドウを消費しない |
| **CLAUDE.md** | **低** | セッション開始時に一度読み込まれる。簡潔に保つことが重要 |

R-097: コスト順:
```
MCP >> Agent定義 >> Skills >> CLAUDE.md >> Hooks ≒ 0
```

R-098: 最適化戦略:
- MCP: Tool Search（デフォルト有効）でツール名のみ先行読み込み
- Skills: Progressive Disclosureで段階的読み込み
- CLAUDE.md: 普遍的に適用される最小限の指示のみ記載
- Hooks: コンテキストコストなし、積極的に活用すべき

> 情報源: [Manage Costs](https://code.claude.com/docs/en/costs), [Claude Code MCP](https://code.claude.com/docs/en/mcp), [How Claude Code Works](https://code.claude.com/docs/en/how-claude-code-works)

## 9. Trail of Bits セキュリティ設定参考

R-099: 危険コマンドの拒否対象: `rm -rf`, `sudo`, `mkfs`, `dd`, force git push。

R-100: 機密ファイルへのアクセス拒否対象:
- `~/.ssh/**`, `~/.gnupg/**` — SSH鍵・暗号資格情報
- `~/.aws/**`, `~/.azure/**` — クラウドプロバイダー資格情報
- `~/.npmrc`, `~/.pypirc` — パッケージマネージャー資格情報
- `~/.bashrc`, `~/.zshrc` — シェル設定ファイル

R-101: サンドボックスの3段階:
- `/sandbox`なし: deny ルールはClaude組み込みツールのみブロック、Bashコマンドはバイパス可能
- `/sandbox`あり: OSレベル（Seatbelt/bubblewrap）で同じルールを強制、Bashもブロック
- 完全分離: devcontainerを使用

> 情報源: [trailofbits/claude-code-config](https://github.com/trailofbits/claude-code-config), [trailofbits/claude-code-devcontainer](https://github.com/trailofbits/claude-code-devcontainer)

## extension-evaluator用 評価チェックリスト

### MCP設定チェック
- [ ] トランスポートプロトコルは用途に適切か（ローカル→stdio、リモート→Streamable HTTP+SSE）（R-002）
- [ ] スコープ設定は適切か（個人用→Local/User、チーム共有→Project）（R-003）
- [ ] 環境変数にシークレットを直接ハードコードしていないか（R-005）
- [ ] Tool Searchが有効か（デフォルト有効）。不要なスキーマのupfront読み込みがないか（R-007, R-008）
- [ ] MAX_MCP_OUTPUT_TOKENSの値は適切か（R-010）
- [ ] 使用モデルはTool Search対応か（Sonnet 4以降/Opus 4以降）（R-009）

### Skills設定チェック
- [ ] SKILL.mdは500行以内か（R-019, R-024）
- [ ] name と description フロントマターが記載されているか（R-017）
- [ ] 大量のリファレンスはProgressive Disclosureで分離されているか（R-020, R-021）
- [ ] 意図しない自動呼び出しを防ぐため `disable-model-invocation` を検討したか（R-026）
- [ ] 指示は最小限か（既存システムプロンプトの約50指示に追加される点を考慮）（R-025）
- [ ] パス指定が正しいか（Bash経由読み込みのため）（R-027）

### Hooks設定チェック
- [ ] セキュリティ系Hookが設定されているか（rm -rf, .env, sudo, force push等）（R-039〜R-043）
- [ ] Exit Code制御フローを正しく理解して実装しているか（R-031〜R-037）
- [ ] PreToolUseではhookSpecificOutput形式、その他ではトップレベルdecision形式を使用しているか（R-034, R-036）
- [ ] timeoutが適切に設定されているか（R-038）
- [ ] 品質管理系Hook（フォーマット、Lint、型チェック）が必要に応じて設定されているか（R-044〜R-047）
- [ ] UV single-file scriptを使う場合、shebangとscriptメタデータが正しいか（R-061）

### サブエージェント設定チェック
- [ ] 最小権限の原則に従っているか（tools/disallowedToolsの設定）（R-077）

## 拡張レイヤー使い分け判断フロー

```
要件を分析
│
├─ 「毎回確実に実行されなければならない」か？
│   │
│   ├─ YES → Hooks を使用（確実性100%、コンテキストコストほぼゼロ）（R-094, R-096）
│   │   │
│   │   ├─ 操作をブロックしたい → PreToolUse + exit 2（R-032）
│   │   ├─ 操作後に自動処理 → PostToolUse + command（R-044〜R-047）
│   │   ├─ 入力を検証したい → UserPromptSubmit（R-030）
│   │   └─ 通知・ログ → SessionEnd / Stop + command（R-055, R-057）
│   │
│   └─ NO → 次の判断へ
│
├─ 「外部ツール/サービスとの接続」が必要か？
│   │
│   ├─ YES → MCP を使用（コンテキストコスト最大、Tool Searchで軽減）（R-001, R-096）
│   │   │
│   │   ├─ ローカル → stdio トランスポート（R-002）
│   │   └─ リモート → Streamable HTTP + SSE（R-002）
│   │
│   └─ NO → 次の判断へ
│
├─ 「ドメイン知識/ベストプラクティス」の提供か？
│   │
│   ├─ YES → Skills を使用（コンテキストコスト低〜中）（R-013, R-096）
│   │   │
│   │   ├─ 特定タイミングのみ必要 → disable-model-invocation: true + スラッシュコマンド（R-017）
│   │   ├─ 自動的に適用したい → description を明確に記述（R-022）
│   │   └─ 大量のリファレンス → Progressive Disclosure（R-019〜R-021）
│   │
│   └─ NO → 次の判断へ
│
├─ 「複雑なマルチステップタスクの委任」か？
│   │
│   ├─ YES → サブエージェント を使用（R-074〜R-081）
│   │   │
│   │   ├─ 読み取り専用の調査 → Explore エージェント相当（tools を読み取り系に限定）（R-074）
│   │   ├─ ファイル変更を含む → tools に Edit/Write を含める（R-076）
│   │   └─ 並列実行が必要 → isolation: worktree を設定（R-080）
│   │
│   └─ NO → 次の判断へ
│
├─ 「複数エージェントの並列協調」が必要か？
│   │
│   ├─ YES → エージェントチーム を使用（実験的、コスト約7倍）（R-082, R-088）
│   │
│   └─ NO → 次の判断へ
│
├─ 「複数の拡張をバンドルして配布」したいか？
│   │
│   ├─ YES → Plugins を使用（R-064）
│   │
│   └─ NO → CLAUDE.md に記載（確実性~80%、コンテキストコスト低）（R-094）
│
└─ 複数レイヤーの組み合わせを検討
    ├─ セキュリティルール → Hooks（必須）+ CLAUDE.md（ガイドライン補足）
    ├─ コード品質 → Hooks（自動フォーマット/Lint）+ Skills（ベストプラクティス）
    └─ 複雑なワークフロー → Subagents（タスク委任）+ Skills（ドメイン知識注入）+ Hooks（ライフサイクル制御）
```
