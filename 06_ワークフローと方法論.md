## 1. 開発方法論の分類と選定

### 1.1 6ティア分類体系

15種の開発方法論を6階層に分類する。上位ティアほど戦略的、下位ほど実装寄り。

**Tier 1: 戦略的オーケストレーション**

- R-001: BMAD（Business-driven Methodical AI Development）はマルチエージェント・ガバナンスとコンスティチューション・ガードレールを提供する。エンタープライズ（10+チーム）向け
- R-002: GSD（Get Shit Done）は6フェーズワークフローでタスク毎に新鮮な200kトークンコンテキストを確保する。コンテキスト劣化対策向け

**Tier 2: 仕様・アーキテクチャ**

- R-003: SDD（Spec-Driven Development）はコード実装前に仕様を記述する。グリーンフィールド・ガバナンス向け
- R-004: Doc-Driven はリビング・ドキュメンテーションを唯一の真実の情報源とする。ドキュメント中心チーム向け
- R-005: DDD（Domain-Driven Design）はソフトウェアをビジネス言語に整合させる。ドメイン複雑系向け

**Tier 3: 振る舞い・受入**

- R-006: BDD（Behavior-Driven Development）は Given-When-Then シナリオでステークホルダーと協業する。受入基準重視向け
- R-007: ATDD（Acceptance TDD）はコーディング前に受入基準を協調定義する。ステークホルダー合意向け
- R-008: CDD（Contract-Driven Development）は API 契約を実行可能なインターフェースとして定義する。マイクロサービス並行開発向け
- R-009: JiTTesting（Just-in-Time Testing）は PR 提出時に LLM がエフェメラルテストを自動生成し、マージ後に破棄する。Meta 社が大規模導入、従来テスト比4倍のリグレッション捕捉

**Tier 4: フィーチャーデリバリー**

- R-010: FDD（Feature-Driven Development）は1フィーチャー最大2週間のイテレーション。短サイクルデリバリー向け
- R-011: Context Engineering はコンテキストを設計要素として扱う（Thoughtworks 2025）

**Tier 5: 実装**

- R-012: TDD（Test-Driven Development）は Red-Green-Refactor サイクル + 検証ループで品質重視の実装を行う
- R-013: Eval-Driven は LLM 出力を evals でテストする。LLM ネイティブ製品向け
- R-014: Multi-Agent Orchestration はオーケストレーションされたサブエージェントで複雑タスクを処理する

**Tier 6: 最適化**

- R-015: 反復的リファインメントループ、フレッシュコンテキストパターン（Ralph Loop）、プロンプトエンジニアリング基盤を組み合わせて最適化を行う

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/core/methodologies.md`

## 2. Plan-First 開発

### 2.1 Plan Mode（ネイティブ）

- R-039: 基本原則は「計画が良ければコードも良い（Once the plan is good, the code is good）」
- R-040: Plan Mode は Shift+Tab 2回押下で開始（Normal → Auto-Accept → Plan）
- R-041: Plan Mode では Claude は読み取り専用モードで関連ファイルを読み、パターンを検索する（変更不可）
- R-042: プランは `.claude/plans/[name].md` に生成される
- R-043: 人間がレビュー後「Proceed with the plan」で実装開始

**プラン適用基準**:

- R-044: API の変更、DB マイグレーション、新機能（3+ファイル or 50行以上の変更）にはプラン必須
- R-045: 10行未満のバグ修正にはプランはオプション
- R-046: タイポ、1行修正のみはプランスキップ可能

**プランファイル構成**:

- R-047: プランには Summary / Context / Files to Modify / Files to Create / Files to Delete / Implementation Steps / Testing Strategy / Risks & Mitigations / Open Questions を含める

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/workflows/plan-driven.md`

### 2.2 Boris Tane パターン（カスタム Markdown プラン）

Cloudflare Engineering から発祥した3フェーズアプローチ。ネイティブ Plan Mode の代わりに永続化マークダウンを使用する。

- R-048: Phase 1（Emphatic Research）では「Research deeply」「in great detail」等の強調言語を使用し、`research.md` にファイル出力する
- R-049: Phase 1 の出力には既存パターン、ファイルパス、依存関係、制約を含める
- R-050: Phase 2（Annotation Cycle）では Claude が `plan.md` をドラフトし、人間がコメントで注釈、Claude がプラン更新を全質問解決まで繰り返す
- R-051: Phase 2 ではガードプロンプト「Do NOT implement anything yet」を必ず使用する
- R-052: Phase 3（Mechanical Implementation）では plan.md を順次実行し、完了項目にチェックボックスを付ける。タスク間で停止せず連続実行する
- R-053: Boris Tane パターンの利点はコンテキスト圧縮を生き延びる、構造化 diff でレビュー可能、人間-エージェント間の共有可変状態

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/workflows/plan-driven.md`

### 2.3 Dual-Instance Planning（二重インスタンス計画）

- R-054: Claude Zero（Planner）はコードベース探索、ユーザーインタビュー、詳細プラン作成、実装レビューを担当
- R-055: Claude One（Implementer）は承認済みプランを読み、コーディング、コミットを担当
- R-056: フローは Claude Zero がプラン作成 → `Review/` フォルダ → 人間がレビュー・承認 → `Active/` へ移動 → Claude One が実装 → Claude Zero がレビュー → `Completed/` へアーカイブ

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/workflows/dual-instance-planning.md`

### 2.4 クロスモデル戦略

Claude Code（Opus 4.6）と Codex CLI（GPT-5.4）を4つの逐次フェーズで組み合わせる。

- R-057: Step 1（Planning）は Claude Code で Plan Mode によるインタビュー、フェーズドプラン生成 → `plans/{feature-name}.md`
- R-058: Step 2（QA Review）は Codex CLI で実コードベースに対してプラン分析、中間フェーズ（"Phase 2.5"等）を挿入
- R-059: Step 3（Implementation）は Claude Code の新セッションでフェーズ毎にテストゲート付き実装
- R-060: Step 4（Verification）は Codex CLI の新セッションで実装をプランと照合検証
- R-061: 設計原則は加算的コラボレーション（Additive Collaboration）-- 各ツールは前の出力を上書きせず拡張する

> 情報源: [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice) `/development-workflows/cross-model-workflow/`

### 2.5 Command / Agent / Skill 連携

- R-062: Commands は `.claude/commands/<name>.md` に保存、現在のコンテキストで動作、ユーザー起動プロンプトテンプレート
- R-063: Agents は `.claude/agents/<name>.md` に保存、新規隔離コンテキストで動作、カスタムツール・権限を持つ自律アクター
- R-064: Skills は `.claude/skills/<name>/SKILL.md` に保存、注入（設定可能）で動作、プリロード・自動検出可能な知識
- R-065: 実行フローは Command がトリガー → Agent がタスク受領 → Agent が Skills を活用 → 結果返却
- R-066: 全体オーケストレーションは Research → Plan → Execute → Review → Ship

> 情報源: [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice)

## 3. TDD とテスト戦略

### 3.1 Red-Green-Refactor サイクル

- R-067: Claude は本来「実装を先に書き、テストで検証」する傾向がある。TDD はその逆（テストが設計を駆動）を要求するため、明示的プロンプトが必須
- R-068: RED フェーズではまだ存在しない関数/メソッドを参照するテストケースを作成する。"Write a FAILING test for [feature]. Do NOT write implementation yet." と明示する
- R-069: GREEN フェーズでは現在のテストを満たすだけのコードを実装する。"Only write enough code to pass the current tests, nothing more."
- R-070: REFACTOR フェーズではテストが通る状態を維持しつつコード品質を向上させる。"Tests must stay green after refactoring."

**カバレッジ要件**:

- R-071: 標準コードは80%以上のカバレッジを目標とする
- R-072: 金融計算/認証/セキュリティクリティカル/コアビジネスロジックは100%カバレッジ必須

**主要アンチパターン**:

- R-073: "Write tests for this feature" は Claude が先に実装するため、FAILING テストを明示的に要求する
- R-074: Red + Green を結合するとテストファースト利点を喪失するため、逐次プロンプトに分離する
- R-075: Refactor をスキップすると技術的負債が蓄積するため、必ず Refactor フェーズを含める
- R-076: 複数機能を同時に扱うとフォーカスを喪失するため、1機能 = 1 TDD サイクルとする

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/workflows/tdd-with-claude.md`, [affaan-m/everything-claude-code](https://github.com/affaan-m/everything-claude-code) `/commands/tdd.md`

### 3.2 JiTTesting（Just-in-Time Testing）

- R-077: PR 提出時に LLM がミューテーションテストを自動生成するエフェメラルテスト手法。マージ後にテストは破棄される
- R-078: Meta 社の実績として大規模導入で従来テスト比4倍のリグレッション捕捉改善

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

### 3.3 検証スペクトラム / Trust Calibration

- R-079: ボイラープレートコードはスキム（軽い確認）のみでよい
- R-080: 標準ロジックは通常レビュー
- R-081: セキュリティクリティカルはフルレビュー + テスト必須

**フィードバックループピラミッド**:

- R-082: Level 1 は即時フィードバック（IDE/エディタ）
- R-083: Level 2 は保存時フィードバック（Git hooks）
- R-084: Level 3 はコミット時フィードバック（CI）

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/core/visual-reference.md`

### 3.4 高度なテストパターン

- R-085: Property-Based Testing は fast-check 等のフレームワークで不変条件を検証する（出力長==入力長、全入力が出力に含まれる、出力がソート済み等）
- R-086: Mutation Testing はテスト通過後にミューテーションテストでカバレッジの弱点を特定する
- R-087: Legacy Code ではリファクタリング前に現在の振る舞いを捕捉する特性テスト（characterization tests）を作成する

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

## 4. コードレビューパターン

### 4.1 マルチエージェントレビュー

- R-088: 逐次チェーンは planner → tdd-guide → code-reviewer → security-reviewer の順で実行する
- R-089: セキュリティレビュー（CRITICAL）ではハードコードされた認証情報/APIキー/トークン、SQLインジェクション、XSS、入力バリデーション欠如を重点的に確認する
- R-090: 品質基準（HIGH）では50行超の関数、800行超のファイル、ネスト深度制限、エラーハンドリング欠落を確認する
- R-091: 重大度は Must Fix（セキュリティ、データ整合性）、Should Fix（品質、DRY違反）、Can Skip（オプション改善）の3段階で分類する

> 情報源: [affaan-m/everything-claude-code](https://github.com/affaan-m/everything-claude-code) `/commands/code-review.md`, `/commands/orchestrate.md`

### 4.2 レビュー自動修正ループ

- R-092: Review → Fix → Re-Review → Convergence の反復サイクルで品質を収束させる
- R-093: セーフガードとして最大イテレーション上限（3回）を設定する
- R-094: 品質ゲート（tsc + lint）、保護ファイル（package.json、マイグレーション）、変更閾値を設ける
- R-095: イテレーション間の git commit チェックポイントを作成する

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

### 4.3 Anti-rationalization Gate

- R-096: Stop フックで Claude が不完全な作業を完了と宣言することを防止する。高速モデル（Haiku）に最終レスポンスを評価させる
- R-097: 拒否すべきパターン: 問題を「既存のもの」「スコープ外」と主張、「修正すべき問題が多すぎる」と回避、要求されていない「フォローアップ」に先送り、問題をリスト化するだけで修正しない、テスト/lint失敗を言い訳付きでスキップ

> 情報源: [trailofbits/claude-code-config](https://github.com/trailofbits/claude-code-config)

### 4.4 セキュリティフック（Trail of Bits）

- R-098: PreToolUse フックで `rm -rf` のブロック、main/master への直接プッシュ阻止、監査ログ（CLI ツールミューテーション記録）、パッケージマネージャー強制を行う
- R-099: クレデンシャル・ブロック対象は SSH、AWS、Azure、Kubernetes、npm、PyPI、git credentials、macOS keychain、暗号ウォレット

> 情報源: [trailofbits/claude-code-config](https://github.com/trailofbits/claude-code-config)

### 4.5 PR レビュー

- R-100: Claude-Driven PR Review（Beta）はマルチエージェント PR 分析でバグ、セキュリティ脆弱性、リグレッションを検出する。GitHub App 統合（マネージド）
- R-101: Second Instance Review は2つ目の Claude インスタンスがスタッフエンジニアとしてアーキテクチャ焦点でレビューする
- R-102: Cross-Model Review は別モデル（Codex 等）の視点で実装正確性を検証する
- R-103: Challenge-Based Validation は "grill me on these changes and don't make a PR until I pass your test" で品質を担保する

> 情報源: [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice), [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

## 5. オーケストレーションパターン

### 5.1 逐次パイプライン

- R-104: フェーズ構成は RESEARCH → PLAN → IMPLEMENT → REVIEW → VERIFY の5段階
- R-105: 各エージェントは1つの明確な入力を受け取り、1つの明確な出力を生成する。出力は次フェーズの入力になる。フェーズをスキップしない
- R-106: ハンドオフドキュメントには Context / Findings / Files Modified / Open Questions / Recommendations を含める
- R-107: バリデーションパイプラインは Stage 1（型チェック）→ Stage 2（リンティング）→ Stage 3（テスト実行）。ファイル変更に基づき不要ステージをスマートスキップ

> 情報源: [affaan-m/everything-claude-code](https://github.com/affaan-m/everything-claude-code), [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

### 5.2 Ralph Wiggum Loop（Ralph Loop）

- R-108: 「生成 → レビュー → 修正 → 繰り返し、出力が品質基準を満たすまで」の反復パターン（Geoffrey Huntley のワークフローパターン）
- R-109: 完了条件例: 全テスト合格、TypeScript エラーゼロ、ESLint 警告なし、95パーセンタイルでレスポンスタイム < 100ms
- R-110: 常にイテレーション上限を設定する（無限ループ防止）
- R-111: 15-20イテレーション以降はコンテキスト長制約が顕在化する。要件を定期的に再述しないとハルシネーションリスクが増大する

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/workflows/iterative-refinement.md`

### 5.3 Lead-Teammate アーキテクチャ

- R-112: 1つのチームリードが複数のチームメイトを調整する。Git ベース調整でロックファイルによるタスククレーム、継続的 pull/push を行う

- R-114: 推奨上限はデフォルトで5エージェント以下（調整オーバーヘッドが生産性を超える）。100K+行のコードベースではより多いチームも正当化される

## 7. Curse of Instructions（命令の呪い）

### 核心問題

「より多くの命令 = 各ガイドラインへの遵守が悪化する」

### 150命令上限

実プロダクションデータからの実証的知見: 約150の異なるルールを超えるとモデルはそれらの一部を選択的に無視し始める。メカニズムは注意散逸（Attention Diffusion）。

HumanLayer のデータ: 構造化コンテキストを持つチームは未分化ルールリストの場合より15-25%良い遵守率を達成。

### ファイルサイズ別遵守率

| 設定ファイル行数 | 推定遵守率 |
|----------------|----------|
| 1-100行 | ~95% |
| 100-200行 | ~88% |
| 200-400行 | ~75% |
| 400-600行 | ~60% |
| 600行以上 | ~45%以下 |

### コンテキスト過負荷の兆候

1. **ルール沈黙**: 規約の80%は遵守するが特定ルールを一貫して無視
2. **矛盾する振る舞い**: ルールが一部ファイルでは適用され他では適用されない
3. **初回応答の遅延**: 大量コンテキスト処理による単純タスクの高レイテンシ
4. **汎用出力**: プロジェクト固有パターンではなく一般的ベストプラクティスにフォールバック

### 解決策: パススコーピング

常時オンコンテキストをコンテキストウィンドウの5%以下に維持する。

- パススコーピングなし: モノリシックな root CLAUDE.md（全ルール）= 8,000トークン常時オン
- パススコーピングあり: Root CLAUDE.md（共有ルールのみ）= 2,000トークン + 作業ディレクトリ別に+800〜1,500トークン

結果: カバレッジを失わず40-50%のコンテキスト削減。

### ルール vs スキルの区別

- **ルール**: 制約と基準（常時オン、常に適用）-- 例: 「API応答で生DBのIDを公開しない」
- **スキル**: 手順と能力（オンデマンドロード）-- 例: 「新APIエンドポイント作成のステップバイステップ」

両者を混同すると、そのタスクを実行していない時でも40行以上の手順的指示を毎セッションでロードすることになる。

### 品質原則

「ルールの質 > ルールの量」: 20の具体的・実行可能なルールが、200の一般的・理想的ルールを一貫して上回る。

「AIの出力失敗のほとんどはコンテキスト失敗であり、モデル失敗ではない。」

> 情報源: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/core/context-engineering.md`

## 10. workflow-evaluator 用 評価チェックリスト

### A. 方法論選定チェック

- [ ] プロジェクト規模（ソロ / 小規模 / エンタープライズ）を特定したか
- [ ] グリーンフィールド or ブラウンフィールドを判定したか
- [ ] ドメイン複雑度を評価したか（単純 / 中程度 / 複雑）
- [ ] ステークホルダー関与度を確認したか（低 / 中 / 高）
- [ ] 選定した方法論がティア分類体系と整合しているか

### B. Plan-First チェック

- [ ] プラン適用基準（R-044〜R-046）に基づきプランの要否を判断したか
- [ ] プランファイル構成（R-047）の必須項目が揃っているか
- [ ] 実装前にプランレビューが完了しているか
- [ ] コンテキスト圧縮対策（Boris Tane パターン等）を検討したか

### C. TDD チェック

- [ ] RED → GREEN → REFACTOR の順序を厳守しているか
- [ ] "Write a FAILING test" を明示的にプロンプトしているか
- [ ] 1機能 = 1 TDD サイクルを守っているか
- [ ] カバレッジ要件（R-071、R-072）を満たしているか
- [ ] Refactor フェーズをスキップしていないか

### D. コードレビューチェック

- [ ] セキュリティレビュー（R-089）の重点項目を確認したか
- [ ] 品質基準（R-090）の閾値を超えていないか
- [ ] レビュー自動修正ループのイテレーション上限（3回）を設定しているか
- [ ] Anti-rationalization Gate（R-096、R-097）を導入しているか
- [ ] セキュリティフック（R-098、R-099）が有効か

### E. オーケストレーションチェック

- [ ] 逐次パイプラインのフェーズをスキップしていないか
- [ ] ハンドオフドキュメント（R-106）が作成されているか
- [ ] Ralph Loop のイテレーション上限を設定しているか（R-110）
- [ ] エージェント数が推奨上限（5以下）を超えていないか（R-114）
- [ ] ワークフロー種別に応じたチェーン構成を選択しているか

### F. Curse of Instructions チェック

- [ ] ルール総数が150以下か
- [ ] 設定ファイルが200行以下か（遵守率75%以上を維持）
- [ ] パススコーピングを適用しているか
- [ ] ルールとスキルを区別しているか
- [ ] コンテキスト過負荷の兆候（ルール沈黙、矛盾、遅延、汎用出力）がないか

## 11. 判断フローチャート

```
[START] プロジェクト特性を確認
  |
  v
[Q1] チーム規模は？
  |-- ソロ -----------> SDD + TDD / Single Instance / ネイティブ Plan Mode
  |-- 5-10名 ---------> [Q2] へ
  |-- 10+チーム -------> BMAD + Spec Kit + Specmatic / Multi-Instance
  |
  v
[Q2] グリーンフィールド or ブラウンフィールド？
  |-- グリーンフィールド --> Spec Kit + TDD + BDD / Dual-Instance
  |-- ブラウンフィールド --> [Q3] へ
  |
  v
[Q3] マイクロサービスか？
  |-- Yes --> CDD + Specmatic / Cross-Model Review
  |-- No ---> [Q4] へ
  |
  v
[Q4] 既存 SaaS（100+フィーチャー）か？
  |-- Yes --> OpenSpec + BDD / Boris Tane パターン
  |-- No ---> [Q5] へ
  |
  v
[Q5] LLM ネイティブ製品か？
  |-- Yes --> Eval-Driven + Multi-Agent / Ralph Loop
  |-- No ---> SDD + TDD（デフォルト）
  |
  v
[Q6] Plan パターンの選択
  |-- 変更が3+ファイル or 50行以上 --> Plan 必須
  |-- 10行未満のバグ修正 -----------> Plan オプション
  |-- タイポ・1行修正 --------------> Plan スキップ
  |
  v
[Q7] コンテキスト管理
  |-- ルール数 > 150 or 設定 > 200行 --> パススコーピング適用
  |-- ルールとスキル混在 -------------> 分離してスキルをオンデマンド化
  |-- 常時オン > コンテキスト5% ------> 削減策を実施
  |
  v
[Q8] レビューパターンの選択
  |-- セキュリティクリティカル --> マルチエージェント + セキュリティフック
  |-- 品質重視 -----------------> Second Instance / Cross-Model Review
  |-- 標準 ---------------------> 自動修正ループ（最大3回）
  |
  v
[END] 選定完了 → 実行
```
