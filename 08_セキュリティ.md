## 1. 脅威モデル

### R-001: 3つの脅威ベクトルを認識すること
- **入力汚染（Input Poisoning）**: PDF、スクリーンショット、メール添付、GitHub PRへのプロンプトインジェクション埋め込み（不可視Unicode、HTMLコメント、Base64エンコーディング）
- **ツール侵害（Tool Compromise）**: MCPサーバーのデータ窃取、MCP Rug Pull攻撃、ツール汚染（tool poisoning）
- **設定悪用（Configuration Abuse）**: `.claude/`設定、`.mcp.json`、環境変数、hooksがユーザー同意前に実行

### R-002: `.claude/`フォルダは攻撃ベクトルである
リポジトリに`.claude/`を埋め込むことで、事前構成されたエージェント・コマンド・hooksが自動ロードされ、Skillマーケットプレイスをバイパスする。

### R-003: `.claude/`内の攻撃ベクトル分類
| ベクトル | リスクレベル |
|---------|-------------|
| 悪意あるhooks（`.claude/hooks/`のBashスクリプトが全ツール呼び出しで実行） | クリティカル |
| 悪意あるエージェント（`allowed-tools: ["Bash"]` + 窃取指示） | 高 |
| 悪意あるコマンド（プロンプト内の隠し指示、引数インジェクション） | 高 |
| 汚染されたCLAUDE.md（セキュリティ設定をオーバーライドする指示） | 高 |
| トロイのsettings.json（寛容な`permissions.allow`ルール） | 高 |

### R-004: `.claude/`監査チェックリスト
```bash
ls -la .claude/
grep -r "curl\|wget\|nc\|base64\|eval\|exec" .claude/ 2>/dev/null
grep -r "allowed-tools.*Bash" .claude/agents/ 2>/dev/null
grep -r "permissions.allow" .claude/ 2>/dev/null
cat .claude/CLAUDE.md | grep -i "disable\|skip\|override"
```

**情報源:** affaan-m/everything-claude-code, FlorianBruniaux/claude-code-ultimate-guide

## 3. プロンプトインジェクション回避技術

### R-009: 回避テクニック一覧を把握すること
| テクニック | メカニズム | 検出方法 |
|-----------|-----------|---------|
| ゼロ幅文字 | U+200B, U+200C, U+200D（人間に不可視） | Unicode正規表現パターンマッチング |
| 右から左オーバーライド | U+202Eがテキスト表示を反転 | 双方向テキストスキャン |
| ANSIエスケープシーケンス | `\x1b[`ターミナル操作 | エスケープフィルター/ブロック |
| Nullバイト切り捨て | `\x00`が文字列を切り捨て | バリデーションでのNull検出 |
| コメント内Base64 | LLMが自動デコード | エントロピーチェック（20文字以上） |
| ネストされたコマンド実行 | `$(evil_command)`やバッククォート | パターンブロック + 置換検出 |
| ホモグリフ置換 | キリル文字`а`がラテン文字`a`を模倣 | Unicode正規化 |
| タグ文字 | 不可視Unicodeブロック U+E0000-U+E007F | 範囲検出 |

### R-010: 検出用正規表現パターンを適用すること
```regex
# ゼロ幅/RTL/Bidi文字
[\x{200B}-\x{200D}\x{FEFF}\x{202A}-\x{202E}\x{2066}-\x{2069}]

# ANSIエスケープ
\x1b\[|\x1b\]|\x1b\(

# Nullバイト
\x00

# タグ文字
[\x{E0000}-\x{E007F}]

# Base64ペイロード
[#;].*[A-Za-z0-9+/]{20,}={0,2}

# ネストされたコマンド
\$\([^)]+\)|\`[^\`]+\`
```

### R-011: MCP固有のインジェクションパターンを認識すること
| パターン | 説明 |
|---------|------|
| ツール汚染（Tool Poisoning） | ツールスキーマ内の悪意あるメタデータがLLMの動作に影響 |
| Rug Pull攻撃 | 信頼確立後に良性MCPが悪意あるものに変換 |
| 混乱した代理人（Confused Deputy） | 信頼されていないサーバーが信頼された名前空間でツールを登録 |

### R-012: 防御戦略を5段階で実施すること
1. プリコミット検出hookがファイルをスキャン（ロールオーバーライド、ジェイルブレイク、権限偽装）
2. Unicodeスキャン（ゼロ幅文字、ANSIエスケープ、Nullバイト、タグ文字）
3. パターンブロック（ネストされたコマンド、Base64ペイロード）
4. リポジトリプリスキャン（README、SECURITY.md、package.json、.cursor/、.claude/の隠しHTMLコメントと悪意ある指示をチェック）
5. 高リスクファイル検査（CONTRIBUTING.mdのソーシャルエンジニアリング、SECURITY.mdの隠し指示）

### R-013: Anthropic公式のサーバーサイド防御を把握すること
サーバーサイドのプロンプトインジェクションプローブがツール出力をエージェントコンテキストに入る前にスキャンし、ハイジャック試行検出時に警告を追加する。

## 4. 防御アーキテクチャ（多層防御）

### R-014: 3層セキュリティスタックを構築すること
```
Tier 1（予防）:
  ├─ MCPベッティングワークフロー（5分監査チェックリスト）
  ├─ バージョンピンニング + ハッシュ検証
  ├─ サプライチェーンスキャン（mcp-scan by Snyk）
  ├─ エージェントスキルバリデーション（skills-ref）
  └─ リポジトリプリスキャン（.claude/監査）

Tier 2（検出）:
  ├─ プロンプトインジェクション検出hook
  ├─ Unicode/ANSIインジェクションスキャナー
  ├─ シークレット監視（Gitleaks, TruffleHog）
  ├─ 出力スキャナーhook（認証情報漏洩を捕捉）
  └─ .claude/整合性監視（SessionStart hook）

Tier 3（対応）:
  ├─ 即時認証情報取り消し（AWS, GitHub等）
  ├─ Git履歴監査 + 必要に応じてfilter-repo
  ├─ MCP設定整合性検証
  ├─ 自動セキュリティ監査エージェント
  └─ インシデント文書化 + コンプライアンス報告
```

### R-015: 5層防御深度アーキテクチャ（FlorianBruniaux版）
| レイヤー | 説明 |
|---------|------|
| Layer 1 | MCPバージョンピンニング + ハッシュ検証 |
| Layer 2 | permissions.denyルールによるRead/Edit/Write/Bashパターンのブロック |
| Layer 3 | PreToolUse hooksによるセカンダリバリデーション |
| Layer 4 | 外部シークレット管理（AWS Secrets Manager, 1Password, Vault） |
| Layer 5 | リポジトリプリスキャン（インジェクションベクトルの開封前チェック） |

### R-019: Anthropic公式サンドボックスアーキテクチャを理解すること
- **ファイルシステム分離**: 書き込みはCWDとサブディレクトリに制限
- **ネットワーク分離**: プロキシサーバーがドメイン接続制限を強制
- **OSレベルプリミティブ**: macOS Seatbelt、Linux bubblewrap
- **統計**: サンドボックスにより権限プロンプトが**84%削減**
- **クラウドデプロイ**: 各タスクがネットワーク・ファイルシステム制限付きの分離サンドボックスで実行

### R-021: MCP Rug Pull攻撃パターンを理解すること
```
Phase 1: 攻撃者が良性MCP公開 → ユーザーが承認
Phase 2: 2週間正常動作で信頼構築
Phase 3: 悪意ある更新をプッシュ（再承認不要） → ~/.ssh/*, .env, 認証情報を窃取
根本原因: ワンタイム承認モデル（更新は再同意なしで自動実行）
```

### R-022: Rug Pull攻撃の緩和策
- バージョンピンニング（例: @1.2.3、@latestは不可）
- ハッシュ検証（バイナリのsha256sum）
- 監視（設定変更のトラッキング）
- Git hooks（不正な更新の検出）

### R-023: 5分間MCPベッティングチェックリスト
| ステップ | アクション | 合格基準 |
|---------|-----------|---------|
| 1 | `gh repo view <mcp-repo>` | >50 stars、コミットが30日以内 |
| 2 | mcp.jsonをレビュー | `--dangerous-*`フラグがないこと |
| 3 | バージョン文字列を確認 | ピンニング済み（"latest"/"main"でない） |
| 4 | `sha256sum <binary>` | リリースチェックサムと一致 |
| 5 | 最近のコミットをレビュー | 疑わしい変更がないこと |

### R-024: MCPセーフリスト
| MCP | ステータス | 備考 |
|-----|----------|------|

## 6. キルスイッチと運用安全

### R-026: 3レベルキルスイッチシステムを構築すること
| レベル | メカニズム | トリガー | 実装 |
|--------|-----------|---------|------|
| 1. スコープ付き取り消し | 特定機能の無効化 | 疑わしい動作 | `dangerous-actions-blocker.sh` hook + `permissions.deny` |
| 2. 速度ガバナー | レート制限/閾値トリガー | 不安定なエージェント動作 | 5分間で20回以上のBashコマンド = ブロック |
| 3. グローバルハードストップ | 即時終了 | 確認された侵害 | Ctrl+C / Esc、`claude config set --disable`、アンインストール |

### R-027: プロセスグループキルを理解すること
負のPID（Node.js）による`SIGKILL`で、子プロセスを含む全プロセスグループを一括終了する。

### R-028: デッドマンスイッチ（ハートビート監視）
- エージェントの応答がない場合に自動終了
- 10分間の自動タイムアウト（リモートコントロール時の切断時）

### R-029: 速度ガバナーhookの仕様
- タイムスタンプ付きのコマンド呼び出しをトラッキング
- 5分間のローリングウィンドウを維持
- 閾値（20コマンド）超過時にブロック
- 暴走エージェントループを防止

### R-030: メモリハイジーンを実施すること
- 高リスクワークフローでは永続メモリを無効化
- プロジェクトメモリとグローバルメモリを分離
- 信頼できない実行の後にメモリをローテーション

### R-031: 監視可能な実行のためのログ記録対象
- ツール名、入力サマリー、触れたファイル
- 承認決定、ネットワーク試行
- セッションID

### R-032: 承認境界（人間の承認が必要な操作）
- サンドボックス外のシェル実行
- ネットワーク送出（egress）
- シークレットパスの読み取り
- リポジトリ外への書き込み
- ワークフローディスパッチ
- MCPサーバーの同意（自動承認しないこと）

      "matcher": "Bash",

        {

### R-034: シークレット露出時の対応プロトコル
**最初の15分:**
1. 即時取り消し（AWS, GitHub, Stripe等）
2. スコープ確認 - リモートにプッシュされたか確認
3. 全Git履歴でパターンを検索

**最初の1時間:**
4. Git履歴で露出ウィンドウを監査
5. ログ内の漏洩キーについて依存関係をスキャン
6. CI/CDログで露出を確認

**最初の24時間:**
7. 関連する全認証情報をローテーション
8. 必要に応じてコンプライアンスに通知
9. インシデントタイムラインを文書化

### R-040: denyルールの制限を理解すること
- `permissions.deny`はClaudeの組み込みツールのみをブロックする
- `/sandbox`が有効でない場合、Bashコマンドはdenyルールをバイパスする
- インデックス露出やBashワイルドカードバイパスというアーキテクチャ上の制限がある
- denyルールを多層防御の1つのレイヤーとして扱い、完全な解決策としてはならない

### R-041: プライバシー環境変数を設定すること（Trail of Bits推奨）
```json
{
  "env": {
    "DISABLE_TELEMETRY": "1",
    "DISABLE_ERROR_REPORTING": "1",
    "CLAUDE_CODE_DISABLE_FEEDBACK_SURVEY": "1"
  }
}
```

### R-042: MCP設定のセキュリティ
```json
{
  "enableAllProjectMcpServers": false,
  "cleanupPeriodDays": 365,
  "alwaysThinkingEnabled": true
}
```

## 8. Trail of Bitsの3層サンドボックス戦略

### R-043: Layer 1 - 組み込みサンドボックス（`/sandbox`）
| 項目 | 詳細 |
|------|------|
| 書き込み制限 | CWDとサブディレクトリに制限 |
| 読み取り | `/sandbox`未有効時は無制限 |
| ネットワーク | 明示的に許可されたドメインに制限 |
| OS分離 | macOS: Seatbelt、Linux: bubblewrap |

### R-044: Layer 2 - 権限denyルール
| 項目 | 詳細 |
|------|------|
| 適用範囲 | Read/Edit/Writeマッチャーによる認証情報アクセスのブロック |
| 対象 | Claudeの組み込みツールのみに適用 |
| 制限 | `/sandbox`が有効でない場合、直接Bashでバイパス可能 |

### R-045: Layer 3 - リモート/コンテナ分離
| 項目 | 詳細 |
|------|------|
| Devcontainer | プロジェクトファイルのみがマウントされる |
| リモートドロップレット | ローカルマシンからの完全分離 |
| アクセス制限 | ホストファイルシステム、SSHキー、認証情報へのアクセスなし |

### R-046: 推奨コンテナベースライン
```yaml
cap_drop: [ALL]
security_opt: [no-new-privileges:true]
```

### R-047: Anti-Rationalization Gate（Stop Hook）
エージェントが不完全な作業を正当化するのを防ぐStop Hook:
```json
{
  "Stop": [
    {
      "hooks": [
        {
          "type": "prompt",
          "prompt": "You are a JSON-only evaluator. You MUST respond with a single JSON object and NOTHING else. No markdown, no code fences, no explanation, no preamble. Just the raw JSON object.\n\nReview the assistant's final response. Reject if the assistant is rationalizing incomplete work: claiming issues are 'pre-existing' or 'out of scope', saying 'too many issues' to fix, deferring to unrequested 'follow-ups', listing problems without fixing them, or skipping test/lint failures with excuses.\n\nIf any of those patterns apply, respond:\n{\"ok\": false, \"reason\": \"You are rationalizing incomplete work. [specific issue]. Go back and finish.\"}\n\nOtherwise respond:\n{\"ok\": true}"
        }
      ]
    }
  ]
}
```
**重要**: プロンプトは評価者に「生のJSONのみで応答する」ことを明示的に指示する必要がある。

### R-048: Hook終了コード
| コード | 動作 |
|--------|------|
| 0 | アクション許可（stdoutがJSON制御として解析される） |
| 1 | エラー、非ブロッキング（stderrがverboseモードで表示） |
| 2 | ブロッキングエラー（stderrがClaudeにエラーメッセージとしてフィードバック） |

### R-049: 重要な安全上の注意事項
- Hooksはセキュリティ境界ではない（プロンプトインジェクションはhooksを回避できる）
- bypass-permissionsモードはサンドボックスと組み合わせる
- プロジェクト`.mcp.json`ファイルはgitに存在する（侵害されたリポジトリが悪意あるMCPサーバーを配布する可能性）
- Devcontainerが完全な読み書き分離を提供する

**情報源:** trailofbits/claude-code-config, Anthropic Engineering - Claude Code Sandboxing
