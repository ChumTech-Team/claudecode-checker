## 1. ファイル操作ツール

### 1.1 Read

ファイルを読み取るツール。あらゆるファイルに直接アクセス可能。

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `file_path` | string | Yes | 絶対パス（相対パス不可） |
| `offset` | number | No | 読み取り開始行番号 |
| `limit` | number | No | 読み取り行数（デフォルト: 最大2000行） |
| `pages` | string | No | PDFのページ範囲（例: "1-5", "3", "10-20"） |

**仕様・制約:**
- 結果は `cat -n` 形式（行番号付き、1始まり）
- **画像ファイル**（PNG, JPG等）を読み取ると視覚的にコンテンツが提示される（マルチモーダルLLM）
- **PDFファイル**（.pdf）を読み取り可能。10ページ超の大きなPDFは `pages` パラメータ必須（最大20ページ/リクエスト）
- **Jupyter Notebook**（.ipynb）を読み取り可能。全セルとその出力（コード、テキスト、可視化）を返す
- ディレクトリは読み取り不可（`ls` コマンドを使用）
- 空ファイルを読むとシステム警告が返る
- スクリーンショットのパスが提供された場合、常にこのツールで閲覧

**ソース:** システムプロンプト（Read tool定義）

### 1.2 Write

ファイルを書き込むツール。

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `file_path` | string | Yes | 絶対パス |
| `content` | string | Yes | 書き込む内容 |

**仕様・制約:**
- 既存ファイルがある場合は上書き
- 既存ファイルの場合、事前にReadツールで読み取り必須（未読だとエラー）
- 既存ファイルの変更にはEditツールを優先（差分のみ送信で効率的）
- 新規ファイル作成 or 完全書き換えの場合のみ使用
- ドキュメントファイル（*.md）やREADMEは明示的に要求されない限り作成しない
- 絵文字はユーザーが明示的に要求しない限り使用しない

**ソース:** システムプロンプト（Write tool定義）

### 1.3 Edit

ファイル内の文字列を正確に置換するツール。

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `file_path` | string | Yes | 絶対パス |
| `old_string` | string | Yes | 置換対象の文字列 |
| `new_string` | string | Yes | 置換後の文字列（old_stringと異なる必要あり） |
| `replace_all` | boolean | No | 全出現箇所を置換（デフォルト: false） |

**仕様・制約:**
- 会話中に少なくとも1回Readツールを使用してからでないとエラー
- Readツール出力の行番号プレフィックスのインデント（タブ/スペース）をそのまま保持する必要あり
- `old_string` がファイル内で一意でない場合、編集失敗。より多くの周辺コンテキストを含めて一意にするか、`replace_all` を使用
- `replace_all` はファイル全体での変数名変更等に有用
- 既存ファイルの編集を常に優先。新規ファイル作成は明示的に必要な場合のみ

**ソース:** システムプロンプト（Edit tool定義）

### 1.4 Glob

高速なファイルパターンマッチングツール。

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `pattern` | string | Yes | globパターン（例: `**/*.js`, `src/**/*.ts`） |
| `path` | string | No | 検索ディレクトリ（省略時: カレントディレクトリ） |

**仕様・制約:**
- あらゆるコードベースサイズで動作
- 結果は変更日時順にソート
- ファイル名パターンでの検索に使用
- `path` パラメータに "undefined" や "null" を入力しない（省略する）

**ソース:** システムプロンプト（Glob tool定義）

### 1.5 Grep

ripgrepベースの強力な検索ツール。

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `pattern` | string | Yes | 正規表現パターン |
| `path` | string | No | 検索パス（デフォルト: カレントディレクトリ） |
| `glob` | string | No | ファイルフィルタ（例: `*.js`, `*.{ts,tsx}`） |
| `type` | string | No | ファイルタイプ（例: js, py, rust, go） |
| `output_mode` | enum | No | `files_with_matches`(デフォルト), `content`, `count` |
| `-A` | number | No | マッチ後のコンテキスト行数 |
| `-B` | number | No | マッチ前のコンテキスト行数 |
| `-C` / `context` | number | No | マッチ前後のコンテキスト行数 |
| `-n` | boolean | No | 行番号表示（デフォルト: true） |
| `-i` | boolean | No | 大文字小文字無視 |
| `multiline` | boolean | No | 複数行マッチ（デフォルト: false） |
| `head_limit` | number | No | 出力行数制限（デフォルト: 250、0で無制限） |
| `offset` | number | No | スキップするエントリ数（デフォルト: 0） |

**仕様・制約:**
- 検索タスクには常にGrepを使用。Bashでの `grep` や `rg` コマンドは使わない
- ripgrepの正規表現構文を使用（`interface\{\}` のようにリテラルブレースはエスケープ）
- `multiline: true` で `struct \{[\s\S]*?field` のような行跨ぎパターンに対応
- `type` パラメータは `glob` より効率的（標準ファイルタイプの場合）
- `head_limit: 0` は大量結果でコンテキストを浪費するため控えめに使用

**ソース:** システムプロンプト（Grep tool定義）

### 1.6 ツールの使い分け

| 目的 | 推奨ツール | 備考 |
|------|-----------|------|
| ファイル名で検索 | Glob | `find` や `ls` の代わり |
| ファイル内容で検索 | Grep | `grep` や `rg` の代わり |
| ファイル読み取り | Read | `cat`/`head`/`tail` の代わり |
| ファイル編集 | Edit | `sed`/`awk` の代わり |
| ファイル書き込み | Write | `echo >` の代わり |

> Boris Cherny（Claude Code開発者）のTIP: "agentic search (glob + grep) beats RAG -- Claude Code tried and discarded vector databases because code drifts out of sync and permissions are complex"

**ソース:** システムプロンプト、https://github.com/shanraisshan/claude-code-best-practice

## 2. Bash Tool

シェルコマンドを実行するツール。

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `command` | string | Yes | 実行するコマンド |
| `description` | string | No | コマンドの説明（能動態、簡潔に） |
| `timeout` | number | No | タイムアウト（ミリ秒、最大600,000ms = 10分） |
| `run_in_background` | boolean | No | バックグラウンド実行 |
| `dangerouslyDisableSandbox` | boolean | No | サンドボックスモードを無効化 |

**仕様・制約:**
- 作業ディレクトリはコマンド間で保持されるが、シェル状態は保持されない
- シェル環境はユーザーのプロファイル（bash/zsh）から初期化
- デフォルトタイムアウト: 120,000ms（2分）
- スペースを含むファイルパスはダブルクォートで囲む
- バックグラウンド実行時は結果を後でReadで読み取り可能
- 複数コマンド実行ルール:
  - 独立したコマンド: 並列で複数のBash呼び出し
  - 依存関係あり: `&&` でチェーン
  - 前のコマンド失敗を気にしない場合: `;` で区切り
  - 改行でコマンドを区切らない（クォート文字列内は可）
- 不要な `sleep` コマンドは避ける
- `find`, `grep`, `cat`, `head`, `tail`, `sed`, `awk`, `echo` は専用ツールを優先

**ソース:** システムプロンプト（Bash tool定義）

## 4. Git連携

### 4.1 コミットの安全ルール

Claude Codeはコミット作成時に厳格な安全プロトコルに従う:

1. **ユーザーが明示的に要求した場合のみコミットを作成**
2. **git configを変更しない**
3. **破壊的コマンド禁止**: `push --force`, `reset --hard`, `checkout .`, `restore .`, `clean -f`, `branch -D` はユーザーの明示的指示なしに実行しない
4. **hook スキップ禁止**: `--no-verify`, `--no-gpg-sign` は使わない
5. **常に新規コミット**: amend ではなく新規コミットを作成（pre-commitフック失敗時、amendは前のコミットを破壊するリスク）
6. **ファイル個別指定**: `git add -A` や `git add .` より個別ファイル指定を優先（.env等の誤コミット防止）
7. **main/masterへの force push 警告**
8. **インタラクティブフラグ禁止**: `-i`（`git rebase -i`, `git add -i`）は非対応
9. **`--no-edit` 禁止**: `git rebase` には使用不可

### 4.2 コミットワークフロー

```
1. git status + git diff + git log を並列実行
2. 変更を分析しコミットメッセージを起草
3. ステージング + コミット作成（HEREDOCでメッセージ）+ git status確認
4. pre-commitフック失敗時は修正して新規コミット作成
```

**コミットメッセージ例（HEREDOC形式）:**
```bash
git commit -m "$(cat <<'EOF'
Add user authentication middleware

Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>
EOF
)"
```

### 4.3 PR作成ワークフロー

```
1. git status + git diff + リモート追跡状態確認 + git log + git diff [base]...HEAD を並列実行
2. 全コミットを分析しPRタイトル・サマリーを起草（70文字以内）
3. ブランチ作成 + push + gh pr create を並列実行
```

**PR作成コマンド例:**
```bash
gh pr create --title "the pr title" --body "$(cat <<'EOF'
## Summary
<1-3 bullet points>

## Test plan
[Bulleted markdown checklist of TODOs for testing the pull request...]

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

### 4.4 PRコメント閲覧

```bash
gh api repos/foo/bar/pulls/123/comments
```

### 4.5 Git / PR ベストプラクティス（Boris Cherny推奨）

| プラクティス | 詳細 |
|-------------|------|
| **PRは小さく保つ** | p50で118行（141 PR、1日で45K行変更の実績） |
| **常にsquash merge** | クリーンな線形履歴、1機能=1コミット、git revert/bisectが容易 |
| **頻繁にコミット** | 最低1時間に1回、タスク完了時すぐに |
| **@claudeでレビュー** | 同僚のPRに@claudeタグでlintルール自動生成 |
| **`/code-review`** | マルチエージェントPR分析（バグ、セキュリティ脆弱性、リグレッション検出） |

### 4.6 Git Worktrees

```bash
# CLI起動時にworktree指定
claude --worktree feature-name

# エージェントのfrontmatterで指定
isolation: worktree
```

- 機能ごとに隔離されたブランチで並列開発
- 各エージェントが独自の作業コピーを持つ
- `/batch` コマンドが自動的にworktreeを作成

**ソース:** システムプロンプト、https://github.com/shanraisshan/claude-code-best-practice

## 5. Web検索/フェッチ

### 5.1 WebSearch

Web検索を実行し、結果を応答に活用するツール。

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `query` | string | Yes | 検索クエリ（最小2文字） |
| `allowed_domains` | string[] | No | 結果を限定するドメイン |
| `blocked_domains` | string[] | No | 除外するドメイン |

**仕様・制約:**
- Anthropicのサーバーサイド `web_search_20250305` ツールを使用
- 検索結果はマークダウンハイパーリンク付きで返される
- 使用後は必ず「Sources:」セクションにURLを記載する必要あり
- 米国のみで利用可能
- 検索クエリには現在の年（2026年）を使用すること

### 5.2 WebFetch

URLからコンテンツを取得し、AIモデルで処理するツール。

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `url` | string (URI) | Yes | 取得するURL |
| `prompt` | string | Yes | コンテンツに対する質問/指示 |

**仕様・制約:**
- URLを取得しHTMLをマークダウンに変換後、小型高速モデル（Claude Haiku）で処理
- 生のHTML/マークダウンは返さず、promptに対する回答のみ返す
- HTTPは自動的にHTTPSにアップグレード
- 15分間のセルフクリーニングキャッシュ
- リダイレクト時は新しいURLが通知される（再リクエスト必要）
- **認証が必要なURL（Google Docs, Confluence, Jira, GitHub）では失敗** -- 専用MCPツールを使用
- GitHub URLには `gh` CLIを推奨
- MCPベースのWeb fetchツールが利用可能な場合はそちらを優先

**ソース:** システムプロンプト（WebSearch/WebFetch tool定義）、https://mikhail.io/2025/10/claude-code-web-tools/

## 6. マルチモーダル入力

### 6.1 画像

- Readツールで画像ファイル（PNG, JPG等）を読み取ると、視覚的にコンテンツが提示される
- `Ctrl+V` で画像をペースト可能
- スクリーンショットのパスが提供された場合は常にReadツールで閲覧
- 一時ファイルパスを含むあらゆる画像パスに対応

### 6.2 PDF

- Readツールで直接PDF読み取り可能
- 10ページ超は `pages` パラメータ必須（例: `pages: "1-5"`）
- 最大20ページ/リクエスト
- `pages` なしで大きなPDFを読むと失敗
- WebFetchでもPDFコンテンツの取得・処理が可能

### 6.3 Jupyter Notebook

- Readツールで `.ipynb` ファイルを読み取り可能
- 全セルとその出力を返す（コード、テキスト、可視化を統合）

**ソース:** システムプロンプト（Read tool定義）

## 7. ドキュメント生成Skills

### 7.1 DOCX（Word文書）

**トリガー:** Word doc、.docx、レポート・メモ・レター・テンプレート等のWord形式要求

#### アプローチ別ワークフロー

| タスク | 手法 |
|--------|------|
| 読み取り/分析 | `pandoc` またはXML展開 |
| 新規作成 | `docx-js`（JavaScript） |
| 既存編集 | 展開 → XML編集 → 再パック |

#### 新規作成（docx-js）

```javascript
const { Document, Packer, Paragraph, TextRun, Table, TableRow, TableCell,
        ImageRun, Header, Footer, AlignmentType, PageOrientation,
        LevelFormat, TableOfContents, HeadingLevel, BorderStyle,
        WidthType, ShadingType, PageNumber, PageBreak } = require('docx');

const doc = new Document({ sections: [{ children: [/* content */] }] });
Packer.toBuffer(doc).then(buffer => fs.writeFileSync("doc.docx", buffer));
```

#### 重要ルール（docx-js）

| ルール | 詳細 |
|--------|------|
| ページサイズ明示指定 | デフォルトはA4。US Letterは `12240 x 15840 DXA` |
| Landscape | portrait寸法を渡し `orientation: PageOrientation.LANDSCAPE` 設定 |
| 改行に `\n` 禁止 | 別々のParagraph要素を使用 |
| Unicode弾丸禁止 | `LevelFormat.BULLET` + numbering config |
| PageBreakはParagraph内 | スタンドアロンだと無効なXML |
| ImageRunに `type` 必須 | png/jpg等を常に指定 |
| テーブル幅はDXA | `WidthType.PERCENTAGE` はGoogle Docsで破綻 |
| テーブルは二重幅指定 | `columnWidths` 配列 + 各セルの `width`（合計一致必須） |
| セルマージンは内部パディング | コンテンツ領域を減らす（セル幅に加算されない） |
| `ShadingType.CLEAR` | SOLIDは黒背景になる |
| テーブルを区切り線に使わない | Paragraphの `border` を使用 |
| TOCにはHeadingLevelのみ | カスタムスタイル不可 |
| `outlineLevel` 必須 | TOC用（H1=0, H2=1等） |

#### 既存編集ワークフロー

```bash
# Step 1: 展開
python scripts/office/unpack.py document.docx unpacked/

# Step 2: XML編集（Editツールで直接文字列置換。Pythonスクリプト不可）

# Step 3: 再パック
python scripts/office/pack.py unpacked/ output.docx --original document.docx
```

- トラッキング変更の著者名は "Claude"
- スマートクォートはXMLエンティティ（`&#x201C;` 等）を使用
- コメント追加は `comment.py` スクリプトを使用

**依存関係:** pandoc、docx（npm）、LibreOffice、Poppler

**ソース:** https://github.com/anthropics/skills/blob/main/skills/docx/SKILL.md

### 7.2 XLSX（スプレッドシート）

**トリガー:** .xlsx/.xlsm/.csv/.tsv の入出力、スプレッドシート作成/編集/分析

#### ライブラリ選択

| ライブラリ | 用途 |
|-----------|------|
| **pandas** | データ分析、一括操作、シンプルなデータエクスポート |
| **openpyxl** | 複雑なフォーマット、数式、Excel固有機能 |

#### 重要: 数式を使う（ハードコード禁止）

```python
# NG: Pythonで計算してハードコード
total = df['Sales'].sum()
sheet['B10'] = total  # 5000をハードコード

# OK: Excel数式を使用
sheet['B10'] = '=SUM(B2:B9)'
sheet['C5'] = '=(C4-C2)/C2'
sheet['D20'] = '=AVERAGE(D2:D19)'
```

#### 共通ワークフロー

1. ツール選択（pandas or openpyxl）
2. ワークブック作成/読み込み
3. データ・数式・書式設定
4. 保存
5. **数式再計算（必須）**: `python scripts/recalc.py output.xlsx`
6. エラー検証・修正

#### 財務モデルの色コーディング規約

| 色 | 用途 |
|----|------|
| **青テキスト** (0,0,255) | ハードコード入力、シナリオ変更値 |
| **黒テキスト** (0,0,0) | 全数式・計算 |
| **緑テキスト** (0,128,0) | 同ワークブック内の他シートリンク |
| **赤テキスト** (255,0,0) | 外部ファイルリンク |
| **黄背景** (255,255,0) | 要注意の仮定・更新必要セル |

#### 数式検証チェックリスト

- セル参照2-3箇所をテスト
- Excelの列マッピング確認（列64 = BL）
- 行オフセット（DataFrame行5 = Excel行6、1-indexed）
- NaN処理（`pd.notna()`）
- ゼロ除算チェック（#DIV/0!）
- クロスシート参照形式（Sheet1!A1）

### 7.3 PPTX（プレゼンテーション）

**トリガー:** .pptx、スライドデッキ、プレゼンテーション、ピッチデッキ

#### アプローチ別ワークフロー

| タスク | 手法 |
|--------|------|
| 読み取り/分析 | `python -m markitdown presentation.pptx` |
| テンプレートから編集 | 展開 → スライド操作 → XML編集 → 再パック |
| ゼロから作成 | pptxgenjs（JavaScript） |

#### デザインガイドライン

- **大胆な配色**: トピックに合った色を選ぶ（デフォルト青を避ける）
- **支配色**: 1色が60-70%の視覚ウェイト、1-2サポート色、1アクセント
- **ダーク/ライトコントラスト**: タイトル+結論はダーク背景、コンテンツはライト
- **視覚モチーフ**: 1つの特徴的要素を全スライドで繰り返す
- **全スライドにビジュアル要素必須**: テキストのみは禁止

#### タイポグラフィ

| 要素 | サイズ |
|------|--------|
| スライドタイトル | 36-44pt 太字 |
| セクションヘッダ | 20-24pt 太字 |
| 本文 | 14-16pt |
| キャプション | 10-12pt ミュート |

#### 禁止事項

- 同じレイアウトの繰り返し
- 本文のセンター揃え（タイトルのみセンター可）
- タイトルと本文のサイズコントラスト不足
- テキストのみのスライド
- **タイトル下のアクセントライン禁止**（AI生成スライドの特徴）

#### QA（必須）

1. `python -m markitdown output.pptx` でコンテンツ確認
2. テンプレート使用時はプレースホルダテキスト残留チェック
3. **サブエージェントで視覚QA**（2-3スライドでも必須）
4. スライドを画像変換して検査
5. 修正 → 再検証のループ（最低1回の修正・検証サイクル必須）

**依存関係:** markitdown[pptx]、Pillow、pptxgenjs（npm）、LibreOffice、Poppler

**ソース:** https://github.com/anthropics/skills/blob/main/skills/pptx/SKILL.md

### 7.4 PDF

**トリガー:** .pdfファイルに関するあらゆる操作

#### 主要ライブラリ

| ライブラリ | 用途 |
|-----------|------|
| **pypdf** | 基本操作（マージ、分割、回転、暗号化） |
| **pdfplumber** | テキスト・テーブル抽出 |
| **reportlab** | PDF新規作成 |
| **pytesseract** | OCR（スキャンPDF） |

#### 基本操作（pypdf）

```python
from pypdf import PdfReader, PdfWriter

# 読み取り
reader = PdfReader("document.pdf")
text = "".join(page.extract_text() for page in reader.pages)

# マージ
writer = PdfWriter()
for pdf_file in ["doc1.pdf", "doc2.pdf"]:
    for page in PdfReader(pdf_file).pages:
        writer.add_page(page)
writer.write(open("merged.pdf", "wb"))

# 暗号化
writer.encrypt("userpassword", "ownerpassword")
```

#### テーブル抽出（pdfplumber）

```python
import pdfplumber, pandas as pd

with pdfplumber.open("document.pdf") as pdf:
    for page in pdf.pages:
        tables = page.extract_tables()
        for table in tables:
            df = pd.DataFrame(table[1:], columns=table[0])
```

#### PDF作成（reportlab）

```python
from reportlab.lib.pagesizes import letter
from reportlab.platypus import SimpleDocTemplate, Paragraph, Spacer
from reportlab.lib.styles import getSampleStyleSheet

doc = SimpleDocTemplate("report.pdf", pagesize=letter)
styles = getSampleStyleSheet()
story = [Paragraph("Title", styles['Title'])]
doc.build(story)
```

#### 重要な制約

- **Unicode下付き/上付き文字禁止**: ReportLabの組み込みフォントでは黒いボックスになる
- 代わりに `<sub>` / `<super>` タグを使用:
  ```python
  Paragraph("H<sub>2</sub>O", styles['Normal'])
  Paragraph("x<super>2</super>", styles['Normal'])
  ```

## 8. Anthropic Skills リポジトリ概要

Skillsはフォルダ構成の指示・スクリプト・リソースで、Claudeが特定タスクのパフォーマンスを向上させるために動的にロードする。

### インストール方法（Claude Code）

```bash
# マーケットプレイス追加
/plugin marketplace add anthropics/skills

# プラグイン直接インストール
/plugin install document-skills@anthropic-agent-skills
/plugin install example-skills@anthropic-agent-skills
```

### Skill構造

```markdown
---
name: my-skill-name
description: スキルの説明（トリガー条件）
---
# 指示内容
```

| フィールド | 説明 |
|-----------|------|
| `name` | 一意な識別子（小文字、ハイフン区切り） |
| `description` | トリガー条件を含む完全な説明 |
| `allowed-tools` | パーミッションプロンプトスキップ |
| `model` | モデルオーバーライド |
| `effort` | 推論レベルオーバーライド |
| `context: fork` | サブエージェントで実行 |

### ドキュメントスキルのライセンス

docx, xlsx, pptx, pdf のスキルは**Proprietary（ソースアベイラブル、オープンソースではない）**。Claude のドキュメント機能を裏で動かしている本番スキルを開発者向けに共有したもの。
