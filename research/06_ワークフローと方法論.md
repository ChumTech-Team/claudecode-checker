## 1. 15種の開発方法論 比較

**ソース**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/core/methodologies.md`

### 6ティア分類体系

#### Tier 1: 戦略的オーケストレーション
| 方法論 | 説明 | 適合シーン |
|--------|------|------------|
| **BMAD** (Business-driven Methodical AI Development) | マルチエージェント・ガバナンス、コンスティチューション・ガードレール | エンタープライズ（10+チーム） |
| **GSD** (Get Shit Done) | 6フェーズワークフロー、タスク毎に新鮮な200kトークンコンテキスト | コンテキスト劣化対策 |

#### Tier 2: 仕様・アーキテクチャ
| 方法論 | 説明 | 適合シーン |
|--------|------|------------|
| **SDD** (Spec-Driven Development) | コード実装前に仕様を記述 | グリーンフィールド・ガバナンス |
| **Doc-Driven** | リビング・ドキュメンテーションを唯一の真実の情報源とする | ドキュメント中心チーム |
| **DDD** (Domain-Driven Design) | ソフトウェアをビジネス言語に整合 | ドメイン複雑系 |

#### Tier 3: 振る舞い・受入
| 方法論 | 説明 | 適合シーン |
|--------|------|------------|
| **BDD** (Behavior-Driven Development) | Given-When-Then シナリオでステークホルダーと協業 | 受入基準重視 |
| **ATDD** (Acceptance TDD) | コーディング前に受入基準を協調定義 | ステークホルダー合意 |
| **CDD** (Contract-Driven Development) | API契約を実行可能なインターフェースとして定義 | マイクロサービス並行開発 |
| **JiTTesting** (Just-in-Time Testing) | PR提出時にLLMがエフェメラルテストを自動生成、マージ後に破棄 | Meta社が大規模導入、従来テスト比4倍のリグレッション捕捉 |

#### Tier 4: フィーチャーデリバリー
| 方法論 | 説明 | 適合シーン |
|--------|------|------------|
| **FDD** (Feature-Driven Development) | 1フィーチャー最大2週間のイテレーション | 短サイクルデリバリー |
| **Context Engineering** | コンテキストを設計要素として扱う（Thoughtworks 2025） | コンテキスト最適化 |

#### Tier 5: 実装
| 方法論 | 説明 | 適合シーン |
|--------|------|------------|
| **TDD** (Test-Driven Development) | Red-Green-Refactorサイクル + 検証ループ | 品質重視実装 |
| **Eval-Driven** | LLM出力をevalsでテスト | LLMネイティブ製品 |
| **Multi-Agent Orchestration** | オーケストレーションされたサブエージェントで複雑タスク処理 | 大規模タスク分解 |

#### Tier 6: 最適化
- 反復的リファインメントループ
- フレッシュコンテキストパターン（Ralph Loop）
- プロンプトエンジニアリング基盤

## 2. Plan-First開発

### 2.1 Plan Mode（ネイティブ）

**ソース**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/workflows/plan-driven.md`

**基本原則**: 「計画が良ければコードも良い（Once the plan is good, the code is good）」

**5ステップワークフロー**:
1. **Plan Mode 開始**: Shift+Tab を2回押下（Normal → Auto-Accept → Plan）
2. **Claude が探索**: 読み取り専用モードで関連ファイルを読み、パターンを検索（変更不可）
3. **Claude がプラン記述**: `.claude/plans/[name].md` にプラン生成
4. **人間がレビュー**: 要件との整合性、スコープ適切性を確認
5. **承認 → 実行**: "Proceed with the plan" で実装開始

**プラン適用基準**:
- 適用すべき: APIの変更、DB マイグレーション、新機能（3+ファイル or 50行以上の変更）
- オプション: 10行未満のバグ修正
- スキップ可能: タイポ、1行修正のみ

**プランファイル構成**:
- Summary / Context / Files to Modify / Files to Create / Files to Delete
- Implementation Steps / Testing Strategy / Risks & Mitigations / Open Questions

### 2.2 Boris Tane パターン（カスタムMarkdownプラン）

Cloudflare Engineeringから発祥した3フェーズアプローチ。ネイティブPlan Modeの代わりに永続化マークダウンを使用する。

**Phase 1: Emphatic Research（強調的リサーチ）**
- 「Research deeply」「in great detail」等の強調言語を使用
- `research.md` に出力（口頭サマリーではなくファイル出力）
- 既存パターン、ファイルパス、依存関係、制約を含む

**Phase 2: The Annotation Cycle（アノテーション・サイクル）**
- Claude が `plan.md` をドラフト（アプローチ、影響ファイル、スニペット、トレードオフ、未解決質問）
- 人間がコメントで注釈
- Claude がプラン更新
- 全質問解決まで繰り返し
- ガードプロンプト: 「Do NOT implement anything yet」

**Phase 3: Mechanical Implementation（機械的実装）**
- plan.md を順次実行、完了項目にチェックボックス
- タスク間で停止せず連続実行
- フィードバックは簡潔に（決定は既に完了している）

**利点**: コンテキスト圧縮を生き延びる、構造化diffでレビュー可能、人間-エージェント間の共有可変状態

### 2.3 Dual-Instance Planning（二重インスタンス計画）

**ソース**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/workflows/dual-instance-planning.md`

**アーキテクチャ**:
- **Claude Zero（Planner）**: コードベース探索、ユーザーインタビュー、詳細プラン作成、実装レビュー
- **Claude One（Implementer）**: 承認済みプランを読み、コーディング、コミット

**5フェーズ**:
1. Claude Zero がコードベース探索、質問、プラン作成 → `Review/` フォルダ
2. 人間がレビュー・承認 → `Active/` へ移動
3. Claude One が `Active/` のプランを読み順次実装
4. Claude Zero が完成コードをプラン要件と照合レビュー
5. 完了プランを `Completed/` へアーカイブ

**コスト比較**:

| パターン | スケーリング | 月額コスト | 最適シーン |
|---------|-------------|-----------|-----------|
| Single Instance | なし | $50-100 | 一般用途 |
| Dual-Instance | 垂直 | $100-200 | スペック重視・品質重視 |
| Multi-Instance (Boris) | 水平（5-15並列） | $500-1,000 | チーム・高ボリューム |

### 2.4 クロスモデル戦略

**ソース**: [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice) `/development-workflows/cross-model-workflow/`

Claude Code（Opus 4.6）とCodex CLI（GPT-5.4）を4つの逐次フェーズで組み合わせる。

| ステップ | ツール | 役割 |
|---------|--------|------|
| Step 1: Planning | Claude Code | Plan Modeでインタビュー、フェーズドプラン生成 → `plans/{feature-name}.md` |
| Step 2: QA Review | Codex CLI | 実コードベースに対してプラン分析、中間フェーズ（"Phase 2.5"等）を挿入 |
| Step 3: Implementation | Claude Code | 新セッションでフェーズ毎にテストゲート付き実装 |
| Step 4: Verification | Codex CLI | 新セッションで実装をプランと照合検証 |

**設計原則**: 加算的コラボレーション（Additive Collaboration）-- 各ツールは前の出力を上書きせず拡張する

### 2.5 Command → Agent → Skill 連携

**ソース**: [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice)

| コンポーネント | 保存場所 | コンテキスト種別 | ユースケース |
|---------------|---------|----------------|-------------|
| **Commands** | `.claude/commands/<name>.md` | 現在のコンテキスト | ユーザー起動プロンプトテンプレート、ワークフローオーケストレーション |
| **Agents** | `.claude/agents/<name>.md` | 新規隔離コンテキスト | カスタムツール・権限を持つ自律アクター |
| **Skills** | `.claude/skills/<name>/SKILL.md` | 注入（設定可能） | 設定可能・プリロード・自動検出可能な知識 |

**実行フロー**: Command がトリガー → Agent がタスク受領 → Agent が Skills を活用 → 結果返却

**全体オーケストレーション**: Research → Plan → Execute → Review → Ship

## 3. TDDとテスト戦略

### 3.1 Red-Green-Refactor サイクル

**ソース**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/workflows/tdd-with-claude.md`, [affaan-m/everything-claude-code](https://github.com/affaan-m/everything-claude-code) `/commands/tdd.md`

**根本的課題**: Claude は本来「実装を先に書き、テストで検証」する傾向がある。TDD はその逆（テストが設計を駆動）を要求する。

#### Phase 1: RED（失敗するテスト作成）
- まだ存在しない関数/メソッドを参照するテストケースを作成
- テストは "function not defined" 等のエラーで失敗すべき
- **明示的プロンプト必須**: "Write a FAILING test for [feature]. Do NOT write implementation yet."

#### Phase 2: GREEN（最小限の実装）
- 現在のテストを満たすだけのコードを実装
- "Only write enough code to pass the current tests, nothing more."

#### Phase 3: REFACTOR（品質改善）
- テストが通る状態を維持しつつコード品質向上
- "Tests must stay green after refactoring."

**カバレッジ要件**:
- 標準コード: 80%以上
- 金融計算/認証/セキュリティクリティカル/コアビジネスロジック: 100%必須

**主要アンチパターン**:

| エラー | 影響 | 修正 |
|--------|------|------|
| "Write tests for this feature" | Claude が先に実装する | FAILING テストを明示的に要求 |
| Red + Green を結合 | テストファースト利点喪失 | 逐次プロンプトに分離 |
| "Make tests pass" | 実装ファーストを助長 | "Write tests, then implement minimally" |
| Refactor スキップ | 技術的負債蓄積 | 必ず Refactor フェーズ含む |
| 複数機能を同時 | フォーカス喪失 | 1機能 = 1 TDDサイクル |

### 3.2 JiTTesting（Just-in-Time Testing）

**ソース**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

- PR提出時にLLMがミューテーションテストを自動生成するエフェメラルテスト手法
- マージ後にテストは破棄される
- **Meta社の実績**: 大規模導入で従来テスト比 **4倍のリグレッション捕捉** 改善

### 3.3 検証スペクトラム / Trust Calibration

**ソース**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/core/visual-reference.md`

リスクベースのレビュー深度調整:
- **ボイラープレートコード**: スキム（軽い確認）のみ
- **標準ロジック**: 通常レビュー
- **セキュリティクリティカル**: フルレビュー + テスト必須

**フィードバックループピラミッド**:
- **Level 1**: 即時（IDE/エディタ）
- **Level 2**: 保存時（Git hooks）
- **Level 3**: コミット時（CI）

### 3.4 高度なテストパターン

**Property-Based Testing**: fast-check等のフレームワークで不変条件を検証（出力長==入力長、全入力が出力に含まれる、出力がソート済み等）

**Mutation Testing**: テスト通過後、ミューテーションテストでカバレッジの弱点を特定

**Legacy Code**: リファクタリング前に現在の振る舞いを捕捉する特性テスト（characterization tests）を作成

## 4. コードレビューパターン

### 4.1 マルチエージェントレビュー

**ソース**: [affaan-m/everything-claude-code](https://github.com/affaan-m/everything-claude-code) `/commands/code-review.md`, `/commands/orchestrate.md`

**逐次チェーン**: planner → tdd-guide → code-reviewer → security-reviewer

**セキュリティレビュー重点**（CRITICAL レベル）:
- ハードコードされた認証情報、APIキー、トークン
- SQLインジェクション脆弱性
- XSS脆弱性
- 入力バリデーション欠如

**品質基準**（HIGH レベル）:
- 50行超の関数
- 800行超のファイル
- ネスト深度制限
- エラーハンドリングの欠落

**重大度分類**:
- :red_circle: Must Fix（セキュリティ、データ整合性）
- :yellow_circle: Should Fix（品質、DRY違反）
- :green_circle: Can Skip（オプション改善）

### 4.2 レビュー自動修正ループ

**ソース**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

Review → Fix → Re-Review → Convergence の反復サイクル

**セーフガード**:
- 最大イテレーション上限（3回）
- 品質ゲート（tsc + lint）
- 保護ファイル（package.json、マイグレーション）
- 変更閾値
- イテレーション間の git commit チェックポイント

### 4.3 Anti-rationalization Gate

**ソース**: [trailofbits/claude-code-config](https://github.com/trailofbits/claude-code-config)

Claude が不完全な作業を完了と宣言することを防止する Stop フック。高速モデル（Haiku）に最終レスポンスを評価させる。

**設定例**:
```json
{
  "Stop": [
    {
      "hooks": [
        {
          "type": "prompt",
          "prompt": "Review the assistant's final response. Reject if the assistant is rationalizing incomplete work: claiming issues are 'pre-existing' or 'out of scope', saying 'too many issues' to fix, deferring to unrequested 'follow-ups', listing problems without fixing them, or skipping test/lint failures with excuses."
        }
      ]
    }
  ]
}
```

**拒否すべきパターン**:
- 問題を「既存のもの」「スコープ外」と主張
- 「修正すべき問題が多すぎる」と回避
- 要求されていない「フォローアップ」に先送り
- 問題をリスト化するだけで修正しない
- テスト/lint失敗を言い訳付きでスキップ

### 4.4 セキュリティフック（Trail of Bits）

**ソース**: [trailofbits/claude-code-config](https://github.com/trailofbits/claude-code-config)

**PreToolUse フック**:
- `rm -rf` のブロック
- main/master への直接プッシュ阻止
- 監査ログ（CLI ツールミューテーション記録）
- パッケージマネージャー強制

**クレデンシャル・ブロック対象**: SSH、AWS、Azure、Kubernetes、npm、PyPI、git credentials、macOS keychain、暗号ウォレット

### 4.5 Claude-Driven PR Review（Beta）

**ソース**: [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice)

- マルチエージェント PR 分析
- バグ、セキュリティ脆弱性、リグレッションを検出
- GitHub App 統合（マネージド）

### 4.6 ピアレビューパターン

- **Second Instance Review**: 2つ目の Claude インスタンスがスタッフエンジニアとしてアーキテクチャ焦点でレビュー
- **Cross-Model Review**: 別モデル（Codex等）の視点で実装正確性を検証
- **Challenge-Based Validation**: "grill me on these changes and don't make a PR until I pass your test"

## 5. オーケストレーションパターン

### 5.1 逐次パイプライン（Sequential Pipeline）

**ソース**: [affaan-m/everything-claude-code](https://github.com/affaan-m/everything-claude-code) `/commands/orchestrate.md`, [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

**フェーズ構成**:
1. Phase 1: RESEARCH（Explore agent）→ `research-summary.md`
2. Phase 2: PLAN（planner agent）→ `plan.md`
3. Phase 3: IMPLEMENT（tdd-guide agent）→ code changes
4. Phase 4: REVIEW（code-reviewer agent）→ `review-comments.md`
5. Phase 5: VERIFY（build-error-resolver if needed）→ done or loop back

**原則**: 「各エージェントは1つの明確な入力を受け取り、1つの明確な出力を生成。出力は次フェーズの入力になる。フェーズをスキップしない。」

**ハンドオフドキュメント形式**: Context / Findings / Files Modified / Open Questions / Recommendations

**バリデーションパイプライン**:
- Stage 1: 型チェック
- Stage 2: リンティング
- Stage 3: テスト実行
- ファイル変更に基づき不要ステージをスマートスキップ

### 5.2 Ralph Wiggum Loop（Ralph Loop）

**ソース**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/workflows/iterative-refinement.md`, `/guide/core/glossary.md`

**定義**: 「生成 → レビュー → 修正 → 繰り返し、出力が品質基準を満たすまで」（Geoffrey Huntleyのワークフローパターン）

**自律ループの完了条件例**:
- 全テスト合格
- TypeScriptエラーゼロ
- ESLint警告なし
- 95パーセンタイルでレスポンスタイム < 100ms

**重要制約**:
- 常にイテレーション上限を設定（無限ループ防止）
- サイクル間の改善が意味ある閾値を下回った場合は早期停止
- 15-20イテレーション以降はコンテキスト長制約が顕在化
- 要件を定期的に再述しないとハルシネーションリスク増大

### 5.3 反復検索パターン（Iterative Retrieval）

**ソース**: [affaan-m/everything-claude-code](https://github.com/affaan-m/everything-claude-code)

オーケストレータがサブエージェントの返却を評価し、フォローアップ質問を行い、十分になるまでループ（最大3サイクル）。

### 5.4 Lead-Teammate アーキテクチャ

**ソース**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/workflows/agent-teams.md`

- 1つのチームリードが複数のチームメイトを調整
- Git ベース調整: ロックファイルでタスククレーム、継続的 pull/push
- P2P メッセージング（メールボックスシステム）、階層的報告だけでなくピア間通信
- 各エージェントが互いのアプローチに積極的に異議を唱える（自己組織化）
- 各エージェントは独立した1Mトークンコンテキストウィンドウ（約30,000行のコード）

**推奨上限**: デフォルトでは **5エージェント以下**（調整オーバーヘッドが生産性を超える）。100K+行のコードベースではより多いチームも正当化。

### 5.5 並列実行

独立した品質チェック向け: code-reviewer（品質）、security-reviewer（セキュリティ）、architect（設計）を同時実行し、結果をマージ。

### 5.6 ワークフロー種別

| ワークフロー | チェーン構成 |
|------------|------------|
| Feature | planning → TDD → code review → security audit |
| Bugfix | planning → TDD → code review（security reviewer省略） |
| Refactor | architecture review → implementation → code review |
| Security | vulnerability assessment first → implementation |

### 5.7 マルチセッション調整

`orchestrate-worktrees.js` でワーカーが個別 git worktree を使用。`seedPaths` でローカルファイルを隔離ブランチにオーバーレイ。コントロールプレーンがアクティブセッション、diff、承認、テレメトリを追跡。

## 7. Curse of Instructions

**ソース**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) `/guide/core/context-engineering.md`

### 核心問題

**「より多くの命令 = 各ガイドラインへの遵守が悪化する」**

### 150命令上限

実プロダクションデータからの実証的知見: 約150の異なるルールを超えると、モデルはそれらの一部を選択的に無視し始める。これはハード技術的カットオフではないが、量が質を劣化させる信頼性の高いシグナル。

**メカニズム**: 注意散逸（Attention Diffusion）。プロンプトに数百の潜在的に関連する制約が含まれると、モデルの注意がそれらに分散。早期配置・最近宣言・強い語調のルールが低顕著性ルールを押し出す。

**HumanLayerのデータ**: 構造化コンテキストを持つチームは、未分化ルールリストの場合より **15-25% 良い遵守率** を達成。

### ファイルサイズ別遵守率

| 設定ファイル行数 | 推定遵守率 |
|----------------|----------|
| 1-100行 | ~95% |
| 100-200行 | ~88% |
| 200-400行 | ~75% |
| 400-600行 | ~60% |
| 600行以上 | ~45%以下 |

### コンテキスト過負荷の兆候

1. **ルール沈黙**: 規約の80%は遵守するが特定ルールを一貫して無視
2. **矛盾する振る舞い**: ルールが一部ファイルでは適用され他では適用されない
3. **初回応答の遅延**: 大量コンテキスト処理による単純タスクの高レイテンシ
4. **汎用出力**: プロジェクト固有パターンではなく一般的ベストプラクティスにフォールバック

### 解決策: パススコーピング

常時オンコンテキストを **コンテキストウィンドウの5%以下** に維持。

**パススコーピングなし**: モノリシックな root CLAUDE.md（backend + frontend + DB + API ルール全部）= 8,000トークン常時オン

**パススコーピングあり**:
- Root CLAUDE.md（共有ルールのみ）= 2,000トークン
- `src/api/` 作業時 = +1,500トークン
- `src/components/` 作業時 = +1,200トークン
- `prisma/` 作業時 = +800トークン

**結果**: カバレッジを失わず **40-50% のコンテキスト削減**

### ルール vs スキルの区別

- **ルール**: 制約と基準（常時オン、常に適用） -- 例: 「API応答で生DBのIDを公開しない」
- **スキル**: 手順と能力（オンデマンドロード） -- 例: 「新APIエンドポイント作成のステップバイステップ」

両者を混同すると、そのタスクを実行していない時でも40行以上の手順的指示を毎セッションでロードすることになる。

### 品質原則

**「ルールの質 > ルールの量」**: 20の具体的・実行可能なルールが、200の一般的・理想的ルールを一貫して上回る。

**診断リフレーム**: 「AIの出力失敗のほとんどはコンテキスト失敗であり、モデル失敗ではない。」
