## 1. トークン経済学（Token Economics）

### コンテキストウィンドウの構成

Claude Codeのコンテキストウィンドウ（標準200Kトークン、Max/Team/Enterpriseで1Mトークン）は以下の要素で構成される:

| 要素 | トークン消費量 |
|------|-------------|
| システムプロンプト + 指示 | 〜50命令スロットを消費 |
| ツール定義 + MCPサーバーオーバーヘッド | サーバー1つにつき〜2Kトークン |
| 会話履歴（ターンごとに蓄積） | 20メッセージの会話で〜30,000トークン |
| ファイル読み込み内容 | 100行のファイルで〜500トークン |

### トークン単価

| モデル | 入力 | 出力 |
|--------|------|------|
| Sonnet | $3/MTok | $15/MTok |
| Opus 4.6 | $5/MTok | $25/MTok |
| キャッシュ書き込み | 1.25倍 | - |
| キャッシュ読み取り | 0.1倍 | - |

### 実際のコスト目安（Anthropic公式）

- 開発者1人あたり平均: **$6/日**
- 90%のユーザーが **$12/日以下**
- 月額平均（Sonnet 4.6）: **$100-200/開発者/月**（利用頻度で大きく変動）

### セッション別コスト目安

| タスク種別 | 所要時間 | コスト |
|-----------|---------|-------|
| バグ修正 | 5-10分 | $0.05-$0.10 |
| 機能開発 | 1-2時間 | $0.20-$0.50 |
| 大規模リファクタリング | 半日 | $1.00-$2.00 |

### 月額予算目安

| プロファイル | 月間利用時間 | 月額 |
|------------|-----------|------|
| 個人趣味開発者 | 10-20時間 | $5-$15 |
| プロ開発者 | 40-80時間 | $20-$50 |
| 5人チーム | 200時間以上 | $100-$250 |

## 2. ルール遵守率の劣化カーブ

### 基本データ

- フロンティアLLMが確実に従える命令数の上限: **約150-200命令**
- Claude Codeのシステムプロンプトが消費する命令スロット: **〜50**
- ユーザーが使えるルール枠: **100-150命令**
- エージェントシナリオでの完全遵守率: **30%未満**

### 劣化パターン

- **線形減衰（Linear Decay）**: 命令数を2倍にすると、遵守率は半減する
- 低価値ルールの追加は、高価値ルールの遵守率も一様に低下させる
- 200行を超えるCLAUDE.mdは最適パラメータを約33%超過している
- 命令の増殖は知識ベースの利用率低下とも相関する（258ファイルのKBが一度も自律的に参照されなかった事例）

### コンテキスト充填度による品質変化

| コンテキスト使用率 | 状態 |
|------------------|------|
| 0-50% | 最適パフォーマンス、制約なし |
| 50-75% | パフォーマンス安定、選択的ファイル読み込み推奨 |
| 65%以上 | 圧縮された領域のニュアンスが失われ始める |
| 75-90% | 自動compact発動、品質が目に見えて劣化（ファイル再読み込み、以前の判断との矛盾） |
| 90%以上 | 強制クリア必要、出力の深刻な不整合 |

## 3. 否定制約 vs 肯定指示の効果

### 基本データ

- 否定制約（明示的な「〜するな」文）は、肯定のみのガイダンスと比較して **15-25%の遵守率改善**
- 実装方法: CLAUDE.mdの肯定パターンセクションの後に「Do NOT」セクションを設ける

### ただし注意点（別の研究結果）

- **10個の否定ルールを肯定表現に書き換えたところ、ルール違反が約半減した**という逆の報告もある
  - 例: 「デフォルトエクスポートを使うな」 → 「名前付きエクスポートを排他的に使用する」
- 結論: **否定形と肯定形の最適な組み合わせ**が重要。完全に一方だけに偏るのではなく、ルールの性質に応じて使い分ける

> ソース: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide), [DEV Community - 5 Patterns](https://dev.to/docat0209/5-patterns-that-make-claude-code-actually-follow-your-rules-44dh)

## 4. Path-Scopingテクニック

### 実装方法

特定の関数や行範囲を指定してファイルを読み込む:
```
@src/auth.ts:45-60
```

### 効果

- ファイル読み込みコストを **〜5,000トークンから〜500トークンに削減**
- **90%のトークン削減**（シンボル特定クエリとの組み合わせ時）

### サブディレクトリ別CLAUDE.md（Scoped CLAUDE.md）

- サブディレクトリ固有の指示ファイルを作成（例: `src/api/CLAUDE.md`）
- ファイルの場所に基づいて階層的にロード
- ルートファイルをスリムに保ちつつ、文脈に適した精密なガイダンスを提供

> ソース: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide), [DEV Community - 5 Patterns](https://dev.to/docat0209/5-patterns-that-make-claude-code-actually-follow-your-rules-44dh)

## 5. 20%ルール（CLAUDE.md / Skills / Commands の使い分け基準）

### 頻度ベースの配置基準

| セッション出現頻度 | 配置先 | トークンコスト |
|------------------|--------|-------------|
| **20%以上のセッション** | CLAUDE.mdルール（毎セッション自動ロード） | 0トークン追加（常時ロード） |
| **5-20%のセッション** | スキルモジュール（`/skill`でオンデマンドロード） | 500-1,000トークン |
| **5%未満のセッション** | スラッシュコマンド（明示的呼び出しのみ） | 呼び出し時のみ |

### 根拠

20%以上の作業で繰り返し使われるパターンは、常時ロードされることで生産性向上が複利的に蓄積される。それ以下の頻度では、永続的ロードのコンテキスト税が利用頻度の便益を上回る。

> ソース: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

## 6. Strategic Compact

### いつすべきか

- コンテキスト使用率が **70%** に達したとき
- 多くの実践者が独立して **60%を超えないこと** を閾値として共有
- `/compact`を70%で実行すると、利用可能なスペースの**約50%を回復**

### 何が残るか

- タスクに重要な判断の要約
- 会話が単一ブロックに圧縮され、キャッシュヒット率が回復（後続トークン読み取りコストが0.1倍に低減）

### 何が失われるか

- 会話の詳細な文脈（ニュアンス）
- 65%以降に圧縮された領域のニュアンス
- 具体的なコードスニペットや詳細な議論の内容

### すべきでないとき

- まだコンテキストに余裕がある場合（50%未満）
- タスクの途中で詳細な文脈が必要な場合
- 代わりに**新しいセッションを開始する方が良い**ケースも多い（Fresh Context Pattern参照）

### コスト最適化機能（Anthropic公式）

- **プロンプトキャッシング**: 繰り返しコンテンツ（システムプロンプト等）のコスト削減
- **自動圧縮（auto-compaction）**: コンテキスト限界に近づくと会話履歴を自動要約
- **effort level**: `/effort`や`/model`で調整可能。深い推論が不要な単純タスクではeffortを下げてコスト削減
- **MAX_THINKING_TOKENS=8000**: 思考トークンの予算上限設定

> ソース: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide), [Anthropic公式 - Costs](https://docs.anthropic.com/en/docs/claude-code/costs)

## 7. セッション管理

### 1タスク1チャット原則

- **最も単純なルール: 1つの会話に1つのタスク**
- 新しいセッション開始のコスト: **〜20Kトークン**（汚染されたセッションの品質低下に比べれば微々たるもの）

### Context Rot（コンテキスト腐敗）

会話履歴が長くなるにつれて、古い・無関係・矛盾する情報が蓄積し、AI応答品質が段階的に劣化する現象。

**具体的な閾値:**
- **65%**: 圧縮領域のニュアンスが失われ始める
- **75%**: エージェントが目に見えて劣化（ファイルの再読み込み、以前の判断との矛盾）
- **2時間以上のセッション**: 品質劣化が顕著になる
- LLMパフォーマンスは **〜16Kトークンの蓄積コンテキストを超えると20-30%劣化**（総使用量がウィンドウ内でも）

### Fresh Context Pattern（Ralph Loop）

1時間以上の自律タスクに対して、外部状態ファイルを通じて新しいセッションで再開するパターン。

**実装方法:**
1. 各イテレーションで完全に新しいClaude Codeプロセスを生成
2. コンテキストウィンドウを0%にリセット
3. 永続ファイル（`TASK.md`, `PROGRESS.md`）+ gitコミットで状態を保存
4. 状態は永続ファイルからエフェメラルセッションへ一方向に流れる
5. 更新はPROGRESS.mdとgitコミットを通じて永続層に戻る
6. SessionStartフックが最新のハンドオーバードキュメントを検出し、コンテキストとして注入

**効果:** タスクごとにフル200Kのアテンションを維持

> ソース: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide), [Vincent van Deth - Context Rot](https://vincentvandeth.nl/blog/context-rot-claude-code-automatic-rotation), [MindStudio - Context Rot](https://www.mindstudio.ai/blog/what-is-context-rot-claude-code)

## 8. .tmpファイル戦略 & Context Injection

### .tmpファイル戦略

大きな一時出力を一時ファイル（`~/.claude/.tmp/`）に退避し、会話履歴から除去する手法。

**対象:**
- テスト出力
- デバッグトレース
- 大量の検索結果
- 再参照しない大きな出力

**効果:** アクセス性を失わずにコンテキストを解放

### Context Injection（v2.1.32以降）

特定のチェックポイントから`/rewind`を使用して「ここから先」を要約する機能。

**特徴:**
- チェックポイント以前のコンテキストを保存しつつ、チェックポイント以降のターンを圧縮
- 完全な`/compact`よりも外科的なアプローチ
- `@path`インポート: 外部ファイルを参照してコンテキストに注入（会話を散らかさずに）

> ソース: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

## 9. Context Debt Score

### コンテキスト消耗ゾーン

| ゾーン | 使用率 | 状態 | 推奨アクション |
|-------|-------|------|-------------|
| Green | 0-50% | 最適パフォーマンス | 制約なし |
| Yellow | 50-75% | パフォーマンス安定 | 選択的ファイル読み込み |
| Orange | 75-90% | 自動compact発動 | 品質劣化開始 |
| Red | 90%以上 | 深刻な不整合 | 強制クリア必要 |

### Instruction Debt（命令負債）

セッション間でプロジェクトの慣習を手動で繰り返し説明することで生じる累積的なコンテキストオーバーヘッド。

**定量化:**
- 未保存の判断1つあたり: **200-500トークンのコスト**（再説明のたびに発生）
- 解決策: CLAUDE.mdへの一度の記述で解消

> ソース: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

## 10. Trigger-Table Lazy Loading

全てのプロジェクトコンテキストを起動時にロードするのではなく、トリガーを宣言してオンデマンドでロードする手法。

### 実装例

```markdown
# CLAUDE.md 抜粋
## Auth Moduleをロードする条件:
- ユーザーが言及: auth, login, session, JWT
- 編集ファイル: src/auth/*
```

### 効果

- ベースラインのコンテキスト消費を **20-30%削減**
- 必要時にはフルアクセスを維持

> ソース: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide)

## 11. MCP vs CLI+Skills のトークン比較

### オーバーヘッド比較

| 方式 | トークンオーバーヘッド | 特徴 |
|------|---------------------|------|
| CLAUDE.mdルール（スラッシュコマンド） | 0トークン追加 | 常時ロード |
| スキルモジュール | 500-1,000トークン | オンデマンドロード |
| MCPサーバー（1つ） | 〜2,000トークン以上 | 永続接続 |
| MCPサーバー（5つ） | 〜10,000トークン | 永続接続 |

### MCP固有の利点

- Figma MCP: 画像解析トークンコストを **3-10倍削減**（ビジョンAPIではなく構造化データとして処理）
- シーケンシャルパターン: ツール結果を複数クエリで再利用（オーバーヘッドを償却）

### 推奨方針

- **CLAUDE.mdルール + スキル**: プロジェクト固有パターンに使用
- **MCP**: 外部統合（GitHub API、データベース、デザインツール）に限定使用

## 12. CLAUDE.mdメモリシステム（Anthropic公式）

### ファイル階層

| レベル | 場所 | スコープ |
|-------|------|---------|
| グローバル | `~/.claude/CLAUDE.md` | 全プロジェクト共通 |
| プロジェクト | プロジェクトルートの`CLAUDE.md` | プロジェクト全体 |
| ローカル | サブディレクトリの`CLAUDE.md` | 該当ディレクトリ配下 |
| ルールファイル | `.claude/rules/` | プロジェクトレベルの制約 |
| ユーザールール | `~/.claude/rules/` | 個人/組織の標準 |

### ロード仕様

- CLAUDE.mdファイルは**長さに関係なく全文ロード**される（ただし短いほど遵守率が高い）
- MEMORY.mdは**最初の200行 または 最初の25KB**（いずれか先に到達した方）のみロード
- 作業ディレクトリより上位のCLAUDE.mdファイルは起動時に全文ロード
- ルールファイル（`.claude/rules/`）は明示的な言及なしに自動ロード

### 自動メモリ（Auto Memory）

Claudeが自ら知見を蓄積する機能:
- ビルドコマンド
- デバッグインサイト
- アーキテクチャメモ
- コードスタイルの好み
- ワークフローの習慣

> ソース: [Anthropic公式 - Memory](https://docs.anthropic.com/en/docs/claude-code/memory)

## 13. 実践的なルール遵守改善パターン（5つのパターン）

### パターン1: 30行ルール
- CLAUDE.mdを200行から**〜30行**に削減
- Claudeが既存コードから推測できないものだけを含める
- テスト: 「これを削除したら、Claudeは間違うか？」

### パターン2: 重要ルールのアンカリング
- 最も違反されるルールをファイルの**先頭と末尾**に配置（初頭効果/新近効果を利用）
- 重要ルールを先頭と末尾に意図的に**重複記載**

### パターン3: Hooksによるハード強制
- 絶対的な要件をプロンプトからコードレベルのフック（PreCommit, PostWrite）に移行
- プロンプト命令は「提案」、フックは「強制」

### パターン4: 実装時間
- 全5パターンの適用に**合計約1時間**

## 14. 成熟度フレームワーク（Maturity Framework）

| ステージ | 特徴 | コンテキスト管理 |
|---------|------|---------------|
| **Novice（初心者）** | 手動ファイル読み込み、CLAUDE.mdなし | 頻繁に90%以上に到達 |
| **Intermediate（中級）** | CLAUDE.mdあり、定期的な`/compact` | 50-75%に留まる |
| **Advanced（上級）** | スキルモジュール、フック、構造化クエリ | 30-50%、70%で先制的に対応 |
| **Expert（エキスパート）** | マルチセッション・オーケストレーション、Fresh Context Pattern | 30%未満、自律的最適化 |
