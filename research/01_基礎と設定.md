## 1. Claude Codeのアーキテクチャ

### 概要

Claude Codeはエージェント型コーディングツールであり、コードベースの読み取り、ファイル編集、コマンド実行、開発ツールとの統合を行う。ターミナル、IDE、デスクトップアプリ、ブラウザで利用可能。

> ソース: https://docs.anthropic.com/en/docs/agents-and-tools/claude-code/overview

### エージェントループ（Agent Loop）

Claude Codeの実行モデルは `while(tool_call)` ループである。分類器（classifier）、RAGパイプライン、DAGオーケストレーターは一切使用していない。

- Claudeがツール使用リクエストで応答する
- アプリケーションがそのリクエストを評価して結果を返す
- このサイクルが繰り返される（エージェントループ）
- 継続的なユーザー入力なしに自律的にアクションを実行可能

タスクを受け取ると3つのフェーズで動作する:
1. **コンテキスト収集（gather context）** - ファイル検索でコードを理解
2. **アクション実行（take action）** - 編集を行い変更を適用
3. **結果検証（verify results）** - テスト実行で動作確認

これらのフェーズは明確に分離されず、ツールを使いながら流動的に進む。

> ソース: https://github.com/FlorianBruniaux/claude-code-ultimate-guide

### ツールシステム

- **クライアントツール**: ユーザー定義ツール、Anthropicスキーマツール（bash、text_editor）はユーザーのアプリケーション上で実行
- **サーバーツール**: web_search、code_execution、web_fetch、tool_search はAnthropicのインフラ上で実行
- **拡張ツール**: Model Context Protocol（MCP）によりGoogle Drive、Jira、Slackなど外部サービスと接続可能
- **マルチエージェント**: Task toolでサブエージェントを生成し、タスクの異なる部分を並列処理可能

> ソース: https://docs.anthropic.com/en/docs/agents-and-tools/claude-code/overview

### 拡張システム

| 拡張方法 | 説明 |
|----------|------|
| Skills | Claudeの知識をオンデマンドで拡張するMarkdownファイル |
| MCP | 外部サービスに接続 |
| Hooks | ワークフローの自動化（シェルスクリプトをトリガーベースで実行） |
| Subagents | 独自コンテキストウィンドウを持つ隔離されたサブエージェントを生成 |

> ソース: https://github.com/FlorianBruniaux/claude-code-ultimate-guide

## 2. セットアップ方法

### システム要件

| OS | バージョン |
|----|-----------|
| macOS | 10.15+ |
| Ubuntu/Debian | Ubuntu 20.04+ / Debian 10+ |
| Windows | 10+（WSL 1、WSL 2、またはGit for Windows必須） |

### インストール

**ネイティブインストーラー（推奨）:**
- npm版より高速
- 依存関係不要
- バックグラウンドで自動更新

**npm版（非推奨）:**
- Node.js 18+ が必要
- `sudo npm install -g` は使用禁止（パーミッション問題やセキュリティリスクの原因）

### 認証

- **必要なアカウント**: Pro、Max、Teams、Enterprise、またはConsoleアカウント
- 無料のClaude.aiプランではClaude Codeは利用不可
- **デフォルト認証**: Anthropic Consoleを経由しOAuthプロセスを完了（console.anthropic.comでのアクティブな課金が必要）
- **サードパーティプロバイダー**: Amazon Bedrock、Google Vertex AI、Microsoft Foundryでも利用可能

> ソース: https://docs.anthropic.com/en/docs/claude-code/setup

## 3. 基本操作

### スラッシュコマンド

`/` を入力すると全コマンドが表示される。`/` に続けて文字を入力するとフィルタリング可能。

#### ビルトインコマンド

| コマンド | 説明 |
|---------|------|
| `/clear` | 会話履歴をクリアして新しい会話を開始 |
| `/compact` | コンテキストを圧縮（カスタム圧縮指示も追加可能） |
| `/model` | モデルとリーズニングを選択 |
| `/effort` | エフォートレベルを変更（low/medium/high） |
| `/cost` | APIトークン使用量を表示 |
| `/context` | コンテキスト使用状況を確認 |
| `/usage` | プラン制限を確認 |
| `/extra-usage` | オーバーフロー課金を設定 |
| `/config` | 設定を構成 |
| `/memory` | メモリ設定（auto memoryのトグルなど） |
| `/plan` | 読み取り専用のプランニングモードに入る（変更なしでアプローチを提案） |
| `/vim` | vim スタイル編集を有効化 |
| `/help` | ヘルプを表示 |
| `/terminal-setup` | Shift+Enterバインディングをインストール |
| `/keybindings` | キーバインド設定ファイルを開く |
| `/doctor` | キーバインド警告を含む診断チェック |
| `/output-style` | 出力スタイルを変更 |
| `/output-style [style]` | 直接スタイルを指定（例: `/output-style explanatory`） |
| `/output-style:new` | 新しい出力スタイルを作成 |

#### バンドルスキル（Skills）

ビルトインコマンドとは異なり、バンドルスキルはプロンプトベースでClaudeにプレイブックを与え、ツールを使って作業を遂行させる。

| スキル | 説明 |
|--------|------|
| `/simplify` | 最近変更されたファイルのコード再利用・品質・効率の問題をレビューして修正 |

> ソース: https://docs.anthropic.com/en/docs/claude-code/interactive-mode, https://docs.anthropic.com/en/docs/claude-code/slash-commands, https://github.com/shanraisshan/claude-code-best-practice

### キーボードショートカット

| ショートカット | 機能 |
|---------------|------|
| `Shift+Enter` | 改行挿入（iTerm2, WezTerm, Ghostty, Kittyは設定不要。VS Code, Alacritty, Zed, Warpは `/terminal-setup` が必要） |
| `Ctrl+R` | コマンド履歴のインタラクティブ検索 |
| `Ctrl+C` | 現在の処理をキャンセル |
| `Ctrl+O` | トランスクリプトビューアーのトグル |
| `Escape` | 入力のキャンセル/モード切替 |
| `/` | スラッシュコマンドメニューを表示 |

#### ターミナル設定（Option/Altキー）

| ターミナル | 設定 |
|-----------|------|
| Terminal.app | 設定 → プロファイル → キーボード → 「Use Option as Meta Key」をチェック |
| VS Code | 設定 → プロファイル → Keys → Left/Right Option keyを「Esc+」に設定 |
| iTerm2 | 設定不要 |

#### vim モード

- `/vim` コマンドまたは `/config` で永続的に有効化
- vim normal modeでカーソルが入力の先頭/末尾にあり移動できない場合、矢印キーでコマンド履歴をナビゲート

> ソース: https://docs.anthropic.com/en/docs/claude-code/interactive-mode

### カスタムキーバインド

`/keybindings` を実行すると `~/.claude/keybindings.json` を作成/オープン。

```json
{
  "ctrl+shift+r": "/review",
  "ctrl+shift+c": "/compact",
  "ctrl+shift+p": "/plan"
}
```

**コードバインディング（Chord Sequences）:**
- 2つのキーの組み合わせを順番に押す（同時ではない）
- スペースで区切って定義
- 最初のコードで「chord mode」に入り、2番目のキーを待つ
- 1500ms以内に2番目のキーが押されなければキャンセル

**即時適用:** keybindings.jsonの変更は自動検出・即時適用（再起動不要）

**診断:** `/doctor` でキーバインド警告を確認

> ソース: https://code.claude.com/docs/en/keybindings

## 4. CLAUDE.md

### 概要

CLAUDE.mdはClaudeが会話開始時に読み込む特別なファイル。Bashコマンド、コードスタイル、ワークフロールールなどの永続的なコンテキストを提供する。

### ファイルの配置場所

| 場所 | スコープ | 用途 |
|------|---------|------|
| `~/.claude/CLAUDE.md` | 全セッション | グローバル設定 |
| `./CLAUDE.md`（プロジェクトルート） | プロジェクト | gitにチェックインしてチームで共有 |
| 親ディレクトリ | モノレポ | モノレポ全体の設定 |
| 子ディレクトリ | サブディレクトリ | そのディレクトリのファイルを操作する時にオンデマンドで読み込み |

### 読み込みの仕組み

- 起動時にカレントワーキングディレクトリからファイルシステムのルートに向かって上方に走査
- サブディレクトリのCLAUDE.mdはそのディレクトリのファイルを操作する時のみ読み込み
- `@path/to/import` 構文で追加ファイルをインポート可能

### 遵守率（Compliance Rate）

- **よく構造化されたCLAUDE.mdでも約70%しか遵守されない**
- Claudeは全てを永続的なコマンドとして扱うのではなく、能動的にフィルタリングする
- 約150-200のインストラクション予算があり、システムプロンプトが既に約50を使用
- **ユーザーが使える実質的なインストラクション枠は約100-150**

> ソース: https://github.com/shanraisshan/claude-code-best-practice, https://code.claude.com/docs/en/best-practices

### ベストプラクティス

1. **簡潔に保つ**: 肥大化したCLAUDE.mdはClaudeに指示を無視させる原因になる。各行について「これを削除したらClaudeがミスを犯すか？」と自問し、NOなら削除
2. **WHAT/WHY/HOWフレームワーク**: 技術スタック、プロジェクト構造、目的を記述。特にモノレポでは重要
3. **コピーよりポインタを使う**: コードスニペットをCLAUDE.mdに含めない（すぐに古くなる）。代わりに `file:line` 参照を使用
4. **反復的な改善**: Claudeがミスしたら「CLAUDE.mdを更新してこのミスを繰り返さないようにして」と指示。時間とともにCLAUDE.mdは実際のミスに基づいた生きたドキュメントになる
5. **Skillsで専門知識を切り離す**: CLAUDE.mdは毎セッション読み込まれるが、Skillsは関連する時のみ読み込まれる。コンテキストをリーンに保つ
6. **Hooksで確実性を担保**: CLAUDE.mdの指示は助言的だがHooksは決定論的。100%実行が必要なアクションにはHooksを使用

### CLAUDE.md記述テンプレート例

```markdown
# プロジェクト概要
[プロジェクトの目的と技術スタック]

# コードスタイル
- [具体的なスタイルルール]

# ビルド・テストコマンド
- `npm run build` - ビルド
- `npm run test` - テスト
- `npm run lint` - リント

# アーキテクチャ
- src/components/ - UIコンポーネント
- src/lib/ - ユーティリティ
- src/api/ - APIルート

# 重要なルール
- [必ず守るべきルール]
```

> ソース: https://code.claude.com/docs/en/best-practices, https://github.com/shanraisshan/claude-code-best-practice

## 5. settings.json

### ファイルの配置と優先順位

設定は以下の優先順位でマージされる（上が最高優先）:

| 優先順位 | ファイル | 説明 |
|---------|--------|------|
| 1（最高） | `.claude/settings.local.json` | ローカル設定（gitにコミットしない） |
| 2 | `.claude/settings.json` | プロジェクト設定（gitにコミットしてチーム共有） |
| 3（最低） | `~/.claude/settings.json` | ユーザー設定（全プロジェクトに適用） |
| 上書き | プログラマティックオプション（agents, allowed_tools） | 常にファイルシステム設定を上書き |

**優先順位の注意点**: ユーザー設定でallowされていてもプロジェクト設定でdenyされていればブロックされる。

### JSON Schemaによるバリデーション

`$schema` 行を追加するとVS Code、Cursorなどでオートコンプリートとインラインバリデーションが有効になる。

### 主要な設定キー

| キー | 説明 |
|-----|------|
| `permissions` | allow/deny/askルールの配列 |
| `env` | 環境変数（セッションごとに適用） |
| `hooks` | トリガーベースの自動化 |
| `skills` | Skill拡張の設定 |
| `mcpServers` | MCPサーバー接続設定 |
| `allowedTools` | ツール許可リスト（allowルールリストにエントリを追加） |
| `effort` | エフォートレベル |
| `colorTheme` | カラーテーマ |
| `autoMemoryEnabled` | 自動メモリの有効/無効 |
| `allowManagedHooksOnly` | マネージド設定でのみ設定可能 |

### permissions設定例

```json
{
  "$schema": "https://cdn.anthropic.com/claude-code/settings.schema.json",
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test *)",
      "Read(~/.zshrc)"
    ],
    "deny": [
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"
    ]
  }
}
```

### ルール評価順序

1. **deny** ルールが最初に評価される
2. 次に **ask** ルール
3. 最後に **allow** ルール
4. 最初にマッチしたルールが適用される

ルールの形式: `Tool` または `Tool(specifier)`

> ソース: https://docs.anthropic.com/en/docs/claude-code/settings

### Hookイベント一覧

| イベント | 説明 |
|---------|------|
| `PreToolUse` | ツール使用前 |
| `PostToolUse` | ツール使用後 |
| `PostToolUseFailure` | ツール使用失敗後 |
| `Notification` | 通知時 |
| `UserPromptSubmit` | ユーザープロンプト送信時 |
| `SessionStart` | セッション開始時 |
| `SessionEnd` | セッション終了時 |
| `Stop` | 停止時 |
| `SubagentStart` | サブエージェント開始時 |
| `SubagentStop` | サブエージェント停止時 |
| `PreCompact` | コンパクト前 |
| `PermissionRequest` | パーミッション要求時 |
| `Setup` | セットアップ時 |
| `TeammateIdle` | チームメイトアイドル時 |
| `TaskCompleted` | タスク完了時 |
| `ConfigChange` | 設定変更時 |
| `WorktreeCreate` | ワークツリー作成時 |

## 6. メモリシステム

### 2つの補完的メモリメカニズム

| メカニズム | 説明 | 管理者 |
|-----------|------|--------|
| **CLAUDE.md** | ユーザーが書く永続的なインストラクション | ユーザー |
| **Auto Memory** | Claudeが自動的に学習内容を記録 | Claude自身 |

両方とも会話開始時にコンテキストとして読み込まれるが、強制的な設定ではない。

### Auto Memory

- **デフォルトで有効**
- セッション間で知識を蓄積（ビルドコマンド、デバッグ知見、アーキテクチャノート、コードスタイル、ワークフロー習慣）
- トグル方法: セッション内で `/memory` を開きトグル、または `autoMemoryEnabled` を設定

### メモリストレージ

- 各プロジェクトのメモリディレクトリ: `~/.claude/projects/<project>/memory/`
- `<project>` パスはgitリポジトリから導出
- 同一リポジトリの全ワークツリーとサブディレクトリで1つのauto memoryディレクトリを共有

### メモリファイルの読み込み制限

- **MEMORY.md の最初の200行、または最初の25KB**（いずれか先に到達した方）が会話開始時に読み込まれる
- しきい値を超えたコンテンツは読み込まれない
- Claudeは詳細なノートを個別のトピックファイルに移動してMEMORY.mdを簡潔に保つ

> ソース: https://docs.anthropic.com/en/docs/claude-code/memory

### Auto Dream

Auto Dreamはバックグラウンドのメモリ統合機能で、メモリファイルの自動的なプルーニング、マージ、リフレッシュを行う。

#### 動作条件
- **24時間ごと**に実行（5セッション以上の後）

#### 解決する問題
20セッション以上経過すると、auto memoryのノートが矛盾する記述、意味を失った相対日付（「昨日」等）、存在しないファイルへの参照で混乱する。

#### 4フェーズの動作（脳の睡眠時記憶統合にインスパイア）

1. **スキャン**: メモリディレクトリとMEMORY.mdを読み取り現状を把握
2. **マップ構築**: 既存の内容のマップを作成
3. **パターン検索**: 最近のセッションのトランスクリプト（JSONLファイル）から高価値パターンを特定（ユーザー修正、明示的な保存、繰り返しテーマ、重要な決定）
4. **統合**: 重複排除、古いノートの削除、相対日付を絶対日付に変換、MEMORY.mdを200行以下に維持

#### パフォーマンス
- 913セッション分のメモリを約8-9分で統合した事例あり

#### 睡眠のメタファー
- アクティブセッション中ではなく、セッション間で実行される
- 人間のREM睡眠が短期記憶を長期記憶に統合するプロセスと同様

> ソース: https://claudefa.st/blog/guide/mechanics/auto-dream, https://www.mindstudio.ai/blog/what-is-claude-code-autodream-memory-consolidation-2, https://institute.sfeir.com/en/articles/claude-code-dream-auto-dream-memory-consolidation/

## 7. モデル設定

### 利用可能なモデル

| モデル | 用途 |
|-------|------|
| Opus 4.6 | 最も高性能。複雑なタスク向け |
| Sonnet 4.6 | バランス型。コーディング向け推奨 |
| Haiku | 高速・低コスト。シンプルなタスク向け |

### モデル選択

- `/model` コマンドでセッション中にモデルを切り替え可能
- 推奨: **Plan modeにはOpus**、**コーディングにはSonnet** を使い分ける

### 環境変数によるモデル設定

| 環境変数 | 説明 |
|---------|------|
| `ANTHROPIC_DEFAULT_OPUS_MODEL` | Opusモデルのバージョン指定 |
| `ANTHROPIC_DEFAULT_SONNET_MODEL` | Sonnetモデルのバージョン指定 |
| `ANTHROPIC_DEFAULT_HAIKU_MODEL` | Haikuモデルのバージョン指定 |
| `ANTHROPIC_SMALL_FAST_MODEL` | **非推奨**。`ANTHROPIC_DEFAULT_HAIKU_MODEL` を使用 |

### エフォートレベル（Effort Levels）

「Fast Mode」ではなく「エフォートレベル」として実装されている。

| レベル | シンボル | 説明 |
|-------|---------|------|
| low | ○ | 高速・低コスト。単純なタスク向け |
| medium | ◐ | デフォルト（Opus 4.6、Sonnet 4.6） |
| high | ● | 深い推論。複雑な問題向け |
| max | - | 最も深い推論。トークン使用制限なし |

- 3つのレベル（low, medium, high）はセッション間で持続
- `/effort` または `/model` で変更
- `/effort auto` でリセット
- **対応モデル**: Opus 4.6 と Sonnet 4.6

### デプロイ時のモデルバージョン固定

Bedrock、Vertex AI、Foundry経由でデプロイする場合:
- ユーザーにロールアウトする前にモデルバージョンを固定すること
- 固定しない場合、Claude Codeはモデルエイリアス（sonnet, opus, haiku）を使用し、最新バージョンに解決される

> ソース: https://docs.anthropic.com/en/docs/claude-code/model-config, https://github.com/shanraisshan/claude-code-best-practice

## 8. パーミッションモデル

### 4つのパーミッションモード

| モード | 説明 | 用途 |
|-------|------|------|
| `default` | 標準的なパーミッション動作。allowされていないツールプロンプトは拒否に変換。pre-approvedなツール（allowed_tools、settings.json allowルール、hook）は通常通り実行 | 通常の開発 |
| `acceptEdits` | ファイル操作を自動承認。Claudeがプロンプトなしでコードを編集可能。他のツール（ファイル操作以外のBashコマンドなど）は通常のパーミッションが必要 | プロトタイピング、隔離ディレクトリでの作業 |
| `bypassPermissions` | 全てのパーミッションチェックをバイパス。**ただしdenyルールにマッチした場合はbypassPermissionsモードでもブロックされる** | 完全な自動化 |
| `plan` | ツール実行を完全に防止。Claudeはコードを分析しプランを作成できるが変更は不可 | コードレビュー、変更の事前承認 |

### allowlist（許可リスト）

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test *)",
      "Read(~/.zshrc)"
    ]
  }
}
```

- `allowedTools` は allowルールリストにエントリを追加する
- ツールコールが発生した時に承認/拒否を制御（ツールの利用可能性自体は制御しない）

### deny（拒否リスト）

```json
{
  "permissions": {
    "deny": [
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"
    ]
  }
}
```

- denyルールはファイル探索・検索結果からも除外し、読み取り操作もブロック
- **denyルールはbypassPermissionsモードでも有効**

> ソース: https://docs.anthropic.com/en/docs/claude-code/settings, https://docs.anthropic.com/en/docs/claude-code/sdk/sdk-permissions

## 9. 出力スタイル（Output Styles）

### 設定方法

- `/config` → Output style を選択 → メニューから選択（`.claude/settings.local.json` に保存）
- `/output-style` でメニューにアクセス
- `/output-style [style]` で直接指定（例: `/output-style explanatory`）
- `/output-style:new I want an output style that ...` で新規作成

### ビルトイン出力スタイル

| スタイル | 説明 |
|---------|------|
| **Default** | ソフトウェアエンジニアリングタスクを効率的に完了するよう設計 |
| **Explanatory** | 教育的な「Insights」をタスク実行の合間に提供 |
| **Learning** | 協調的な学習モード。「Insights」に加え、ユーザーに小さな戦略的コードの貢献を求める |

### カスタム出力スタイル

| 場所 | パス |
|------|-----|
| ユーザーレベル | `~/.claude/output-styles/` |
| プロジェクトレベル | `.claude/output-styles/` |

### 動作の仕組み

- 出力スタイルはシステムプロンプトにカスタム指示として追加される
- セッション開始時に設定されるため、変更は次のセッション開始時に反映
- プロンプトキャッシュによりレイテンシとコストを削減するため、会話中はシステムプロンプトが安定的に保持

> ソース: https://docs.anthropic.com/en/docs/claude-code/output-styles

## 10. ステータスラインとコスト管理

### ステータスライン

- `/cost` で現在のトークン使用量を確認
- ステータスラインに継続的にトークン使用量を表示する設定が可能
- Claude.aiのレート制限使用状況（5時間と7日のウィンドウ、used_percentage、resets_at）をステータスラインスクリプトに表示可能

### コスト管理のヒント

- エフォートレベルを下げることでコスト削減可能（`/effort` または `/model` で設定）
- 深い推論が不要なシンプルなタスクにはlowエフォートを使用

> ソース: https://docs.anthropic.com/en/docs/claude-code/costs

## 11. コミュニティベストプラクティス補足

### shanraisshan/claude-code-best-practice からの知見

- **Stop hookの活用**: ターン終了時にClaudeに作業の継続や検証を促す
- **手動 `/compact`**: コンテキスト使用率が50%に達したら手動でコンパクト（Agent Dumb Zoneを回避）
- **`/clear` でコンテキストリセット**: タスクを切り替える際にミッドセッションでリセット
- **2つのSkillパターン**: agent skills（skills:フィールドでプリロード）vs skills（Skillツール経由で呼び出し）

> ソース: https://github.com/shanraisshan/claude-code-best-practice

### affaan-m/everything-claude-code からの知見

- **Skills**: `~/.claude/skills/` に保存。ルールとして機能し、特定のスコープとワークフローに制限される。コードマップを含めることでClaudeがコードベースをコンテキスト消費なしに迅速にナビゲート可能
- **Rules vs CLAUDE.md**: 単一のCLAUDE.mdファイル（ユーザーまたはプロジェクトレベル）か、関心事ごとにグループ化された.mdファイルを持つモジュラーrulesフォルダの2つのアプローチ
- **Subagents**: オーケストレーター（メインClaude）がタスクを委譲。サブエージェントごとにallowed tools、MCP、パーミッションを設定可能

> ソース: https://github.com/affaan-m/everything-claude-code

### FlorianBruniaux/claude-code-ultimate-guide からの知見

- Claude Codeの実行モデルは `while(tool_call)` ループ（分類器、RAGパイプライン、DAGオーケストレーターは不使用）
- 6ヶ月間の日常的な実践に基づくガイド
- プロダクション対応テンプレート、エージェントワークフローガイド、クイズ、チートシートを含む

> ソース: https://github.com/FlorianBruniaux/claude-code-ultimate-guide
