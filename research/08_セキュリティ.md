## 1. 脅威モデル（攻撃面の分類）

### 3つの収束する脅威ベクトル

| 攻撃面 | 説明 | 具体例 |
|--------|------|--------|
| **入力汚染（Input Poisoning）** | PDF、スクリーンショット、メール添付、GitHub PRに埋め込まれたプロンプトインジェクション | 不可視Unicode（ゼロ幅スペース、bidiオーバーライド）、HTMLコメント、Base64エンコーディング |
| **ツール侵害（Tool Compromise）** | MCPサーバーが正常に見えながらデータを窃取。Skillsはサプライチェーン成果物 | MCP Rug Pull攻撃、ツール汚染（tool poisoning） |
| **設定悪用（Configuration Abuse）** | `.claude/`プロジェクト設定、`.mcp.json`、環境変数、hooksがユーザー同意前に実行 | 悪意あるagents、hooks、CLAUDE.md、settings.json |

### `.claude/`フォルダの攻撃ベクトル

リポジトリに`.claude/`を埋め込むことで、事前構成されたエージェント・コマンド・hooksが自動ロードされ、Skillマーケットプレイスを完全にバイパスする。

| ベクトル | メカニズム | リスクレベル |
|---------|-----------|-------------|
| **悪意あるエージェント** | `allowed-tools: ["Bash"]` + 窃取指示 | 高 |
| **悪意あるコマンド** | プロンプト内の隠し指示、引数インジェクション | 高 |
| **悪意あるhooks** | `.claude/hooks/`のBashスクリプトが全ツール呼び出しで実行 | クリティカル |
| **汚染されたCLAUDE.md** | セキュリティ設定をオーバーライドする指示 | 高 |
| **トロイのsettings.json** | 寛容な`permissions.allow`ルール | 高 |

### `.claude/`監査チェックリスト

```bash
# Step 1: 存在確認
ls -la .claude/

# Step 2: hooksのネットワーク呼び出しを確認
grep -r "curl\|wget\|nc\|base64\|eval\|exec" .claude/ 2>/dev/null

# Step 3: Bashアクセス許可を確認
grep -r "allowed-tools.*Bash" .claude/agents/ 2>/dev/null

# Step 4: 過剰な許可がないか確認
grep -r "permissions.allow" .claude/ 2>/dev/null

# Step 5: CLAUDE.mdのオーバーライド指示をスキャン
cat .claude/CLAUDE.md | grep -i "disable\|skip\|override"
```

## 3. プロンプトインジェクション回避技術

### 回避テクニック一覧

| テクニック | メカニズム | 検出方法 | 例 |
|-----------|-----------|---------|-----|
| **ゼロ幅文字** | U+200B, U+200C, U+200D - 人間には不可視 | Unicode正規表現パターンマッチング | `text\u200Bhidden` |
| **右から左オーバーライド** | U+202Eがテキスト表示を反転 | 双方向テキストスキャン | 表示上は正常だが内部は反転 |
| **ANSIエスケープシーケンス** | `\x1b[`ターミナル操作 | エスケープフィルター/ブロック | ターミナル制御コード |
| **Nullバイト切り捨て** | `\x00`が文字列を切り捨て | バリデーションでのNull検出 | 文字列チェックのバイパス |
| **コメント内Base64** | LLMが自動的にデコード | エントロピーチェック（20文字以上の高エントロピー） | `# SGlkZGVuOiBpZ25vcmU=` |
| **ネストされたコマンド実行** | `$(evil_command)`またはバッククォート | パターンブロック + 置換検出 | シェルコマンドのネスト |
| **ホモグリフ置換** | キリル文字`а`がラテン文字`a`を模倣 | Unicode正規化 | キーワードフィルターのバイパス |
| **タグ文字** | 不可視Unicodeブロック U+E0000-U+E007F | 範囲検出 | メタデータインジェクション |

### 検出用正規表現パターン

```regex
# ゼロ幅/RTL/Bidi文字
[\x{200B}-\x{200D}\x{FEFF}\x{202A}-\x{202E}\x{2066}-\x{2069}]

# ANSIエスケープ
\x1b\[|\x1b\]|\x1b\(

# Nullバイト
\x00

# タグ文字
[\x{E0000}-\x{E007F}]

# Base64ペイロード
[#;].*[A-Za-z0-9+/]{20,}={0,2}

# ネストされたコマンド
\$\([^)]+\)|\`[^\`]+\`
```

### MCP固有のインジェクションパターン

| パターン | 説明 |
|---------|------|
| **ツール汚染（Tool Poisoning）** | ツールスキーマ内の悪意あるメタデータがLLMの動作に影響 |
| **Rug Pull攻撃** | 信頼確立後に良性MCPが悪意あるものに変換 |
| **混乱した代理人（Confused Deputy）** | 信頼されていないサーバーが信頼された名前空間でツールを登録 |

### 防御戦略

1. **プリコミット検出** - プロンプトインジェクション検出hookがファイルをスキャン（ロールオーバーライド、ジェイルブレイク、権限偽装）
2. **Unicodeスキャン** - ゼロ幅文字、ANSIエスケープ、Nullバイト、タグ文字を検出
3. **パターンブロック** - ネストされたコマンド（`$()`やバッククォート）、Base64ペイロード
4. **リポジトリプリスキャン** - README、SECURITY.md、package.json、.cursor/、.claude/の隠しHTMLコメントと悪意ある指示をチェック
5. **高リスクファイル検査** - CONTRIBUTING.mdのソーシャルエンジニアリング、SECURITY.mdの隠し指示

### Anthropic公式のサーバーサイド防御

> サーバーサイドのプロンプトインジェクションプローブがツール出力をエージェントコンテキストに入る前にスキャンし、動作をハイジャックしようとする試みが検出された場合に警告を追加する。

## 4. 防御アーキテクチャ（多層防御）

### 3層セキュリティスタック

```
Tier 1（予防 - Prevention）:
  ├─ MCPベッティングワークフロー（5分監査チェックリスト）
  ├─ バージョンピンニング + ハッシュ検証
  ├─ サプライチェーンスキャン（mcp-scan by Snyk）
  ├─ エージェントスキルバリデーション（skills-ref）
  └─ リポジトリプリスキャン（.claude/監査）

Tier 2（検出 - Detection）:
  ├─ プロンプトインジェクション検出hook
  ├─ Unicode/ANSIインジェクションスキャナー
  ├─ シークレット監視（Gitleaks, TruffleHog）
  ├─ 出力スキャナーhook（認証情報漏洩を捕捉）
  └─ .claude/整合性監視（SessionStart hook）

Tier 3（対応 - Response）:
  ├─ 即時認証情報取り消し（AWS, GitHub等）
  ├─ Git履歴監査 + 必要に応じてfilter-repo
  ├─ MCP設定整合性検証
  ├─ 自動セキュリティ監査エージェント
  └─ インシデント文書化 + コンプライアンス報告
```

### 5層防御深度アーキテクチャ（FlorianBruniaux版）

| レイヤー | 説明 |
|---------|------|
| **Layer 1** | MCPバージョンピンニング + ハッシュ検証 |
| **Layer 2** | permissions.denyルールによるRead/Edit/Write/Bashパターンのブロック |
| **Layer 3** | PreToolUse hooksによるセカンダリバリデーション |
| **Layer 4** | 外部シークレット管理（AWS Secrets Manager, 1Password, Vault） |
| **Layer 5** | リポジトリプリスキャン（インジェクションベクトルの開封前チェック） |

### Hook設定例

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": ["~/.claude/hooks/dangerous-actions-blocker.sh"]
      },
      {
        "matcher": "Edit|Write",
        "hooks": [
          "~/.claude/hooks/prompt-injection-detector.sh",
          "~/.claude/hooks/unicode-injection-scanner.sh"
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": ["~/.claude/hooks/output-secrets-scanner.sh"]
      }
    ],
    "SessionStart": ["~/.claude/hooks/mcp-config-integrity.sh"]
  }
}
```

### 危険なコマンドのブロック例（Trail of Bits推奨）

**`rm -rf`のブロック:**
```json
{
  "PreToolUse": [
    {
      "matcher": "Bash",
      "hooks": [
        {
          "type": "command",
          "command": "jq -e '.tool_input.command | test(\"rm\\\\s+-.*f\")' && echo '{\"error\": \"Use trash instead of rm -rf\"}' >&2 && exit 2 || exit 0"
        }
      ]
    }
  ]
}
```

**mainへの直接pushのブロック:**
```json
{
  "PreToolUse": [
    {
      "matcher": "Bash",
      "hooks": [
        {
          "type": "command",
          "command": "jq -e '.tool_input.command | test(\"git push.*main|master\")' && echo '{\"error\": \"Use feature branches, not direct push to main\"}' >&2 && exit 2 || exit 0"
        }
      ]
    }
  ]
}
```

### Anthropic公式のサンドボックスアーキテクチャ

- **ファイルシステム分離**: Claudeは特定のディレクトリのみアクセス・変更可能。書き込みはCWDとそのサブディレクトリに制限
- **ネットワーク分離**: 承認済みサーバーのみに接続可能。プロキシサーバーがドメイン接続制限を強制
- **OS レベルプリミティブ**: macOS Seatbelt、Linux bubblewrap
- **統計**: 内部利用でサンドボックスにより権限プロンプトが **84%削減**
- **クラウドデプロイ**: 各タスクがネットワーク・ファイルシステム制限付きの分離サンドボックスで実行。Git操作はセキュアプロキシ経由

## 5. サプライチェーンセキュリティ

### Snyk ToxicSkills監査結果

3,984件の公開Skillsを調査した結果:

| 指標 | 値 | 詳細 |

| **リモートプロンプト実行** | 2.9% | 外部からのプロンプト実行が可能 |

### MCP Rug Pull攻撃パターン

```
Phase 1: 信頼の確立
  ↓
攻撃者が良性のMCP "code-formatter" を公開
  ↓
ユーザーが ~/.claude.json に追加し、一度承認

Phase 2: 信頼性の構築（2週間）
  ↓
MCPは正常に動作、ユーザーが信頼

Phase 3: 悪用
  ↓
攻撃者が悪意ある更新をプッシュ（再承認不要）
  ↓
MCPが ~/.ssh/*, .env, 認証情報を窃取

根本原因: ワンタイム承認モデル - 更新は再同意なしで自動実行
```

### 緩和策

```
├─ バージョンピンニング（例: @1.2.3、@latestは不可）
├─ ハッシュ検証（バイナリのsha256sum）
├─ 監視（設定変更のトラッキング）
└─ Git hooks（不正な更新の検出）
```

### 5分間MCPベッティングチェックリスト

| ステップ | アクション | 合格基準 |
|---------|-----------|---------|
| 1. ソース検証 | `gh repo view <mcp-repo>` | >50 stars、コミットが30日以内 |
| 2. 権限監査 | mcp.jsonをレビュー | `--dangerous-*`フラグがないこと |
| 3. バージョン | バージョン文字列を確認 | ピンニング済み（"latest"/"main"でない） |
| 4. ハッシュ | `sha256sum <binary>` | リリースチェックサムと一致 |
| 5. 監査 | 最近のコミットをレビュー | 疑わしい変更がないこと |

## 6. キルスイッチと運用安全

### 3レベルキルスイッチシステム

| レベル | メカニズム | トリガー | 実装 |
|--------|-----------|---------|------|
| **1. スコープ付き取り消し** | 特定機能の無効化 | 疑わしい動作 | `dangerous-actions-blocker.sh` hook + `permissions.deny` |
| **2. 速度ガバナー** | レート制限/閾値トリガー | 不安定なエージェント動作 | カスタムhook: 5分間で20回以上のBashコマンド = ブロック |
| **3. グローバルハードストップ** | 即時終了 | 確認された侵害 | Ctrl+C / Esc、`claude config set --disable`、アンインストール |

### プロセスグループキル

- **プロセスグループ終了**: 負のPID（Node.js）による `SIGKILL`
- 子プロセスを含む全プロセスグループを一括終了

### ハートビートベースのデッドマンスイッチ

- ハートビート監視により、エージェントの応答がない場合に自動終了
- 10分間の自動タイムアウト（リモートコントロール時の切断時）

### 速度ガバナーhookの機能

- タイムスタンプ付きのコマンド呼び出しをトラッキング
- 5分間のローリングウィンドウを維持
- 閾値（20コマンド）を超えた場合にブロック
- 暴走エージェントループを防止

### メモリハイジーン

- 高リスクワークフローでは永続メモリを無効化
- プロジェクトメモリとグローバルメモリを分離
- 信頼できない実行の後にメモリをローテーション

### 監視可能な実行

ログ記録対象:
- ツール名、入力サマリー、触れたファイル
- 承認決定、ネットワーク試行
- セッションID

### 承認境界（人間の承認が必要な操作）

- サンドボックス外のシェル実行
- ネットワーク送出（egress）
- シークレットパスの読み取り
- リポジトリ外への書き込み
- ワークフローディスパッチ
- MCPサーバーの同意（自動承認しないこと）

    {
      "matcher": "Bash",

        {
          "type": "command",
          "command": "jq -r '[\"[\"] + (now | todate) + \"]\"] + .tool_input.command' >> ~/.claude/bash-commands.log"

### シークレット露出時の対応プロトコル

**最初の15分:**
1. 即時取り消し（AWS, GitHub, Stripe等）
2. スコープ確認 - リモートにプッシュされたか確認
3. 全Git履歴でパターンを検索

**最初の1時間:**
4. Git履歴で露出ウィンドウを監査
5. ログ内の漏洩キーについて依存関係をスキャン
6. CI/CDログで露出を確認

**最初の24時間:**
7. 関連する全認証情報をローテーション
8. 必要に応じてコンプライアンスに通知
9. インシデントタイムラインを文書化

## 8. Trail of Bitsの3層サンドボックス戦略

### Layer 1: 組み込みサンドボックス（`/sandbox`）

| 項目 | 詳細 |
|------|------|
| **書き込み制限** | 現在の作業ディレクトリとサブディレクトリに制限 |
| **読み取り** | `/sandbox`未有効時は無制限 |
| **ネットワーク** | 明示的に許可されたドメインに制限 |
| **OS分離** | macOS: Seatbelt、Linux: bubblewrap |

### Layer 2: 権限denyルール

| 項目 | 詳細 |
|------|------|
| **適用範囲** | Read/Edit/Writeマッチャーによる認証情報アクセスのブロック |
| **対象** | Claudeの組み込みツールのみに適用 |
| **制限** | `/sandbox`が有効でない場合、直接Bashでバイパス可能 |

### Layer 3: リモート/コンテナ分離

| 項目 | 詳細 |
|------|------|
| **Devcontainer** | プロジェクトファイルのみがマウントされる |
| **リモートドロップレット** | ローカルマシンからの完全分離 |
| **アクセス制限** | ホストファイルシステム、SSHキー、認証情報へのアクセスなし |

### 推奨コンテナベースライン

```yaml
cap_drop: [ALL]
security_opt: [no-new-privileges:true]
```

### Anti-Rationalization Gate（Stop Hook）

Trail of Bitsが提唱する、エージェントが不完全な作業を正当化するのを防ぐStop Hook:

```json
{
  "Stop": [
    {
      "hooks": [
        {
          "type": "prompt",
          "prompt": "You are a JSON-only evaluator. You MUST respond with a single JSON object and NOTHING else. No markdown, no code fences, no explanation, no preamble. Just the raw JSON object.\n\nReview the assistant's final response. Reject if the assistant is rationalizing incomplete work: claiming issues are 'pre-existing' or 'out of scope', saying 'too many issues' to fix, deferring to unrequested 'follow-ups', listing problems without fixing them, or skipping test/lint failures with excuses.\n\nIf any of those patterns apply, respond:\n{\"ok\": false, \"reason\": \"You are rationalizing incomplete work. [specific issue]. Go back and finish.\"}\n\nOtherwise respond:\n{\"ok\": true}"
        }
      ]
    }
  ]
}
```

**重要**: プロンプトは評価者に「生のJSONのみで応答する」ことを明示的に指示する必要がある。

### Hook終了コード

| コード | 動作 |
|--------|------|
| 0 | アクション許可（stdoutがJSON制御として解析される） |
| 1 | エラー、非ブロッキング（stderrがverboseモードで表示） |
| 2 | ブロッキングエラー（stderrがClaudeにエラーメッセージとしてフィードバック） |

### 重要な安全上の注意

- **Hooksはセキュリティ境界ではない** - プロンプトインジェクションはhooksを回避できる
- **bypass-permissionsモードはサンドボックスと組み合わせる** - 最大スループットのため
- **プロジェクト`.mcp.json`ファイルはgitに存在する** - 侵害されたリポジトリが悪意あるMCPサーバーを配布する可能性がある
- **Devcontainerが完全な読み書き分離を提供する**
