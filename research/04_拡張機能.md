## 1. MCP（Model Context Protocol）

### 1.1 アーキテクチャ

MCPはクライアント-サーバーアーキテクチャに基づく。Claude Code（MCPホスト）が各MCPサーバーに対して1つのMCPクライアントを作成し、接続を確立する。

**トランスポートプロトコル:**

| プロトコル | 用途 | 特徴 |
|---|---|---|
| **stdio** | ローカルプロセス間通信 | サブプロセスとしてサーバーを起動。stdin/stdoutでJSON-RPCメッセージを交換。ネットワークオーバーヘッドなし。単一クライアント向け |
| **Streamable HTTP + SSE** | リモートサーバー通信 | HTTP POSTでクライアント→サーバー、SSEでストリーミング。Bearer Token/APIキー等の標準HTTP認証対応。複数クライアント対応 |

> ソース: [MCP Architecture](https://modelcontextprotocol.io/docs/learn/architecture), [MCP Transports](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports)

### 1.2 設定方法

MCPサーバーは3つのスコープで設定可能:

| スコープ | 保存先 | 用途 |
|---|---|---|
| **Local**（デフォルト） | `~/.claude.json` のプロジェクトパス配下 | 個人用、現プロジェクトのみ |
| **Project** | プロジェクトルートの `.mcp.json` | チーム共有、VCS管理可能 |
| **User** | `~/.claude.json` | 全プロジェクト共通の個人設定 |

**CLIでの追加:**
```bash
claude mcp add <server-name> -- <command> [args...]
claude mcp add --scope project my-server -- npx -y @modelcontextprotocol/server-filesystem /path
claude mcp add --scope user my-server -- node server.js
```

**`.mcp.json` の設定例:**
```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/project"],
      "env": {}
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxxx"
      }
    },
    "brave-search": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-brave-search"],
      "env": {
        "BRAVE_API_KEY": "BSA_xxxx"
      }
    }
  }
}
```

> ソース: [Claude Code MCP Docs](https://code.claude.com/docs/en/mcp), [Claude Code Settings](https://code.claude.com/docs/en/settings)

### 1.3 主要MCPサーバー一覧

| サーバー | 説明 |
|---|---|
| **Filesystem** | セキュアなファイル操作、アクセス制御設定可能 |
| **GitHub** | リポジトリ読み取り・検索・操作 |

> ソース: [Top 10 MCP Servers](https://apidog.com/blog/top-10-mcp-servers-for-claude-code/), [MCP Servers Repository](https://github.com/modelcontextprotocol/servers)

### 1.4 Tool Search（遅延ツール読み込み）

MCPツールはデフォルトで**遅延読み込み（deferred）**される:

- セッション開始時にはツール**名のみ**がコンテキストに読み込まれる
- Claudeがタスクに必要なツールを検索ツールで発見した時点でスキーマが読み込まれる
- MCPサーバーを追加してもコンテキストウィンドウへの影響は最小限

**設定オプション:**
- デフォルト: Tool Searchが有効（ツール名のみ読み込み）
- `ENABLE_TOOL_SEARCH=auto`: コンテキストウィンドウの10%以内に収まる場合はスキーマをupfront読み込み、溢れる分のみ遅延

**制限:**
- Tool Search対応モデル: Sonnet 4以降、Opus 4以降
- Haikuモデルは非対応
- MCPツール出力が10,000トークンを超えると警告表示
- `MAX_MCP_OUTPUT_TOKENS`環境変数で最大出力トークン数を調整可能（デフォルト25,000）

**MCPツールの命名規則:**
```
mcp__<server名>__<tool名>
例: mcp__github__search_repositories
     mcp__filesystem__read_file
```

> ソース: [Claude Code MCP](https://code.claude.com/docs/en/mcp), [Manage Costs](https://code.claude.com/docs/en/costs)

### 1.5 デバッグ

```bash
# MCPサーバーの状態確認
claude mcp list

# 特定サーバーの詳細
claude mcp get <server-name>

# サーバーの削除
claude mcp remove <server-name>
```

## 2. Skills

### 2.1 概要

SkillsはClaude Codeの機能を拡張するモジュール式の能力パッケージ。Agent Skills（agentskills.io）のオープンスタンダードに準拠しており、Claude以外のAIプラットフォームでも利用可能。

**動作原理:**
1. Skillがトリガーされると、ClaudeがBashで`SKILL.md`をファイルシステムから読み込む
2. SKILL.mdの指示内容がコンテキストウィンドウに入る
3. 指示が他のファイルを参照していれば、Claudeは追加のBashコマンドでそれらも読み込む

> ソース: [Agent Skills Specification](https://agentskills.io/specification), [Claude Code Skills](https://code.claude.com/docs/en/skills)

### 2.2 SKILL.md 仕様

**ディレクトリ構造:**
```
.claude/skills/
  my-skill/
    SKILL.md          # 必須: スキル定義
    scripts/           # オプション: ヘルパースクリプト
    references/        # オプション: 参照ドキュメント
    assets/            # オプション: リソースファイル
```

**配置場所:**
- プロジェクトスキル: `.claude/skills/` — VCS管理可能、チーム共有
- ユーザースキル: `~/.claude/skills/` — 全プロジェクト共通の個人スキル

### 2.3 フロントマター全項目

```yaml
---
name: my-skill-name              # スキル名（必須推奨）
description: |                   # スキルの説明と使用タイミング（必須推奨）
  このスキルは〇〇の時に使用する
disable-model-invocation: true   # true: ユーザーのみ呼び出し可（Claudeは呼べない）
user-invocable: false            # false: Claudeのみ呼び出し可（ユーザーは呼べない）
effort: high                     # モデルのeffortレベルをオーバーライド
model: sonnet                    # 使用モデル（sonnet/opus/haiku/full ID/inherit）
---

# スキルの指示内容（Markdown）
...
```

**SKILL.md フロントマター例（完全版）:**
```yaml
---
name: testing-patterns
description: |
  テストパターンとベストプラクティスを提供する。
  "テスト" "test" "Jest" "Vitest" に関する作業時に使用。
disable-model-invocation: false
user-invocable: true
effort: medium
model: inherit
---

# テストパターンガイド

## ユニットテスト
テストは `__tests__/` ディレクトリに配置する。
詳細なパターン集は `references/test-patterns.md` を参照。
```

> ソース: [Claude Code Skills](https://code.claude.com/docs/en/skills), [Skills Specification](https://agentskills.io/specification)

### 2.4 Progressive Disclosure（段階的開示）

- **目的:** 必須情報のみSKILL.mdに記載し、詳細資料は別ファイルに分離
- **効果:** 大量のリファレンスドキュメントを事前にコンテキストに読み込まずに済む
- **推奨:** SKILL.mdは**500行以内**に収める
- **超過時:** 詳細リファレンスを`references/`等の別ファイルに分割

**パターン:**
```markdown
# SKILL.md（簡潔に保つ）
API設計ガイドライン。詳細は `references/api-spec.md` を参照。

# references/api-spec.md（必要時のみ読み込み）
## 詳細なAPIエンドポイント仕様
...（大量のドキュメント）
```

### 2.5 トリガー設計

Skillは以下の方法でトリガーされる:

1. **Claudeによる自動発見:** descriptionに基づき、タスクに関連するスキルをClaudeが自動的に読み込む
2. **ユーザーによるスラッシュコマンド:** `/skill-name` で明示的に呼び出し
3. **サブエージェントへの注入:** サブエージェントの`skills`フロントマターで起動時に注入

**スキル推薦エンジン（ChrisWiles/claude-code-showcase）:**
- UserPromptSubmitフックでプロンプトを解析
- キーワード（2点）、正規表現パターン（4点）、ディレクトリマッチ（5点）でスコアリング
- `skill-eval.sh` → `skill-eval.js` → `skill-rules.json` の構成

> ソース: [ChrisWiles/claude-code-showcase](https://github.com/ChrisWiles/claude-code-showcase)

### 2.6 Gotchas（注意点）

- SKILL.mdが500行を超えるとパフォーマンスが低下する
- Claudeのシステムプロンプトは既に約50の個別指示を含む — スキルの追加でさらに増えるため、指示は最小限に
- `disable-model-invocation: true`を設定しないと、Claudeが意図しないタイミングでスキルを呼ぶことがある
- スキルの読み込みはBash経由のため、パスの問題に注意

## 3. Hooks

### 3.1 概要

Hooksはライフサイクルの特定ポイントで実行されるユーザー定義のシェルコマンド/ハンドラ。**決定論的な制御**を提供し、LLMの判断に依存せず確実に実行される。

**4つのハンドラタイプ:**
| タイプ | 説明 |
|---|---|
| **command** | シェルコマンドを実行 |
| **http** | HTTPリクエストを送信 |
| **prompt** | Claudeにプロンプトを注入 |
| **agent** | エージェントを起動 |

> ソース: [Hooks Reference](https://code.claude.com/docs/en/hooks), [Hooks Guide](https://code.claude.com/docs/en/hooks-guide)

### 3.3 Exit Code 制御フロー

Hookは2つのアプローチで制御を行う（同時使用不可）:

**方式1: Exit Code のみ**
```
Exit 0 → 続行（ツール実行を許可）
Exit 2 → ブロック（ツール実行を拒否）
```

**方式2: JSON出力（exit 0 + stdout JSON）**
```
Exit 0 + JSON → JSONの内容に基づく制御
Exit 2 → JSONは無視される
```

**PreToolUse の JSON制御（hookSpecificOutput）:**
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "allow",
    "permissionDecisionReason": "安全な操作です"
  }
}
```

permissionDecision の値:
- `"allow"` — ツール実行を許可
- `"deny"` — ツール実行を拒否
- `"ask"` — ユーザーにエスカレーション

**PostToolUse等の JSON制御（トップレベル）:**
```json
{
  "decision": "block",
  "reason": "この操作は許可されていません"
}
```

**制御フロー図:**
```
Hook実行
  ├── Exit 2 → 即座にブロック（JSON無視）
  └── Exit 0
       ├── stdout にJSON なし → 続行
       └── stdout にJSON あり
            ├── PreToolUse → hookSpecificOutput.permissionDecision で判定
            └── その他 → トップレベル decision で判定
```

> ソース: [Hooks Reference](https://code.claude.com/docs/en/hooks)

### 3.4 settings.json のHooks設定例

**20. MCPヘルスチェック:**
SessionStartフックでMCPサーバーの接続状態を確認。

**21. セキュリティ監査:**
全ツール呼び出しをセキュリティ観点で監査し、怪しいパターンを検出。

> ソース: [Hooks Guide](https://code.claude.com/docs/en/hooks-guide), [disler/claude-code-hooks-mastery](https://github.com/disler/claude-code-hooks-mastery), [karanb192/claude-code-hooks](https://github.com/karanb192/claude-code-hooks), [SmartScope Hooks Guide](https://smartscope.blog/en/generative-ai/claude/claude-code-hooks-guide/)

### 3.6 UV Single-File Script パターン

UV single-fileスクリプトを使うと、仮想環境やrequirements.txtの管理なしにPythonフックを実行できる。

**shebangパターン:**
```python
#!/usr/bin/env -S uv run --script
# /// script
# requires-python = ">=3.11"
# dependencies = ["requests", "pyyaml"]
# ///

import sys
import json

# stdinからフックのJSON入力を読み取り
data = json.load(sys.stdin)
tool_name = data.get("tool_name", "")

# 処理ロジック
if "dangerous" in tool_name:
    print(json.dumps({
        "hookSpecificOutput": {
            "permissionDecision": "deny",
            "permissionDecisionReason": "Dangerous tool detected"
        }
    }))
else:
    sys.exit(0)
```

**settings.json での設定:**
```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "*",
      "hooks": [{
        "type": "command",
        "command": "uv run $CLAUDE_PROJECT_DIR/.claude/hooks/pre_tool_use.py"
      }]
    }]
  }
}
```

**利点:**
- 仮想環境管理が不要
- requirements.txt が不要
- 依存関係がスクリプトファイル内に自己完結
- `$CLAUDE_PROJECT_DIR` でプロジェクトルートからの相対パス解決

> ソース: [pydevtools.com - Claude Code Hooks for uv](https://pydevtools.com/blog/claude-code-hooks-for-uv/), [disler/claude-code-hooks-mastery](https://github.com/disler/claude-code-hooks-mastery)

## 5. サブエージェント

### 5.1 組み込みサブエージェント

Claude Codeには以下の組み込みサブエージェントが含まれる:

| サブエージェント | 説明 | ツール |
|---|---|---|
| **Explore** | 高速・読み取り専用。コードベース検索・分析。thoroughnessレベル: quick/medium/very thorough | 読み取り系のみ |
| **Plan** | planモード用リサーチエージェント。プラン提示前にコンテキスト収集 | 読み取り系のみ |
| **General-purpose** | 複雑なマルチステップタスク。探索と変更の両方が必要な場合 | 全ツール |
| **その他のヘルパーエージェント** | 特定タスク用（自動的に呼び出される） | タスクに応じて |

> ソース: [Create Custom Subagents](https://code.claude.com/docs/en/sub-agents)

### 5.2 カスタムサブエージェント定義

**配置場所:**
- プロジェクト: `.claude/agents/my-agent.md` — VCS管理可能
- ユーザー: `~/.claude/agents/my-agent.md` — 全プロジェクト共通

### 5.3 フロントマター全16項目

```yaml
---
# === 必須フィールド ===
name: code-reviewer                    # 一意の識別子（小文字+ハイフン）
description: |                         # Claudeがこのエージェントに委任するタイミング
  コードレビューを行う。PRの変更を分析し、
  バグ、セキュリティ問題、改善提案を提示する。

# === モデル・パフォーマンス ===
model: sonnet                          # sonnet/opus/haiku/full ID/inherit（デフォルト: inherit）
effort: high                           # モデルのeffortレベル
maxTurns: 20                           # 最大ターン数

# === ツール制御 ===
tools:                                 # 使用可能ツール（省略時は全ツール継承）
  - Read
  - Grep
  - Glob
  - Bash
disallowedTools:                       # 拒否ツール（tools/継承リストから除去）
  - Edit
  - Write

# === 権限・セキュリティ ===
permissionMode: plan                   # default/acceptEdits/dontAsk/bypassPermissions/plan

# === 外部接続 ===
mcpServers:                            # 利用可能なMCPサーバー
  github:
    command: npx
    args: ["-y", "@modelcontextprotocol/server-github"]
skills:                                # 起動時に注入するスキル
  - testing-patterns
  - code-style

# === ライフサイクル ===
hooks:                                 # サブエージェントのライフサイクルフック
  PostToolUse:
    - matcher: Edit
      hooks:
        - type: command
          command: "npx prettier --write $(jq -r '.tool_input.file_path')"

# === 高度な設定 ===
memory: persistent                     # メモリ設定
background: |                          # バックグラウンドコンテキスト
  このプロジェクトはTypeScript + React。
  ESLintとPrettierで品質管理。
isolation: worktree                    # 分離設定（worktree）

# === プロンプト（frontmatter外） ===
# initialPrompt は --agents フラグのJSON形式でのみ使用
---

# コードレビューエージェント

## 手順
1. `git diff` で変更を確認
2. 各ファイルの変更を分析
3. バグ、セキュリティ問題、改善点を報告
4. レビュー結果をMarkdownで出力
```

### 5.4 設計原則

- **最小権限の原則:** 必要なツールのみを付与（`tools` / `disallowedTools`で制御）
- **スキル注入:** `skills`で起動時にドメイン知識を注入（実行中にスキル発見する必要がない）
- **ライフサイクルフック:** `hooks`でサブエージェント固有のフックを定義（そのサブエージェントがアクティブな時のみ実行）
- **worktree分離:** `isolation: worktree` でサブエージェントごとに独立したworktreeを作成（変更なしで完了時は自動クリーンアップ）
- **プラグインサブエージェントの制限:** プラグインからのエージェントは `hooks`, `mcpServers`, `permissionMode` が無視される

> ソース: [Create Custom Subagents](https://code.claude.com/docs/en/sub-agents), [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice)

## 6. エージェントチーム

### 6.1 概要

エージェントチームは複数のClaude Codeインスタンスを協調動作させる**実験的機能**（デフォルトで無効）。

**有効化:**
```bash
# 環境変数で設定
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# または settings.json で設定
```

### 6.2 仕組み

- 1つのセッションが**チームリード**として機能し、作業の調整・タスク割り当て・結果統合を行う
- チームメイトは**独立して動作**し、それぞれ独自のコンテキストウィンドウを持つ
- チームメイト間で**直接メッセージング**が可能
- サブエージェントとの違い: サブエージェントは単一セッション内で動作しメインエージェントにのみ報告するが、チームメイトには個別に直接対話可能

### 6.3 最適なユースケース

| ユースケース | 説明 |
|---|---|
| **リサーチ＆レビュー** | 複数のチームメイトが問題の異なる側面を同時調査、発見を共有・挑戦 |
| **新モジュール/機能** | チームメイトがそれぞれ別の部分を担当（干渉なし） |
| **競合仮説デバッグ** | チームメイトが並列に異なる仮説をテスト、早期収束 |
| **クロスレイヤー調整** | フロントエンド、バックエンド、テストの変更を各チームメイトが担当 |

### 6.4 コスト倍率

- **標準セッション比で約7倍のトークン使用**（planモードのチームメイト）
- 各チームメイトが独自のコンテキストウィンドウを維持
- アクティブチームメイト数と各チームメイトの実行時間に比例してスケール

### 6.5 コスト管理戦略

- チームメイトには**Sonnet**を使用（能力とコストのバランス）
- 作業完了後はチームを**クリーンアップ**（アイドル状態でもトークン消費継続）
- タスクを**小さく自己完結的**に保つ

### 6.6 ガードレール

- プロンプト自体にガードレールを追加して制御
- 各タスクに独自の**permissionMode**を設定可能
- worktree分離でファイル競合を防止

### 6.7 既知の制限事項

- セッション再開の問題
- タスクステータスの遅延
- シャットダウンの遅延
- **1セッションにつき1チーム**のみ
- ネストしたチームは不可

> ソース: [Agent Teams](https://code.claude.com/docs/en/agent-teams), [Manage Costs](https://code.claude.com/docs/en/costs)

## 7. 拡張レイヤーの確実性比較

| レイヤー | 確実性 | 説明 |
|---|---|---|
| **CLAUDE.md** | ~80% | LLMの指示追従に依存。システムプロンプトに約50の指示が既にあり、追加指示が多すぎると追従率が低下 |
| **Skills** | 50-80% | Claudeがdescriptionに基づき自動発見するため、適切なタイミングで読み込まれない可能性あり。`disable-model-invocation`で制御可能 |
| **Hooks** | **100%** | 決定論的実行。イベント発火時に**毎回必ず実行**される。プロンプトの表現やモデルの判断に関係なく確実に動作 |
| **Subagents** | ~80-90% | Claudeの委任判断に依存するが、明示的な呼び出しも可能 |

**設計指針:**
- **絶対に守るべきルール** → Hooks（exit code 2でブロック）
- **強く推奨するガイドライン** → CLAUDE.md
- **ドメイン知識の提供** → Skills
- **複雑なタスクの委任** → Subagents

> ソース: [Extend Claude Code](https://code.claude.com/docs/en/features-overview), [Best Practices](https://code.claude.com/docs/en/best-practices), [HumanLayer Blog](https://www.humanlayer.dev/blog/writing-a-good-claude-md)

## 8. コンテキストコスト比較

| レイヤー | コンテキストコスト | 詳細 |
|---|---|---|
| **MCP** | **最大（高い）** | ツール定義のスキーマがコンテキストに入る。Tool Searchで軽減可能だが、実際にツールを使用するとスキーマ+結果がコンテキストを消費。MAX_MCP_OUTPUT_TOKENS=25,000がデフォルト上限 |
| **Agent定義** | **中〜高** | サブエージェントのプロンプト・設定がコンテキストに入る。ただしサブエージェントは独自のコンテキストウィンドウで動作するため、メインウィンドウへの影響は委任指示分のみ |
| **Skills** | **低〜中** | SKILL.md読み込み時のみコンテキスト消費。Progressive Disclosureで必要な時だけ追加ファイルを読み込み。500行以内推奨 |
| **Hooks** | **ほぼゼロ** | Hooksはコンテキスト外で実行される。JSON入出力のみで、コンテキストウィンドウを消費しない |
| **CLAUDE.md** | **低** | セッション開始時に一度読み込まれる。簡潔に保つことが重要 |

**コスト順:**
```
MCP >> Agent定義 >> Skills >> CLAUDE.md >> Hooks ≒ 0
```

**最適化戦略:**
- MCP: Tool Search（デフォルト有効）でツール名のみ先行読み込み
- Skills: Progressive Disclosureで段階的読み込み
- CLAUDE.md: 普遍的に適用される最小限の指示のみ記載
- Hooks: コンテキストコストなし、積極的に活用すべき

> ソース: [Manage Costs](https://code.claude.com/docs/en/costs), [Claude Code MCP](https://code.claude.com/docs/en/mcp), [How Claude Code Works](https://code.claude.com/docs/en/how-claude-code-works)

## 9. Trail of Bits セキュリティ設定参考

### 9.1 claude-code-config の推奨設定

**危険コマンドの拒否:**
- `rm -rf`, `sudo`, `mkfs`, `dd`, force git push

**機密ファイルへのアクセス拒否:**
- `~/.ssh/**`, `~/.gnupg/**` — SSH鍵・暗号資格情報
- `~/.aws/**`, `~/.azure/**` — クラウドプロバイダー資格情報
- `~/.npmrc`, `~/.pypirc` — パッケージマネージャー資格情報
- `~/.bashrc`, `~/.zshrc` — シェル設定ファイル

**サンドボックス:**
- `/sandbox`なし: deny ルールはClaude組み込みツールのみブロック、Bashコマンドはバイパス可能
- `/sandbox`あり: OSレベル（Seatbelt/bubblewrap）で同じルールを強制、Bashもブロック
- 完全分離: devcontainerを使用

> ソース: [trailofbits/claude-code-config](https://github.com/trailofbits/claude-code-config), [trailofbits/claude-code-devcontainer](https://github.com/trailofbits/claude-code-devcontainer)
