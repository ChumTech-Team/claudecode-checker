# 01 基礎と設定 -- config-evaluator ナレッジベース

> 対象: Claude Code の基礎概念、セットアップ、設定ファイル、メモリ、モデル、パーミッション、出力スタイル
> 作成日: 2026-03-28

---

## 1. アーキテクチャ

### R-001: エージェントループ構造
Claude Code は `while(tool_call)` ループで動作する。分類器、RAGパイプライン、DAGオーケストレーターは使用しない。

### R-002: 3フェーズ動作
タスクは以下の順に進むが、明確に分離されず流動的に遷移する。
1. コンテキスト収集（ファイル検索でコードを理解）
2. アクション実行（編集・変更の適用）
3. 結果検証（テスト実行で動作確認）

### R-003: ツールの実行場所
| 種別 | 実行場所 | 例 |
|------|---------|-----|
| クライアントツール | ユーザーのアプリケーション | bash, text_editor |
| サーバーツール | Anthropicインフラ | web_search, code_execution, web_fetch, tool_search |
| 拡張ツール（MCP） | 外部サービス | Google Drive, Jira, Slack |

### R-004: マルチエージェント
Task tool でサブエージェントを生成し、タスクの異なる部分を並列処理可能。

### R-005: 拡張システムの4種類
| 方法 | 説明 |
|------|------|
| Skills | Markdownファイルで知識をオンデマンド拡張 |
| MCP | 外部サービス接続 |
| Hooks | シェルスクリプトをトリガーベースで実行（決定論的） |
| Subagents | 独自コンテキストウィンドウを持つ隔離されたサブエージェント |

> 情報源: https://docs.anthropic.com/en/docs/agents-and-tools/claude-code/overview, https://github.com/FlorianBruniaux/claude-code-ultimate-guide

## 2. セットアップ

### R-006: システム要件
| OS | バージョン |
|----|-----------|
| macOS | 10.15+ |
| Ubuntu/Debian | Ubuntu 20.04+ / Debian 10+ |
| Windows | 10+（WSL 1/2 または Git for Windows 必須） |

### R-007: インストール方法
ネイティブインストーラーを推奨。npm版は非推奨。

### R-008: npm版の制約
- Node.js 18+ が必要
- `sudo npm install -g` は使用禁止（パーミッション問題・セキュリティリスク）

### R-009: 認証要件
- Pro、Max、Teams、Enterprise、または Console アカウントが必要
- 無料の Claude.ai プランでは利用不可
- デフォルト認証: Anthropic Console 経由の OAuth（console.anthropic.com でのアクティブな課金が必要）

### R-010: サードパーティプロバイダー
Amazon Bedrock、Google Vertex AI、Microsoft Foundry でも利用可能。

> 情報源: https://docs.anthropic.com/en/docs/claude-code/setup

ビルトインコマンドとは異なり、プロンプトベースでプレイブックを与えツールを使って作業を遂行させる。
- `/simplify` -- 最近変更されたファイルのコード再利用・品質・効率の問題をレビュー・修正

### R-013: キーボードショートカット
| ショートカット | 機能 |
|---------------|------|
| `Shift+Enter` | 改行挿入 |
| `Ctrl+R` | コマンド履歴検索 |
| `Ctrl+C` | 現在の処理キャンセル |
| `Ctrl+O` | トランスクリプトビューアー トグル |
| `Escape` | 入力キャンセル/モード切替 |
| `/` | スラッシュコマンドメニュー表示 |

### R-014: Shift+Enter のターミナル互換性
- 設定不要: iTerm2, WezTerm, Ghostty, Kitty
- `/terminal-setup` が必要: VS Code, Alacritty, Zed, Warp

### R-015: Option/Alt キーのターミナル設定
| ターミナル | 設定 |
|-----------|------|
| Terminal.app | 設定 > プロファイル > キーボード > 「Use Option as Meta Key」 |
| VS Code | 設定 > プロファイル > Keys > Left/Right Option key を「Esc+」 |
| iTerm2 | 設定不要 |

### R-016: vim モード
- `/vim` コマンドまたは `/config` で永続的に有効化
- normal mode でカーソルが先頭/末尾の場合、矢印キーでコマンド履歴ナビゲート

### R-017: カスタムキーバインド
`/keybindings` で `~/.claude/keybindings.json` を作成/オープン。

```json
{
  "ctrl+shift+r": "/review",
  "ctrl+shift+c": "/compact",
  "ctrl+shift+p": "/plan"
}
```

### R-018: コードバインディング（Chord Sequences）
- 2つのキーを順番に押す（同時ではない）
- スペースで区切って定義
- 1500ms 以内に2番目のキーが押されなければキャンセル

### R-019: キーバインドの即時適用
`keybindings.json` の変更は自動検出・即時適用（再起動不要）。

### R-020: キーバインド診断
`/doctor` でキーバインド警告を確認。

> 情報源: https://docs.anthropic.com/en/docs/claude-code/interactive-mode, https://docs.anthropic.com/en/docs/claude-code/slash-commands, https://code.claude.com/docs/en/keybindings

## 4. CLAUDE.md

### R-021: CLAUDE.md の役割
会話開始時に読み込まれる特別なファイル。Bashコマンド、コードスタイル、ワークフロールールなどの永続的コンテキストを提供する。

### R-022: 配置場所とスコープ
| 場所 | スコープ | 用途 |
|------|---------|------|
| `~/.claude/CLAUDE.md` | 全セッション | グローバル設定 |
| `./CLAUDE.md`（プロジェクトルート） | プロジェクト | git にチェックインしてチーム共有 |
| 親ディレクトリ | モノレポ | モノレポ全体の設定 |
| 子ディレクトリ | サブディレクトリ | そのディレクトリのファイル操作時にオンデマンド読み込み |

### R-023: 読み込み順序
起動時にカレントワーキングディレクトリからファイルシステムのルートに向かって上方走査する。

### R-024: インポート構文
`@path/to/import` で追加ファイルをインポート可能。

### R-025: 遵守率はルール量に依存する
CLAUDE.md の遵守率は行数・ルール数に応じて劣化する。命令数を2倍にすると遵守率は約半減する（Linear Decay）。エージェントシナリオでの全ルール完全遵守率は30%未満。詳細な劣化カーブとコンテキスト管理については `02_コンテキストエンジニアリング.md`（R-017〜R-028）を参照。

### R-026: インストラクション予算
- フロンティアLLMが確実に従える命令数の上限は約150-200
- システムプロンプトが既に約50を使用
- ユーザーが使える実質的な枠は**約100-150**
- 200行を超えるCLAUDE.mdは最適パラメータを約33%超過している

### R-027: 簡潔に保つ
肥大化した CLAUDE.md は指示を無視させる原因になる。各行について「これを削除したら Claude がミスを犯すか？」と自問し、NO なら削除する。

### R-028: コピーよりポインタ
コードスニペットを CLAUDE.md に含めない（すぐに古くなる）。`file:line` 参照を使用する。

### R-029: 反復的な改善
Claude がミスしたら「CLAUDE.md を更新してこのミスを繰り返さないようにして」と指示する。時間とともに実際のミスに基づいた生きたドキュメントになる。

### R-030: Skills で専門知識を切り離す
CLAUDE.md は毎セッション読み込まれるが、Skills は関連する時のみ読み込まれる。コンテキストをリーンに保つ。

### R-031: Hooks で確実性を担保
CLAUDE.md の指示は助言的だが Hooks は決定論的。100%実行が必要なアクションには Hooks を使用する。

### R-032: CLAUDE.md 記述テンプレート
```markdown
# プロジェクト概要
[プロジェクトの目的と技術スタック]

# コードスタイル
- [具体的なスタイルルール]

# ビルド・テストコマンド
- `npm run build` - ビルド
- `npm run test` - テスト
- `npm run lint` - リント

# アーキテクチャ
- src/components/ - UIコンポーネント
- src/lib/ - ユーティリティ
- src/api/ - APIルート

# 重要なルール
- [必ず守るべきルール]
```

> 情報源: https://code.claude.com/docs/en/best-practices, https://github.com/shanraisshan/claude-code-best-practice

## 5. settings.json

### R-033: 設定ファイルの優先順位
上が最高優先でマージされる。

| 優先順位 | ファイル | 説明 |
|---------|--------|------|
| 1（最高） | `.claude/settings.local.json` | ローカル設定（git にコミットしない） |
| 2 | `.claude/settings.json` | プロジェクト設定（git にコミットしてチーム共有） |
| 3（最低） | `~/.claude/settings.json` | ユーザー設定（全プロジェクトに適用） |
| 上書き | プログラマティックオプション（agents, allowed_tools） | 常にファイルシステム設定を上書き |

### R-034: 優先順位の注意点
ユーザー設定で allow されていてもプロジェクト設定で deny されていればブロックされる。

### R-035: JSON Schema バリデーション
`$schema` 行を追加すると VS Code、Cursor などでオートコンプリートとインラインバリデーションが有効になる。

### R-036: 主要な設定キー
| キー | 説明 |
|-----|------|
| `permissions` | allow/deny/ask ルールの配列 |
| `env` | 環境変数（セッションごとに適用） |
| `hooks` | トリガーベースの自動化 |
| `skills` | Skill 拡張の設定 |
| `mcpServers` | MCP サーバー接続設定 |
| `allowedTools` | ツール許可リスト（allow ルールリストにエントリ追加） |
| `effort` | エフォートレベル |
| `colorTheme` | カラーテーマ |
| `autoMemoryEnabled` | 自動メモリの有効/無効 |
| `allowManagedHooksOnly` | マネージド設定でのみ設定可能 |

### R-037: permissions 設定例
```json
{
  "$schema": "https://cdn.anthropic.com/claude-code/settings.schema.json",
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test *)",
      "Read(~/.zshrc)"
    ],
    "deny": [
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"
    ]
  }
}
```

### R-038: ルール評価順序
1. **deny** ルールが最初に評価
2. 次に **ask** ルール
3. 最後に **allow** ルール
4. 最初にマッチしたルールが適用

### R-039: ルールの形式
`Tool` または `Tool(specifier)` の形式で記述する。

### R-040: Hook イベント一覧
| イベント | 説明 |
|---------|------|
| `PreToolUse` | ツール使用前 |
| `PostToolUse` | ツール使用後 |
| `PostToolUseFailure` | ツール使用失敗後 |
| `Notification` | 通知時 |
| `UserPromptSubmit` | ユーザープロンプト送信時 |
| `SessionStart` | セッション開始時 |
| `SessionEnd` | セッション終了時 |
| `Stop` | 停止時 |
| `SubagentStart` | サブエージェント開始時 |
| `SubagentStop` | サブエージェント停止時 |
| `PreCompact` | コンパクト前 |
| `PermissionRequest` | パーミッション要求時 |
| `Setup` | セットアップ時 |
| `TeammateIdle` | チームメイトアイドル時 |
| `TaskCompleted` | タスク完了時 |
| `ConfigChange` | 設定変更時 |
| `WorktreeCreate` | ワークツリー作成時 |
| `WorktreeRemove` | ワークツリー削除時 |

> 情報源: https://docs.anthropic.com/en/docs/claude-code/settings

## 6. メモリシステム

### R-041: 2つのメモリメカニズム
| メカニズム | 管理者 | 説明 |
|-----------|--------|------|
| CLAUDE.md | ユーザー | ユーザーが書く永続的インストラクション |
| Auto Memory | Claude自身 | 自動的に学習内容を記録 |

両方とも会話開始時にコンテキストとして読み込まれるが、強制的な設定ではない。

### R-042: Auto Memory はデフォルト有効
セッション間で知識を蓄積する（ビルドコマンド、デバッグ知見、アーキテクチャノート、コードスタイル、ワークフロー習慣）。

### R-043: Auto Memory のトグル方法
- セッション内で `/memory` を開きトグル
- `autoMemoryEnabled` 設定で制御

### R-044: メモリストレージの場所
`~/.claude/projects/<project>/memory/`
- `<project>` パスは git リポジトリから導出
- 同一リポジトリの全ワークツリーとサブディレクトリで1つの auto memory ディレクトリを共有

### R-045: メモリファイルの読み込み制限
**MEMORY.md の最初の200行、または最初の25KB**（いずれか先に到達した方）が会話開始時に読み込まれる。しきい値を超えたコンテンツは読み込まれない。

### R-046: メモリの簡潔化
Claude は詳細なノートを個別のトピックファイルに移動して MEMORY.md を簡潔に保つ。

### R-047: Auto Dream の概要
バックグラウンドのメモリ統合機能。メモリファイルの自動プルーニング、マージ、リフレッシュを行う。

### R-048: Auto Dream の動作条件
24時間ごとに実行（5セッション以上の後）。

### R-049: Auto Dream が解決する問題
20セッション以上経過すると、矛盾する記述、意味を失った相対日付（「昨日」等）、存在しないファイルへの参照で混乱する。

### R-050: Auto Dream の4フェーズ
1. **スキャン**: メモリディレクトリと MEMORY.md を読み取り現状把握
2. **マップ構築**: 既存内容のマップを作成
3. **パターン検索**: 最近のセッションのトランスクリプト（JSONL）から高価値パターンを特定（ユーザー修正、明示的保存、繰り返しテーマ、重要な決定）
4. **統合**: 重複排除、古いノート削除、相対日付を絶対日付に変換、MEMORY.md を200行以下に維持

### R-051: Auto Dream のパフォーマンス
913セッション分のメモリを約8-9分で統合した事例あり。

### R-052: Auto Dream の実行タイミング
アクティブセッション中ではなく、セッション間で実行される（人間のREM睡眠が短期記憶を長期記憶に統合するプロセスと同様）。

> 情報源: https://docs.anthropic.com/en/docs/claude-code/memory, https://claudefa.st/blog/guide/mechanics/auto-dream

## 7. モデル設定

### R-053: 利用可能なモデル
| モデル | 用途 |
|-------|------|
| Opus 4.6 | 最も高性能。複雑なタスク向け |
| Sonnet 4.6 | バランス型。コーディング向け推奨 |
| Haiku | 高速・低コスト。シンプルなタスク向け |

### R-054: モデル使い分け推奨
Plan mode には Opus、コーディングには Sonnet を使い分ける。

### R-055: モデル切り替え
`/model` コマンドでセッション中にモデルを切り替え可能。

### R-056: モデル指定の環境変数
| 環境変数 | 説明 |
|---------|------|
| `ANTHROPIC_DEFAULT_OPUS_MODEL` | Opus モデルのバージョン指定 |
| `ANTHROPIC_DEFAULT_SONNET_MODEL` | Sonnet モデルのバージョン指定 |
| `ANTHROPIC_DEFAULT_HAIKU_MODEL` | Haiku モデルのバージョン指定 |

### R-057: 非推奨の環境変数
`ANTHROPIC_SMALL_FAST_MODEL` は非推奨。`ANTHROPIC_DEFAULT_HAIKU_MODEL` を使用すること。

### R-058: エフォートレベル
| レベル | シンボル | 説明 |
|-------|---------|------|
| low | ○ | 高速・低コスト。単純タスク向け |
| medium | ◐ | デフォルト（Opus 4.6、Sonnet 4.6） |
| high | ● | 深い推論。複雑な問題向け |
| max | - | 最も深い推論。トークン使用制限なし |

### R-059: エフォートレベルの持続性
low, medium, high の3レベルはセッション間で持続する。`/effort auto` でリセット。

### R-060: エフォートレベルの対応モデル
Opus 4.6 と Sonnet 4.6 が対応。

### R-061: デプロイ時のモデルバージョン固定
Bedrock、Vertex AI、Foundry 経由でデプロイする場合、ユーザーにロールアウトする前にモデルバージョンを固定すること。固定しない場合、モデルエイリアス（sonnet, opus, haiku）が最新バージョンに解決される。

> 情報源: https://docs.anthropic.com/en/docs/claude-code/model-config, https://github.com/shanraisshan/claude-code-best-practice

## 8. パーミッションモデル

### R-062: 4つのパーミッションモード
| モード | 説明 |
|-------|------|
| `default` | 標準動作。allow されていないツールプロンプトは拒否に変換 |
| `acceptEdits` | ファイル操作を自動承認。他のツールは通常のパーミッションが必要 |
| `bypassPermissions` | 全パーミッションチェックをバイパス。**ただし deny ルールは有効** |
| `plan` | ツール実行を完全防止。分析とプラン作成のみ |

### R-063: deny ルールは bypassPermissions でも有効
deny ルールにマッチした場合、bypassPermissions モードでもブロックされる。

### R-064: allowedTools の動作
`allowedTools` は allow ルールリストにエントリを追加する。ツールコール発生時に承認/拒否を制御するが、ツールの利用可能性自体は制御しない。

### R-065: deny ルールの効果
deny ルールはファイル探索・検索結果からも除外し、読み取り操作もブロックする。

> 情報源: https://docs.anthropic.com/en/docs/claude-code/settings, https://docs.anthropic.com/en/docs/claude-code/sdk/sdk-permissions

## 9. 出力スタイル

### R-066: 出力スタイルの設定方法
- `/config` > Output style を選択
- `/output-style` でメニューアクセス
- `/output-style [style]` で直接指定
- `/output-style:new` で新規作成

### R-067: ビルトイン出力スタイル
| スタイル | 説明 |
|---------|------|
| Default | ソフトウェアエンジニアリングタスクを効率的に完了 |
| Explanatory | 教育的な「Insights」をタスク実行の合間に提供 |
| Learning | 協調的学習モード。Insights + ユーザーに小さなコード貢献を求める |

### R-068: カスタム出力スタイルの保存場所
| レベル | パス |
|-------|-----|
| ユーザー | `~/.claude/output-styles/` |
| プロジェクト | `.claude/output-styles/` |

### R-069: 出力スタイルの動作
- システムプロンプトにカスタム指示として追加
- セッション開始時に設定、変更は次セッション開始時に反映
- プロンプトキャッシュによりレイテンシとコストを削減

> 情報源: https://docs.anthropic.com/en/docs/claude-code/output-styles

## 10. コスト管理

### R-070: トークン使用量の確認
`/cost` で現在のトークン使用量を確認。ステータスラインに継続表示する設定も可能。

### R-071: レート制限情報の表示
Claude.ai のレート制限使用状況（5時間と7日のウィンドウ、used_percentage、resets_at）をステータスラインスクリプトに表示可能。

### R-072: コスト削減方法
エフォートレベルを下げることでコスト削減可能。深い推論が不要なシンプルタスクには low エフォートを使用する。

> 情報源: https://docs.anthropic.com/en/docs/claude-code/costs

---

## 11. コミュニティベストプラクティス

### R-073: Stop hook の活用
ターン終了時に Claude に作業の継続や検証を促す。

### R-074: 手動 /compact の推奨タイミング
コンテキスト使用率が50%に達したら手動でコンパクト（Agent Dumb Zone を回避）。

### R-075: /clear でコンテキストリセット
タスクを切り替える際にミッドセッションでリセットする。

### R-076: 2つの Skill パターン
- agent skills: `skills:` フィールドでプリロード
- skills: Skill ツール経由で呼び出し

### R-077: Skills の保存場所と機能
`~/.claude/skills/` に保存。ルールとして機能し、特定のスコープとワークフローに制限される。コードマップを含めることで Claude がコードベースをコンテキスト消費なしに迅速にナビゲート可能。

### R-078: Rules vs CLAUDE.md
2つのアプローチがある:
1. 単一の CLAUDE.md ファイル（ユーザーまたはプロジェクトレベル）
2. 関心事ごとにグループ化された .md ファイルを持つモジュラー rules フォルダ

### R-079: Subagents の構成
オーケストレーター（メイン Claude）がタスクを委譲。サブエージェントごとに allowed tools、MCP、パーミッションを設定可能。

> 情報源: https://github.com/shanraisshan/claude-code-best-practice, https://github.com/affaan-m/everything-claude-code, https://github.com/FlorianBruniaux/claude-code-ultimate-guide

## config-evaluator 用 評価チェックリスト

### セットアップ
- [ ] システム要件を満たしているか（R-006）
- [ ] ネイティブインストーラーを使用しているか（R-007）
- [ ] `sudo npm install -g` を使用していないか（R-008）
- [ ] 適切な認証アカウントがあるか（R-009）

### CLAUDE.md
- [ ] CLAUDE.md が存在するか（R-021）
- [ ] 適切な場所に配置されているか（R-022）
- [ ] 200行以内に収まっているか（R-025, R-026; 詳細は02_コンテキストエンジニアリング R-017〜R-028）
- [ ] インストラクション数が100-150以内に収まっているか（R-026）
- [ ] 簡潔に記述されているか（不要な行がないか）（R-027）
- [ ] コードスニペットではなくファイル参照を使用しているか（R-028）
- [ ] WHAT/WHY/HOW フレームワークに沿っているか（R-032）
- [ ] 100%実行必須のルールは Hooks に移行しているか（R-031）
