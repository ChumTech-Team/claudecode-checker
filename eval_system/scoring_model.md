# Claude Code 活用度スコアリングモデル

> 全evaluatorが共通で使用するスコアリング基準。
> ユーザーのClaude Code活用状況を定量的に評価し、改善ポイントを明確にする。

---

## 1. スコアリング構造

### 1.1 全体構造

```
総合スコア（1000点満点）
├── 01 基礎と設定        (config)      : 150点
├── 02 コンテキスト管理    (context)     : 200点  ★最重要
├── 03 コア機能活用       (core)        : 100点
├── 04 拡張機能活用       (extension)   : 150点
├── 05 自動化・並列実行    (automation)  : 100点
├── 06 ワークフロー       (workflow)    : 100点
├── 07 業務適用度         (business)    : 50点
├── 08 セキュリティ       (security)    : 100点
└── 09 コスト・運用       (cost)        : 50点
```

### 1.2 配点の根拠

| ドメイン | 配点 | 理由 |
|---------|------|------|
| コンテキスト管理 | 200 | 調査で「最も差がつくスキル」と判明。遵守率・トークン効率に直結 |
| 基礎と設定 | 150 | CLAUDE.md/settings.jsonの品質が全ドメインの土台 |
| 拡張機能 | 150 | MCP/Skills/Hooks/Agentsの活用が生産性を決定づける |
| セキュリティ | 100 | リスク管理。0点だと他が満点でも危険 |
| 自動化・並列実行 | 100 | 繰り返し作業の排除と並列化 |
| ワークフロー | 100 | 開発方法論の適切な選択と実践 |
| コア機能 | 100 | 基本ツールの正しい使い方 |
| 業務適用度 | 50 | 実際の業務にどれだけ適用しているか |
| コスト・運用 | 50 | コスト意識と持続可能な運用 |

## 3. ドメイン別スコアリング基準

### 3.1 各チェック項目の評価方法

各チェック項目は以下の3段階で評価する:

| 評価 | 点数 | 基準 |
|------|------|------|
| **未実施** | 0 | 該当する設定・運用が存在しない |
| **部分的** | 配点の50% | 存在するが不完全、または最適でない |
| **完全** | 配点の100% | ベストプラクティスに沿って実施されている |

### 3.2 セキュリティの特別ルール

セキュリティドメイン（08）には**致命的欠陥ペナルティ**がある:
- denyルール未設定（~/.ssh/**, ~/.aws/**等の保護なし）→ **セキュリティスコアを0点に強制**
- enableAllProjectMcpServers: true のまま → **セキュリティスコアから-50点**

理由: セキュリティの基本が欠けていると、他の高得点が無意味になるため。

## 5. 評価実行プロトコル

### 5.1 Orchestratorのフロー

```
1. ユーザーから評価リクエストを受ける
2. 評価対象の情報を収集:
   - CLAUDE.md の内容
   - settings.json / settings.local.json
   - .mcp.json
   - .claude/agents/ の定義
   - .claude/skills/ の定義
   - hooks設定
   - Git設定とワークフロー
   - 最近のセッション履歴（可能なら）
3. 各evaluatorに該当データを振り分け
4. 各evaluatorが独立してスコアリング
5. Orchestratorが結果を統合し、レポート生成
```

### 5.2 各Evaluatorの評価手順

```
1. 自ドメインのナレッジファイル（01〜09）をロード（<150ルール）
2. 評価対象データを受け取る
3. チェックリストの各項目を 未実施(0) / 部分的(50%) / 完全(100%) で判定
4. セキュリティの致命的欠陥チェック
5. ドメインスコアと改善提案を返す
```

### 5.3 出力形式

## 6. 改善優先度の算出

### 6.1 優先度スコア

```
改善優先度 = (配点 - 現スコア) × 実現容易度
```

| 実現容易度 | 値 | 例 |
|-----------|-----|-----|
| すぐできる | 3 | denyルール追加、/compact習慣化、CLAUDE.md整理 |
| 1日で可能 | 2 | Hook設定、MCP導入、カスタムスキル作成 |
| 1週間以上 | 1 | Agent SDK導入、CI/CD統合、チーム展開 |

### 6.2 レポートでの表示

改善提案は優先度スコア順にソートし、上位3つを `top_3_improvements` として表示する。
各提案には「期待される点数上昇」を明記する。
