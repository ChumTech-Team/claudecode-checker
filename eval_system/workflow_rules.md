# ワークフロー定義

> 作成日: 2026-03-28
> 対象: Claude Code利用時の開発ワークフロールール

---

## 1. 方法論選定ルール

### デフォルト開発方法論（規模別）

| 変更規模 | 方法論 | 詳細 |
|---------|--------|------|
| 小規模（1-3ファイル） | 直接実装 | Plan不要、即座に実装 |
| 中規模（4-10ファイル） | Plan Mode → 実装 | Shift+TabでPlanモードに切替、計画後に実装 |
| 大規模（10+ファイル） | エージェント分担 + ゲート管理 | worktree並列、フェーズゲート |

### プロジェクト別方法論

| PJ | 方法論 | 備考 |
|----|--------|------|
| 1人開発（ほとんどのPJ） | 軽量アジャイル（1週間スプリント） | - |
| チーム開発（<your-project>） | GitHub Flow + PR必須 | squash merge |
| 大規模（system_dev） | 5フェーズゲート管理 | requirements → architect → test-designer → developer → tester |

## 2. Plan-First ルール

### いつPlan Modeを使うか

- **3ファイル以上の変更が見込まれる場合**: 必ずPlan Modeで計画を立ててから実装
- **Plan Modeの使い方**: Shift+Tab でPlanモードに切替

### 計画に含めるもの

1. 変更対象ファイル一覧
2. 変更の順序と依存関係
3. テスト方針
4. ロールバック手順

### 計画テンプレート

```
### 変更計画
- 目的: [1行で]
- 変更ファイル: [一覧]
- 依存関係: [順序]
- リスク: [想定される問題]
- テスト: [確認方法]
- 見積もり: [所要時間]
```

## 3. TDD実践ルール

### Red-Green-Refactor サイクル

| フェーズ | 内容 |
|---------|------|
| Red | テストを先に書く（失敗することを確認） |
| Green | テストが通る最小限の実装を書く |
| Refactor | テストが通る状態を維持しながらリファクタリング |

### 場面別TDDルール

| 場面 | 手順 |
|------|------|
| 新機能 | テスト先行（Red → Green → Refactor） |
| バグ修正 | 再現テスト作成 → 修正 → テスト通過確認 |
| リファクタリング | 既存テスト通過を確認してから開始 |

### テストファイル規約

- TypeScript: `*.test.ts` / `*.test.tsx`（vitestで実行）
- Python: `test_*.py`（pytestで実行）
- テストファイルの変更時はそのテストのみ実行
- ソースファイルの変更時は対応するテストを自動実行（auto-test Hook）

## 4. コードレビュールール

### レビュー確信度フィルタ

| レベル | 意味 | 対応 |
|--------|------|------|
| [HIGH] | 確実に問題がある | 必須修正 |
| [MEDIUM] | 問題の可能性がある | 修正推奨 |
| [LOW] | 好みの問題/提案 | 任意 |

### レビュー観点

1. **セキュリティ**: SQLインジェクション、XSS、CSRF、認証/認可の漏れ、シークレットのハードコード
2. **パフォーマンス**: N+1クエリ、メモリリーク、不要な再レンダリング
3. **可読性**: 命名規則、コメント、複雑度
4. **テスト**: カバレッジ、エッジケース

### マルチエージェントレビュー（大規模PJ向け）

- `security-reviewer`: セキュリティ観点のレビュー（model: sonnet）
- `performance-reviewer`: パフォーマンス観点のレビュー（model: sonnet）
- メインレビューはPR作成時にclaude-code-actionで自動実行

## 5. Gitワークフロー

### ブランチ運用

```
main (protected)
  └── feature/* （機能開発）
  └── fix/* （バグ修正）
  └── chore/* （メンテナンス）
```

### マージルール

- PRはsquash mergeでmainにマージ
- PR作成: `gh pr create --title "..." --body "..."`
- マージ後: ローカルブランチを削除

### コミットメッセージ規約

```
<type>: <description>

type: feat / fix / chore / docs / refactor / test
```

## 6. Wrap-up Rituals（セッション終了時）

セッション終了時に必ず以下を実施:

1. 作業内容のサマリをMEMORY.mdに記録
2. 未完了タスクをTODOコメントとしてコードに残す
3. `git status` で未コミットの変更がないか確認
4. 次のセッションへの申し送り事項をメモ

---

## 7. オーケストレーション（エージェント間パイプライン）

### 逐次パイプライン（system_dev向け）

```
requirements-analyst → architect → test-designer → developer → tester
```

### 各フェーズの完了条件

| フェーズ | 完了条件 |
|---------|---------|
| requirements | 要件書のレビュー完了 |
| architect | 設計書の承認 |
| test-designer | テストケースの網羅性確認 |
| developer | 全テスト通過 |
| tester | E2Eテスト通過 |

### 完全開発サイクル

```
1. Plan Mode で計画（Shift+Tab）
2. TDDで実装（テスト先行）
3. claude-code-action でPRレビュー
4. squash merge でマージ
```
