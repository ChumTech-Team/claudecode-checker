# 03 コア機能

> Claude Code の組み込みツール、CLI、Git連携、ドキュメント生成を体系的にまとめたルール集。
> 「基礎スキル判定」の参照データとして使用する。

## A. ファイル操作ツール

### A-1. Read

| # | ルール | 補足 |
|---|--------|------|
| R-001 | `file_path` は絶対パスで指定する | 相対パスは不可 |
| R-002 | デフォルトで最大2000行を読み取る | 大きいファイルは `offset` / `limit` で範囲指定 |
| R-003 | 結果は `cat -n` 形式（行番号付き、1始まり）で返る | |
| R-004 | 画像ファイル（PNG, JPG等）を読み取ると視覚的にコンテンツが提示される | マルチモーダルLLMとして機能 |
| R-005 | PDFは10ページ超の場合 `pages` パラメータ必須 | 最大20ページ/リクエスト。省略すると失敗 |
| R-006 | Jupyter Notebook（.ipynb）は全セルと出力を統合して返す | コード・テキスト・可視化を含む |
| R-007 | ディレクトリは読み取り不可 | `ls` コマンドを使用する |
| R-008 | スクリーンショットのパスが提供された場合は常にReadで閲覧する | 一時ファイルパスにも対応 |

**情報源:** システムプロンプト（Read tool定義）

### A-2. Write

| # | ルール | 補足 |
|---|--------|------|
| R-009 | 既存ファイルは事前にReadで読み取ってからでないとエラー | 未読の既存ファイルへの書き込みは失敗する |
| R-010 | 既存ファイルの変更にはEditツールを優先する | Writeは新規作成 or 完全書き換え専用 |
| R-011 | ドキュメントファイル（*.md）やREADMEは明示的要求がない限り作成しない | |
| R-012 | 絵文字はユーザーが明示的に要求しない限りファイルに書かない | |

**情報源:** システムプロンプト（Write tool定義）

### A-3. Edit

| # | ルール | 補足 |
|---|--------|------|
| R-013 | 会話中に少なくとも1回Readしてからでないとエラー | |
| R-014 | `old_string` がファイル内で一意でないと編集失敗 | 周辺コンテキストを増やして一意にするか `replace_all: true` を使う |
| R-015 | Readの行番号プレフィックス部分を `old_string` に含めない | 行番号の後のインデントはそのまま保持 |
| R-016 | `replace_all` はファイル全体の変数名リネーム等に使う | |
| R-017 | 既存ファイルの編集を常に優先し、新規ファイル作成は必要最小限にする | |

**情報源:** システムプロンプト（Edit tool定義）

### A-4. Glob

| # | ルール | 補足 |
|---|--------|------|
| R-018 | ファイル名パターンでの検索には `find` ではなくGlobを使う | 例: `**/*.js`, `src/**/*.ts` |
| R-019 | 結果は変更日時順にソートされる | |
| R-020 | `path` パラメータに `"undefined"` や `"null"` を入力しない | 省略すればカレントディレクトリが使われる |

**情報源:** システムプロンプト（Glob tool定義）

### A-5. Grep

| # | ルール | 補足 |
|---|--------|------|
| R-021 | ファイル内容の検索には常にGrepを使う。Bashで `grep` や `rg` を実行しない | |
| R-022 | ripgrepの正規表現構文を使う | リテラルブレースはエスケープ: `interface\{\}` |
| R-023 | 複数行マッチには `multiline: true` を指定する | 例: `struct \{[\s\S]*?field` |
| R-024 | 標準ファイルタイプの絞り込みは `type` パラメータが `glob` より効率的 | 例: `type: "js"` |
| R-025 | `output_mode` のデフォルトは `files_with_matches` | `content` でマッチ行表示、`count` でカウント |
| R-026 | `head_limit` のデフォルトは250。`0`（無制限）はコンテキスト浪費に注意 | |

**情報源:** システムプロンプト（Grep tool定義）

### A-6. ツール使い分け原則

| # | ルール | 補足 |
|---|--------|------|
| R-027 | ファイル名検索 → Glob（`find` / `ls` の代わり） | |
| R-028 | ファイル内容検索 → Grep（`grep` / `rg` の代わり） | |
| R-029 | ファイル読み取り → Read（`cat` / `head` / `tail` の代わり） | |
| R-030 | ファイル編集 → Edit（`sed` / `awk` の代わり） | |
| R-031 | ファイル書き込み → Write（`echo >` の代わり） | |
| R-032 | agentic search（Glob + Grep）はRAGより優れる | ベクトルDBはコードの同期ずれとパーミッション問題があるため不採用 |

**情報源:** システムプロンプト、https://github.com/shanraisshan/claude-code-best-practice

## B. Bash Tool

| # | ルール | 補足 |

## D. Git連携

### D-1. コミットの安全ルール

| # | ルール | 補足 |
|---|--------|------|
| R-096 | ユーザーが明示的に要求した場合のみコミットを作成する | |
| R-097 | git configを変更しない | |
| R-098 | 破壊的コマンドはユーザーの明示的指示なしに実行しない | `push --force`, `reset --hard`, `checkout .`, `restore .`, `clean -f`, `branch -D` |
| R-099 | hookスキップ禁止 | `--no-verify`, `--no-gpg-sign` は使わない |
| R-100 | 常に新規コミットを作成する（amendしない） | pre-commitフック失敗時、amendは前のコミットを破壊するリスクがある |
| R-101 | `git add -A` / `git add .` より個別ファイル指定を優先 | `.env` 等の機密ファイルの誤コミット防止 |
| R-102 | main/masterへのforce pushは警告する | |
| R-103 | `-i` フラグ（インタラクティブ）は非対応 | `git rebase -i`, `git add -i` 等 |
| R-104 | `git rebase` に `--no-edit` は使用不可 | |

### D-2. コミットワークフロー

| # | ルール | 補足 |
|---|--------|------|
| R-105 | Step 1: `git status` + `git diff` + `git log` を並列実行 | |
| R-106 | Step 2: 変更を分析しコミットメッセージを起草 | |
| R-107 | Step 3: ステージング + コミット + `git status` 確認 | |
| R-108 | コミットメッセージはHEREDOC形式で渡す | 下記コマンド例参照 |
| R-109 | pre-commitフック失敗時は修正して新規コミットを作成 | amendしない |

**コミットメッセージ例:**
```bash
git commit -m "$(cat <<'EOF'
Add user authentication middleware

Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>
EOF
)"
```

### D-3. PR作成ワークフロー

| # | ルール | 補足 |
|---|--------|------|
| R-110 | Step 1: `git status` + `git diff` + リモート追跡確認 + `git log` + `git diff [base]...HEAD` を並列実行 | |
| R-111 | Step 2: 全コミットを分析しPRタイトル（70文字以内）・サマリーを起草 | 最新コミットだけでなく全コミットを確認 |
| R-112 | Step 3: ブランチ作成 + push + `gh pr create` を並列実行 | |

**PR作成コマンド例:**
```bash
gh pr create --title "the pr title" --body "$(cat <<'EOF'
## Summary
<1-3 bullet points>

## Test plan
[Bulleted markdown checklist of TODOs for testing the pull request...]

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

### D-4. Git / PRベストプラクティス

| # | ルール | 補足 |
|---|--------|------|
| R-113 | PRは小さく保つ | p50で118行 |
| R-114 | 常にsquash merge | 1機能=1コミット、`git revert`/`bisect` が容易 |
| R-115 | 最低1時間に1回コミット | タスク完了時すぐに |
| R-116 | PRコメント閲覧 | `gh api repos/foo/bar/pulls/123/comments` |
| R-117 | Git worktreeで並列開発 | `claude --worktree feature-name` または frontmatter `isolation: worktree` |

**情報源:** システムプロンプト、https://github.com/shanraisshan/claude-code-best-practice

## E. Web検索/フェッチ

| # | ルール | 補足 |
|---|--------|------|
| R-118 | WebSearchは検索クエリ最小2文字 | `allowed_domains` / `blocked_domains` でドメイン絞り込み可 |

## 付録1: ツール選択フローチャート

```
タスクを受け取った
  |
  +-- ファイルを探したい？
  |     +-- ファイル名/パターンで探す → Glob (R-018)
  |     +-- ファイル内容で探す → Grep (R-021)
  |
  +-- ファイルを読みたい？
  |     +-- 画像/PDF/Notebook → Read (R-004, R-005, R-006)
  |     +-- 通常のファイル → Read (R-001)
  |     +-- ディレクトリ一覧 → Bash (`ls`)  (R-007)
  |
  +-- ファイルを変更したい？
  |     +-- 部分的な編集 → Edit (R-013)
  |     +-- 全体書き換え/新規作成 → Write (R-009, R-010)
  |
  +-- シェルコマンドを実行したい？
  |     +-- find/grep/cat/sed 等 → 専用ツール優先 (R-041)
  |     +-- それ以外 → Bash (R-033)
  |
  +-- Web情報が必要？
  |     +-- 検索したい → WebSearch (R-118)
  |     +-- 特定URLの内容を取得 → WebFetch (R-120)
  |     +-- 認証が必要なURL → 専用MCP / gh CLI (R-122)
  |
  +-- Gitオペレーション？
  |     +-- コミット → 安全ルール確認 (R-096〜R-104) → ワークフロー (R-105〜R-109)
  |     +-- PR作成 → ワークフロー (R-110〜R-112)
  |
  +-- ドキュメント生成？
        +-- Word → docx-js or XML編集 (R-127)
        +-- Excel → pandas or openpyxl (R-131)
        +-- PowerPoint → pptxgenjs or XML編集 (R-135)
        +-- PDF → pypdf / pdfplumber / reportlab (R-139)
```

## 付録2: よくある間違い表

| # | 間違い | 正しい対応 | 関連ルール |
